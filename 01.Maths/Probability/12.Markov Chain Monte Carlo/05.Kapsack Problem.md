

> _"The knapsack problem is one of the most honest problems in all of computer science. It looks trivial — I have items with weights and values, I want to fill a bag optimally — and yet it is NP-hard. Every approach I try turns out to either be exponential in the worst case, or only approximate, or pseudopolynomial (which sounds polynomial but isn't). And then, astonishingly, someone turned it into a public-key cryptosystem — and then someone else broke that cryptosystem using the very structure that was supposed to make it hard. The knapsack problem is where algorithms, complexity theory, and cryptography all collide."_

---

## The Problem

**0-1 Knapsack.** Given $n$ items, each with weight $w_i > 0$ and value $v_i > 0$, and a knapsack of capacity $W$, find the subset $S \subseteq {1, \ldots, n}$ that maximises total value subject to the weight constraint:

$$\max_{S \subseteq {1,\ldots,n}} \sum_{i \in S} v_i \qquad \text{subject to} \qquad \sum_{i \in S} w_i \leq W$$

The "0-1" means each item is either taken ($x_i = 1$) or not ($x_i = 0$) — no fractional items, no duplicates. In integer programming notation:

$$\max \sum_{i=1}^n v_i x_i \qquad \text{s.t.} \qquad \sum_{i=1}^n w_i x_i \leq W, \quad x_i \in {0,1}$$

**Variants:**

- **Fractional knapsack:** Items can be taken in fractions. Solvable in $O(n \log n)$ by a greedy algorithm — take items in decreasing order of value density $v_i/w_i$.
- **Unbounded knapsack:** Each item can be taken any number of times.
- **Bounded knapsack:** Item $i$ can be taken at most $c_i$ times.
- **Multi-dimensional knapsack:** Multiple weight constraints (e.g., weight and volume).

The 0-1 version is the hard one. All complexity discussion below is about 0-1 knapsack unless stated.

## Brute Force: $O(2^n)$

The naive approach: enumerate all $2^n$ subsets, compute total weight and value for each, return the feasible subset with maximum value.

**Correctness:** Trivially correct — we check every possibility.

**Complexity:** $\Theta(2^n \cdot n)$ — one factor of $n$ for computing the sum. For $n = 30$, that's $\approx 10^{10}$ operations. For $n = 60$, $\approx 10^{19}$. Completely impractical.

**Meet-in-the-middle (slight improvement):** Split items into two halves of size $n/2$. Enumerate all $2^{n/2}$ subsets of each half, storing (weight, value) pairs. For each subset of the left half, binary search the right half for the highest-value subset that fits in the remaining capacity. Total: $O(2^{n/2} \cdot n)$. For $n = 60$: $\approx 10^{10}$ — still impractical but better. This is the best known exact algorithm in the exponential family.

## Dynamic Programming: $O(nW)$

The standard exact algorithm for small $W$.

**Recurrence.** Define:

$$\text{dp}[i][w] = \text{maximum value using items } 1, \ldots, i \text{ with weight capacity } w$$

**Base case:** $\text{dp}[0][w] = 0$ for all $w$ (no items, no value).

**Transition:** For item $i$ with weight $w_i$ and value $v_i$:

$$\text{dp}[i][w] = \begin{cases} \text{dp}[i-1][w] & \text{if } w_i > w \quad (\text{can't take item } i) \ \max!\big(\text{dp}[i-1][w],; v_i + \text{dp}[i-1][w - w_i]\big) & \text{if } w_i \leq w \end{cases}$$

The second case: either skip item $i$ (value $\text{dp}[i-1][w]$) or take it (value $v_i$ plus the best we can do with the remaining capacity $w - w_i$, using items $1, \ldots, i-1$).

**Correctness proof by induction.**

_Claim:_ $\text{dp}[i][w]$ is the maximum value achievable using items $1, \ldots, i$ with capacity $w$.

_Base case:_ $\text{dp}[0][w] = 0$ — no items available, maximum value is 0. ✓

_Inductive step:_ Assume $\text{dp}[i-1][\cdot]$ is correct. In any optimal solution for items $1,\ldots,i$ with capacity $w$, either item $i$ is included or not.

- If not included: optimal value $= \text{dp}[i-1][w]$ by the inductive hypothesis.
- If included: $w_i \leq w$ (feasibility), and the remaining items $1,\ldots,i-1$ must be packed optimally in capacity $w - w_i$, giving value $v_i + \text{dp}[i-1][w - w_i]$ by the inductive hypothesis.

Taking the max of both cases gives $\text{dp}[i][w]$. $\blacksquare$

**Complexity:** $O(nW)$ time, $O(W)$ space (maintain only the previous row — note that the transition only reads $\text{dp}[i-1]$, so I can overwrite in place if I scan $w$ from $W$ down to $w_i$).

```python
def knapsack_dp(weights, values, W):
    n = len(weights)
    dp = [0] * (W + 1)
    
    for i in range(n):
        # Scan backwards to avoid using item i twice
        for w in range(W, weights[i] - 1, -1):
            dp[w] = max(dp[w], values[i] + dp[w - weights[i]])
    
    return dp[W]
```

**Reconstruction:** To recover the optimal subset (not just the value), maintain a boolean table $\text{keep}[i][w] = \mathbf{1}[\text{item } i \text{ is taken in the optimal solution for } (i,w)]$. Then backtrack from $(n, W)$.

## Why $O(nW)$ is NOT Polynomial

Here is the subtlety that trips people up.

The complexity $O(nW)$ looks polynomial. It is **not**. It is **pseudopolynomial**.

**Why?** The input size is $O(n \log W + \sum_i \log w_i + \sum_i \log v_i)$ bits — to specify $W$ requires $\log_2 W$ bits, not $W$ bits. If I double the number of bits used to represent $W$ (i.e., replace $W$ with $W^2$), the DP table size doubles exponentially in the input size.

**Formally:** Polynomial time means polynomial in the _length of the input_. The input length is $L = O(n \log W)$. The DP runs in time $O(nW) = O(n \cdot 2^{\log W}) = O(n \cdot 2^{L/n})$ — exponential in $L/n$.

**Intuition:** $W = 10^{18}$ can be written in 60 bits — a tiny input. But the DP table has $10^{18}$ entries — completely infeasible. The DP is only practical when $W$ is small (say, $\leq 10^6$ or so).

This distinction — polynomial vs pseudopolynomial — is critical. The knapsack DP is not a polynomial-time algorithm. The knapsack problem remains NP-hard.

> _I remember being confused about this the first time I saw it. "$O(nW)$ — that's linear in $n$ and $W$, which are the problem parameters!" But the problem parameters are not the input size. The input size is the number of bits needed to write down the problem. $W$ contributes $\log W$ bits, not $W$ bits. Pseudopolynomial algorithms are fast when the parameter values are small but slow when the values are large — even if the values are written compactly._

## NP-Hardness

**Theorem.** 0-1 Knapsack is NP-hard.

**Proof** (reduction from Subset Sum).

**Subset Sum:** Given integers $a_1, \ldots, a_n$ and target $T$, does there exist $S \subseteq {1,\ldots,n}$ with $\sum_{i \in S} a_i = T$?

Subset Sum is NP-complete (standard result — reduce from 3-SAT or from Partition).

**Reduction:** Given a Subset Sum instance $(a_1,\ldots,a_n, T)$, construct a Knapsack instance:

- Set $w_i = v_i = a_i$ for all $i$.
- Set $W = T$.

**Claim:** The Subset Sum instance has a solution iff the Knapsack instance has optimal value $\geq T$.

- ($\Rightarrow$) If $\sum_{i \in S} a_i = T$, then $\sum_{i \in S} w_i = T \leq W$ (feasible) and $\sum_{i \in S} v_i = T$.
- ($\Leftarrow$) If the Knapsack has value $\geq T$, then there exists $S$ with $\sum_{i \in S} v_i \geq T$ and $\sum_{i \in S} w_i \leq T$. Since $v_i = w_i = a_i > 0$ and $\sum a_i \leq T$, combined with $\sum a_i \geq T$, we get $\sum_{i \in S} a_i = T$.

The reduction is polynomial (linear time). So Knapsack is NP-hard. $\blacksquare$

**Knapsack is also in NP:** Given a candidate subset $S$, verify feasibility and value in $O(n)$. So 0-1 Knapsack is **NP-complete** (as a decision problem: "is there a feasible subset with value $\geq V$?").

## Approximation: The FPTAS

Since exact solution is NP-hard (assuming $P \neq NP$), what can we approximate?

**Definition.** An algorithm is a **$(1-\epsilon)$-approximation** for maximization if it returns a solution of value $\geq (1-\epsilon) \cdot \text{OPT}$ for any $\epsilon > 0$.

**Definition.** A **Fully Polynomial-Time Approximation Scheme (FPTAS)** is a $(1-\epsilon)$-approximation algorithm that runs in time polynomial in both $n$ and $1/\epsilon$.

**Theorem.** 0-1 Knapsack has an FPTAS.

**Construction.**

1. Let $v_{\max} = \max_i v_i$. Set the **scaling factor** $K = \lfloor \epsilon v_{\max} / n \rfloor$.
2. Define **scaled values** $v_i' = \lfloor v_i / K \rfloor$ (integer division).
3. Run the DP on the scaled instance: the scaled optimal value is at most $V' = \sum_i v_i' \leq \sum_i v_i/K \leq n \cdot v_{\max}/K \leq n^2/\epsilon$.
4. DP runs in time $O(n \cdot V') = O(n^3/\epsilon)$ — polynomial in $n$ and $1/\epsilon$.
5. Return the optimal solution for the scaled instance.

**Approximation guarantee.** Let $S^*$ be the true optimum and $\hat{S}$ the FPTAS solution. For any item $i$: $v_i' = \lfloor v_i/K \rfloor \geq v_i/K - 1$, so:

$$\sum_{i \in S^_} v_i' \geq \sum_{i \in S^_} (v_i/K - 1) = \frac{\text{OPT}}{K} - |S^*| \geq \frac{\text{OPT}}{K} - n$$

Since $\hat{S}$ is optimal in the scaled problem:

$$\sum_{i \in \hat{S}} v_i' \geq \sum_{i \in S^*} v_i' \geq \frac{\text{OPT}}{K} - n$$

So the unscaled value of $\hat{S}$:

$$\sum_{i \in \hat{S}} v_i \geq K \sum_{i \in \hat{S}} v_i' \geq K!\left(\frac{\text{OPT}}{K} - n\right) = \text{OPT} - nK \geq \text{OPT} - \epsilon v_{\max} \geq \text{OPT}(1-\epsilon)$$

using $nK \leq \epsilon v_{\max}$ and $v_{\max} \leq \text{OPT}$ (the single best item is a lower bound on OPT). $\blacksquare$

> _The FPTAS is one of the most elegant tricks in approximation algorithms. The DP is pseudopolynomial because $W$ and $V$ can be large. The fix: scale down the values so that $V'$ is bounded by a polynomial in $n$ and $1/\epsilon$. The scaling introduces a small additive error per item, which telescopes to a $(1-\epsilon)$ multiplicative bound. The insight is that if I only care about an $\epsilon$-approximation, I don't need the last few bits of precision in the values — rounding them loses at most $\epsilon \cdot \text{OPT}$._

## Greedy and Its Failure

**Fractional knapsack greedy:** Sort items by value density $v_i/w_i$ descending, take items greedily until the knapsack is full (take the last item fractionally if needed). This is optimal for the fractional variant — provably. Standard exchange argument proof.

**0-1 knapsack greedy failure:** The greedy fails for 0-1 knapsack. Counterexample:

|Item|Weight|Value|Density $v/w$|
|---|---|---|---|
|1|1|1|1.0|
|2|10|9|0.9|

$W = 10$. Greedy takes item 1 (highest density), then item 2 (fits, since $1 + 10 > 10$... wait, $1 + 10 = 11 > 10$). So greedy takes only item 1, value $= 1$. Optimal: take item 2 alone, value $= 9$.

**How bad can greedy be?** The greedy can be arbitrarily worse than optimal. In the example above, the ratio is $1/9 \to 0$ as I scale the values. There is no constant approximation ratio for the greedy on 0-1 knapsack.

**A $1/2$-approximation:** Run the greedy on the fractional problem to get a fractional solution of value $\text{OPT}_f \geq \text{OPT}$. The fractional solution uses at most one fractionally taken item (the "break" item). Compare: (value of all items except break item) vs (value of break item alone). Take the better. This gives value $\geq \text{OPT}_f / 2 \geq \text{OPT}/2$. A 1/2-approximation in $O(n \log n)$.

## Probabilistic Analysis: How Hard Is Knapsack on Average?

Worst-case NP-hardness doesn't mean every instance is hard. What does a "random" knapsack instance look like?

**Model.** Draw $w_i \sim \text{Uniform}(0, 1)$ and $v_i \sim \text{Uniform}(0, 1)$ i.i.d. Set $W = n/2$ (half the total expected weight).

**Expected total weight:** $E[\sum w_i] = n/2 = W$. So the capacity is exactly at the "interesting" point — not so loose that everything fits, not so tight that nothing fits.

**Observation.** For random instances with $W = \Theta(n)$, the DP runs in $O(nW) = O(n^2)$ — polynomial in $n$. The pseudopolynomiality bites only when $W$ is exponentially large relative to $n$.

**The continuous relaxation (fractional knapsack) is tight on average.** For random instances:

$$\text{OPT}_{\text{integer}} / \text{OPT}_{\text{fractional}} \to 1 \quad \text{as } n \to \infty$$

**Informal argument:** In the fractional solution, the break item is taken fractionally. But as $n \to \infty$, the break item's fractional part becomes a negligible fraction of the total value — $O(v_{\max}/\text{OPT}) = O(1/n)$ contribution. So the integrality gap vanishes.

**More precisely:** By concentration inequalities (Chernoff/Hoeffding), the total value of items packed by the greedy concentrates sharply around its expectation. The fractional "wasted" portion of the break item has expected value $E[v_{\text{break}}/2] = O(1/n \cdot \text{OPT})$ — vanishing relative to $\text{OPT}$.

> _This is the usual phenomenon with NP-hard problems on random instances: the hard instances are rare, specially structured, and the typical case is much easier. The "hard" instances for knapsack are ones where many subsets have weights very close to $W$ — an arithmetic coincidence that doesn't arise generically. Random instances with continuous weights are easy._

**The hardest instances.** The knapsack problem is hardest when:

1. Many subsets have total weight exactly $W$ — many feasible solutions to compare.
2. Items have similar value densities $v_i/w_i$ — hard to discriminate between items.
3. $W$ is large (making DP infeasible) but not so large that all items fit.

Instances with weights drawn from ${1, \ldots, \text{poly}(n)}$ uniformly are in the "easy" regime because $W$ is polynomial and DP works. The hard regime is $W = \Theta(2^n)$ — but then the weights themselves require $\Omega(n)$ bits to specify, so the input is large anyway.

## Branch and Bound

In practice, exact knapsack solvers use **branch and bound** — an implicit enumeration of the $2^n$ subsets, pruning branches that can't beat the current best solution.

**Algorithm skeleton:**

```
def branch_and_bound(items, W):
    best = greedy_solution(items, W)     # Initial lower bound
    
    def recurse(i, remaining_W, current_value):
        nonlocal best
        if i == n:
            best = max(best, current_value)
            return
        
        # Upper bound: fractional knapsack on remaining items
        ub = current_value + fractional_knapsack(items[i:], remaining_W)
        if ub <= best:
            return   # Prune: can't beat current best
        
        # Branch: include item i
        if items[i].weight <= remaining_W:
            recurse(i+1, remaining_W - items[i].weight,
                    current_value + items[i].value)
        
        # Branch: exclude item i
        recurse(i+1, remaining_W, current_value)
    
    recurse(0, W, 0)
    return best
```

**The bounding function** — fractional knapsack on remaining items — is an upper bound on what's achievable from state $(i, \text{remaining_W})$, because the fractional relaxation is always $\geq$ the integer optimum.

**Correctness:** The tree enumerates all $2^n$ subsets implicitly, but prunes branches where the upper bound can't improve the best known solution. Correctness follows from the validity of the upper bound.

**Complexity:** Worst case $O(2^n)$ (no pruning, all branches explored). In practice, with good bounding and smart branching order (e.g., by value density), branch and bound solves instances with $n = 100$–$1000$ in seconds — far better than the worst case.

**LP relaxation as bound.** An even tighter upper bound: solve the LP relaxation (allow $x_i \in [0,1]$, not just ${0,1}$). This requires solving an LP at each node — more expensive per node, but fewer nodes explored. This is the basis of modern mixed-integer programming (MIP) solvers.

## The Merkle-Hellman Knapsack Cryptosystem

Here is where things get strange. In 1978, Merkle and Hellman proposed a **public-key cryptosystem** based on the knapsack problem. The idea: hiding a structured (easy) knapsack problem inside an apparently random (hard) one.

### Superincreasing Sequences

**Definition.** A sequence $a_1, a_2, \ldots, a_n$ is **superincreasing** if each element exceeds the sum of all preceding elements:

$$a_k > \sum_{j=1}^{k-1} a_j \quad \text{for all } k = 2, \ldots, n$$

**Example:** ${1, 2, 4, 8, 16, \ldots}$ — powers of 2 are superincreasing.

**Key property:** A superincreasing knapsack is solvable in $O(n)$ by a greedy algorithm.

**Algorithm:** Given target $T$ and superincreasing $a_1, \ldots, a_n$ (sorted descending for clarity: $a_n > a_{n-1} > \cdots$):

```
For i = n, n-1, ..., 1:
    if a_i <= T:
        include a_i  (set x_i = 1)
        T -= a_i
```

**Correctness:** Since $a_n > \sum_{j=1}^{n-1} a_j$, if the target $T \geq a_n$, then item $n$ _must_ be included in any solution — all other items combined are insufficient. So we greedily include or exclude the largest element, reduce the target, and recurse. Unique solution exists iff the target is achievable. $\blacksquare$

### Key Generation

1. Generate a **superincreasing sequence** $\mathbf{a} = (a_1, \ldots, a_n)$ — the private key.
2. Choose modulus $m > \sum_i a_i$ and multiplier $w$ with $\gcd(w, m) = 1$ — so $w$ has a multiplicative inverse $w^{-1} \bmod m$.
3. Compute the **public key** $\mathbf{b} = (b_1, \ldots, b_n)$ where $b_i = w \cdot a_i \bmod m$.

The public key $\mathbf{b}$ looks like a random knapsack instance — hard to solve.

### Encryption and Decryption

**Encryption** (sender knows $\mathbf{b}$, wants to send $n$-bit message $\mathbf{x} \in {0,1}^n$):

$$c = \sum_{i=1}^n b_i x_i = \sum_{i=1}^n (w a_i \bmod m) x_i$$

Send ciphertext $c$.

**Decryption** (receiver knows $\mathbf{a}, w, m$):

1. Compute $c' = w^{-1} c \bmod m$.
2. Note: $c' = w^{-1} \sum_i (wa_i \bmod m) x_i \bmod m = \sum_i a_i x_i \bmod m$.
3. Since $m > \sum_i a_i \geq \sum_{i: x_i=1} a_i$, there is no wraparound — $c' = \sum_i a_i x_i$ exactly.
4. Solve the superincreasing knapsack $\sum_i a_i x_i = c'$ in $O(n)$.

**Why it seems secure:** An adversary sees $\mathbf{b}$ and $c$. To decrypt, they must solve $\sum_i b_i x_i = c$ — an apparently random knapsack instance. NP-hard!

### Breaking It: The Shamir-Zippel Attack (1982)

Shamir (and independently others) broke the Merkle-Hellman system in 1982 using **lattice reduction** — specifically, the LLL algorithm.

**The attack, intuitively.** The public key $\mathbf{b}$ is not a truly random knapsack. It is a _modular transformation_ of a superincreasing sequence. The structure is hidden, but not destroyed — it lives in a lattice.

**Setup.** The attacker wants to find $w^{-1} \bmod m$ (or equivalently, a pair $(w', m')$ that transforms $\mathbf{b}$ back into a superincreasing sequence). This is a problem of finding a **short vector in a lattice**.

**The lattice.** Construct the $(n+1) \times (n+1)$ lattice basis matrix:

$$B = \begin{pmatrix} m & 0 & 0 & \cdots & 0 \ b_1 & 1 & 0 & \cdots & 0 \ b_2 & 0 & 1 & \cdots & 0 \ \vdots & & & \ddots & \vdots \ b_n & 0 & 0 & \cdots & 1 \end{pmatrix}$$

Any vector in the lattice $\mathcal{L}(B)$ has the form $Bz$ for integer $z$. One particular short vector encodes $(w^{-1}, a_1, a_2, \ldots, a_n)$ scaled appropriately — the private key.

**LLL algorithm** (Lenstra-Lenstra-Lovász, 1982): given a lattice basis, find a "reasonably short" vector in polynomial time. For the Merkle-Hellman lattice, this short vector recovers (or approximates) the private key.

**Result:** The Merkle-Hellman cryptosystem is completely broken in practice. Any ciphertext can be decrypted by an adversary using the LLL algorithm in polynomial time.

> _The Merkle-Hellman break is one of the most instructive failures in cryptography. The designers believed that modular transformation destroyed the superincreasing structure — and it does, in the sense that $\mathbf{b}$ is indistinguishable from a random sequence by naive tests. But the structure is preserved in a higher-dimensional sense: the transformation $b_i = wa_i \bmod m$ maps the private key to a specific short vector in a lattice, and lattice basis reduction finds short vectors efficiently. "Hiding" structure by modular arithmetic is not the same as destroying it. The lesson: security arguments must be rigorous, not intuitive._

## Connections to Other Topics

**To [[Code-Breaking — Classical Ciphers & Information Theory]]:** The Merkle-Hellman break uses the LLL algorithm, which is a polynomial-time algorithm for an apparently hard problem. Similarly, Vigenère seemed secure but had exploitable structure. The common theme: security must be proved against all polynomial-time algorithms, not just obvious ones.

**To [[Dynamic Programming]]:** The knapsack DP is the canonical example of DP over subsets, illustrating the principle of optimal substructure. The recurrence $\text{dp}[i][w] = \max(\text{dp}[i-1][w], v_i + \text{dp}[i-1][w-w_i])$ is the template I reach for whenever I see a binary include/exclude decision.

**To [[NP and Computational Complexity]]:** The reduction from Subset Sum to Knapsack is a prototypical polynomial-time reduction. The FPTAS illustrates that NP-hardness doesn't preclude efficient approximation — a subtlety that distinguishes knapsack from problems like Max-Clique (which has no FPTAS unless $P = NP$).

**To probability:** The probabilistic analysis (random instances are easy) is an instance of a general phenomenon. Many NP-hard problems are easy on average. The theoretically hard instances are measure-zero in the space of all instances. This motivates the study of **smoothed complexity** (Spielman and Teng) — worst case over small random perturbations of instances — as a more realistic complexity measure than worst-case.

## Implementation: Full DP with Reconstruction

```python
def knapsack_with_reconstruction(weights, values, W):
    n = len(weights)
    # dp[w] = max value with capacity w
    dp   = [0] * (W + 1)
    # keep[i][w] = True if item i is taken in optimal solution for capacity w
    keep = [[False] * (W + 1) for _ in range(n)]
    
    for i in range(n):
        # Scan backwards: ensures each item used at most once
        for w in range(W, weights[i] - 1, -1):
            if values[i] + dp[w - weights[i]] > dp[w]:
                dp[w] = values[i] + dp[w - weights[i]]
                keep[i][w] = True
    
    # Reconstruct optimal subset
    chosen = []
    w = W
    for i in range(n - 1, -1, -1):
        if keep[i][w]:
            chosen.append(i)
            w -= weights[i]
    
    return dp[W], chosen

# Example
weights = [2, 3, 4, 5]
values  = [3, 4, 5, 6]
W       = 8
opt_val, opt_items = knapsack_with_reconstruction(weights, values, W)
print(f"Optimal value: {opt_val}")
print(f"Items taken:   {opt_items}")
# Expected: value = 10, items = [1, 3] (0-indexed: weights 3+5=8, values 4+6=10)
```
