## ✦ The Central Idea

A learning algorithm cannot discover structure that is not expressible within its hypothesis class $\mathcal{H}$. A linear model can only learn linear decision boundaries. But if you engineer the _right features_, you can expose nonlinear structure to a linear model — the model stays simple, the features do the lifting.

Feature engineering is the process of transforming raw variables $x \in \mathcal{X}_{\text{raw}}$ into a representation $\phi(x) \in \mathcal{X}_{\text{engineered}}$ such that:

1. The signal-to-noise ratio is higher in the new space.
2. The relevant structure is more linearly separable (for linear models) or more axis-aligned (for tree models).
3. The representation is stable with respect to the invariances in the problem (e.g., a digit "7" looks like a "7" whether it is shifted left or right — raw pixel features do not capture this).

The **fundamental tension:** richer features reduce bias (the model can fit more complex functions) but increase variance (more degrees of freedom, more room to overfit). Feature engineering is inductive bias design. Every choice encodes an assumption about the problem.

## ✦ 1. Categorical Encoding

### 1.1 The Problem

Machine learning models (except some tree-based models) require numerical input. A variable like `colour ∈ {red, green, blue}` has no inherent numerical order. How you encode it changes what structure the model can discover.

### 1.2 One-Hot Encoding

Replace a categorical variable with $k$ binary variables (one per category):

$$\text{colour} = \text{red} \implies [1, 0, 0], \quad \text{green} \implies [0, 1, 0], \quad \text{blue} \implies [0, 0, 1]$$

**Properties:**

- No ordinal assumption — no category is "greater than" another.
- Introduces perfect multicollinearity: $\sum_{j=1}^k z_j = 1$ always. For models with an intercept (e.g., linear regression), drop one category ("dummy variable trap"). sklearn's `OneHotEncoder` with `drop='first'` handles this.
- Produces sparse, high-dimensional vectors for high-cardinality categoricals.

```python
import numpy as np

def one_hot(categories, vocab):
    """vocab = list of all possible categories."""
    idx = {c: i for i, c in enumerate(vocab)}
    result = np.zeros((len(categories), len(vocab)))
    for i, c in enumerate(categories):
        result[i, idx[c]] = 1.0
    return result

vocab = ['red', 'green', 'blue']
data  = ['red', 'blue', 'green', 'red']
print(one_hot(data, vocab))
# [[1. 0. 0.]
#  [0. 0. 1.]
#  [0. 1. 0.]
#  [1. 0. 0.]]
```

**When to use:** The default for nominal categoricals with low to medium cardinality ($k \lesssim 50$).

---

### 1.3 Label Encoding

Map each category to an integer: `red → 0, green → 1, blue → 2`.

**Critical limitation:** This imposes an ordinal relationship. The model will interpret `blue (2)` as "twice green (1)" and "twice the distance from red (0) as green." For purely nominal categories, this is wrong. For ordinal categories (e.g., `low < medium < high`), it is exactly right.

```python
# Ordinal encoding is correct here:
size_map = {'small': 0, 'medium': 1, 'large': 2, 'xl': 3}
sizes_encoded = [size_map[s] for s in sizes]
```

**Rule:** Label encoding for ordinal, one-hot for nominal.

---

### 1.4 Target Encoding (Mean Encoding)

Replace each category with the mean of the target variable for that category:

$$\hat{\mu}_k = \frac{1}{|S_k|} \sum_{i \in S_k} y_i, \quad S_k = {i : x_i = k}$$

**Advantage:** Naturally handles high-cardinality categoricals (e.g., US zip codes — 40,000+ values). One-hot encoding would produce 40,000 columns; target encoding produces 1.

**The leakage problem:** If computed naively on the training set, the encoding of sample $i$ uses $y_i$ itself in the mean (since $i \in S_{x_i}$). This leaks the target into the feature. For rare categories, the encoding _is_ the target with almost no averaging.

**Fix 1 — Leave-one-out encoding:** $$\hat{\mu}_k^{(i)} = \frac{\sum_{j \in S_k, j \neq i} y_j}{|S_k| - 1}$$

**Fix 2 — Smoothed (Bayesian) encoding:** $$\hat{\mu}_k = \frac{n_k \cdot \bar{y}_k + \lambda \cdot \bar{y}_{\text{global}}}{n_k + \lambda}$$

where $n_k = |S_k|$ and $\lambda$ controls the smoothing strength. For small $n_k$ (rare category), the encoding falls back toward the global mean. For large $n_k$, it approaches the category mean. This is empirical Bayes.

**Fix 3 — Compute within cross-validation folds:** The cleanest approach. For each fold, compute encoding on the out-of-fold data to encode the fold's training samples.

```python
def target_encode_smoothed(X_cat, y, lam=10):
    """Smoothed target encoding. Fit on X_cat, y (training only)."""
    global_mean = y.mean()
    encoding = {}
    for cat in np.unique(X_cat):
        mask = (X_cat == cat)
        n_k  = mask.sum()
        cat_mean = y[mask].mean()
        encoding[cat] = (n_k * cat_mean + lam * global_mean) / (n_k + lam)
    return encoding

# Apply: use training encoding on test set
enc = target_encode_smoothed(X_train_cat, y_train)
X_train_encoded = np.array([enc.get(c, global_mean) for c in X_train_cat])
X_test_encoded  = np.array([enc.get(c, global_mean) for c in X_test_cat])
```

**When to use:** High-cardinality categoricals + tree-based models (XGBoost, LightGBM). Less useful for linear models where one-hot + regularisation is usually sufficient.

---

### 1.5 The Hashing Trick

Map categories to a fixed-size vector of dimension $m$ using a hash function:

$$\phi_j(x) = \mathbf{1}[h(x) \bmod m = j]$$

**Properties:**

- $O(1)$ computation, no vocabulary needed, handles unseen categories at inference.
- Collisions (two categories map to the same bin) introduce noise. For large $m$, collisions are rare.
- Cannot be inverted — you cannot recover which category produced which feature.

Used at scale in web advertising and natural language processing where cardinality is enormous and exact encodings are impractical.

---

### 1.6 Encoding Decision Tree

```
Is the variable ordinal (has a natural order)?
    YES → Label encoding with the correct order
    NO  →
        How many unique values (cardinality)?
            < 50   → One-hot encoding
            50–500 → Target encoding (smoothed, CV-safe) or hashing
            > 500  → Hashing trick or target encoding with strong smoothing
```

## ✦ 2. Numerical Transformations

### 2.1 Log Transform

For a right-skewed feature $x > 0$:

$$x' = \log(x + 1)$$

(The $+1$ avoids $\log(0)$; use $\log(x)$ directly if $x > 0$ always.)

**Why it works:** Many natural quantities are log-normally distributed — income, population, word frequencies, housing prices. If $X \sim \text{LogNormal}(\mu, \sigma^2)$, then $\log X \sim \mathcal{N}(\mu, \sigma^2)$. The log transform _linearises_ multiplicative relationships: if the true model is $y = a \cdot x^\beta \cdot \epsilon$, then $\log y = \log a + \beta \log x + \log \epsilon$ is a linear model in log space.

**Effect on modelling:** Reduces influence of outliers (compresses the tail), stabilises variance, makes the distribution more symmetric.

```
Right-skewed distribution:        Log-transformed:
  ██                                   ████
  ███                                  ██████
  ████                                 ████████
  █████████                            ██████████
  ████████████████████                 ████████████
  ────────────────────                 ────────────
  0    200   1000  10000               0   3    5   9  (log scale)
```

---

### 2.2 Box-Cox Transform

The Box-Cox family generalises the log transform:

$$x^{(\lambda)} = \begin{cases} \frac{x^\lambda - 1}{\lambda} & \lambda \neq 0 \ \log x & \lambda = 0 \end{cases}$$

The parameter $\lambda$ is estimated by MLE (maximise the log-likelihood of a Gaussian fit to the transformed data). Requires $x > 0$.

Special cases: $\lambda = 0$ gives log; $\lambda = 0.5$ gives square root; $\lambda = 1$ is the identity; $\lambda = -1$ gives the reciprocal (up to constants).

```python
from scipy.stats import boxcox
x_transformed, lambda_opt = boxcox(x_train + 1e-6)  # ensure positivity
# Apply same lambda to test:
from scipy.special import boxcox1p
x_test_transformed = boxcox1p(x_test, lambda_opt)
```

---

### 2.3 Yeo-Johnson Transform

Extends Box-Cox to handle $x \leq 0$:

$$\psi(x, \lambda) = \begin{cases} \frac{(x+1)^\lambda - 1}{\lambda} & x \geq 0, \lambda \neq 0 \ \log(x+1) & x \geq 0, \lambda = 0 \ \frac{-((-x+1)^{2-\lambda} - 1)}{2 - \lambda} & x < 0, \lambda \neq 2 \ -\log(-x+1) & x < 0, \lambda = 2 \end{cases}$$

Use Yeo-Johnson by default when features include non-positive values. sklearn's `PowerTransformer` implements both.

### 2.4 Binning / Quantisation

Partition a continuous variable into discrete bins:

$$x' = k \iff x \in [b_k, b_{k+1})$$

**Two strategies:**

- **Equal-width:** Bins have equal range. Sensitive to skew and outliers — most points may fall in one bin.
- **Equal-frequency (quantile):** Each bin contains the same number of points. More robust to skew. This is what `pd.qcut` implements.

**When useful:**

- Adding non-linearity to a linear model without expanding to polynomials.
- When the relationship between the feature and target is piecewise constant (e.g., age groups in medicine).
- Reducing sensitivity to exact values and measurement noise.

**Connection to decision trees:** A decision tree learns optimal bin boundaries automatically. If you are using trees, manual binning is usually unnecessary and may discard useful fine-grained information.

```python
# Equal-frequency binning from scratch:
def quantile_bin(x, n_bins):
    quantiles = np.linspace(0, 100, n_bins + 1)
    boundaries = np.percentile(x, quantiles)
    return np.digitize(x, boundaries[1:-1])  # returns bin index 0..n_bins-1
```

## ✦ 3. Interaction Features and Polynomial Features

### 3.1 The Problem: Non-Linearity in Linear Models

A linear model computes $\hat{y} = w^T x + b$. It cannot capture the interaction "feature $x_1$ matters _only when_ $x_2 > 0$." No single weight can do this. But if you add the feature $x_1 \cdot x_2$ to the input, a linear model _can_ capture this interaction.

### 3.2 Polynomial Features

The degree-$d$ polynomial feature map for a single feature $x$:

$$\phi(x) = [1, x, x^2, \ldots, x^d]$$

For $p$ features, the degree-$d$ polynomial map produces $\binom{p+d}{d}$ features — all monomials up to degree $d$. This grows explosively: $p = 10, d = 2$ gives 66 features; $p = 100, d = 2$ gives 5151.

**The bias-variance view:** Adding polynomial features reduces bias (can fit nonlinear functions) but increases variance (more parameters, easier to overfit). Must be paired with regularisation.

```python
def polynomial_features(x, degree):
    """1D case: returns [1, x, x^2, ..., x^degree]."""
    return np.array([x**d for d in range(degree + 1)]).T

# Multi-feature case — use sklearn
from sklearn.preprocessing import PolynomialFeatures
poly = PolynomialFeatures(degree=2, include_bias=False)
X_poly = poly.fit_transform(X)  # adds x1^2, x2^2, x1*x2, etc.
```

**Geometric interpretation:** A degree-$d$ polynomial in $\mathbb{R}^1$ can fit at most $d$ "bends." In $\mathbb{R}^p$, the polynomial feature map lifts the data to a higher-dimensional space where linear classifiers become more powerful — this is the same idea as the kernel trick in SVMs, but made explicit.

### 3.3 Interaction Features

For two features $x_i$ and $x_j$, the interaction feature is simply $x_i \cdot x_j$. A linear model with this feature can represent:

$$\hat{y} = w_1 x_1 + w_2 x_2 + w_{12} x_1 x_2 + b$$

This is equivalent to: the effect of $x_1$ on $\hat{y}$ is $w_1 + w_{12} x_2$ — it depends on $x_2$. This is the definition of an interaction.

**Domain-driven interaction features are almost always better than automated polynomial expansion.** Domain knowledge tells you which variables interact; automated expansion produces $O(p^2)$ features, most of which are noise.

Examples of domain-driven interactions:

- In house price prediction: `size × neighbourhood_quality`
- In medical risk: `age × smoker_indicator`
- In advertising: `time_on_site × user_segment`

## ✦ 4. Text Features

### 4.1 Bag of Words

Represent a document as a vector of word counts:

$$\phi(d)_j = \text{count of word } j \text{ in document } d$$

The vocabulary $V$ is the set of all words seen during training. The feature vector has dimension $|V|$, typically 10,000–100,000, and is sparse.

**Problems:** Ignores word order and document length. A document twice as long produces counts twice as high.

---

### 4.2 TF-IDF

**TF (Term Frequency):** Normalises for document length:

$$\text{TF}(t, d) = \frac{\text{count}(t, d)}{\sum_{t'} \text{count}(t', d)}$$

**IDF (Inverse Document Frequency):** Down-weights terms that appear in many documents (common words like "the", "and" are not discriminative):

$$\text{IDF}(t) = \log \frac{N + 1}{|{d : t \in d}| + 1} + 1$$

where $N$ is the total number of documents. The $+1$ smoothing prevents division by zero.

**TF-IDF:**

$$\text{TF-IDF}(t, d) = \text{TF}(t, d) \cdot \text{IDF}(t)$$

**Intuition:** A word that appears frequently in a specific document but rarely across the corpus is probably important to that document. "Mitochondria" in a biology paper is more informative than "the."

```python
import numpy as np
from collections import Counter

def tfidf(corpus):
    """Minimal TF-IDF from scratch."""
    N = len(corpus)
    tokenised = [doc.lower().split() for doc in corpus]
    
    # Build vocabulary
    vocab = sorted(set(w for doc in tokenised for w in doc))
    w2i  = {w: i for i, w in enumerate(vocab)}
    
    # IDF: log((N+1)/(df+1)) + 1
    df = np.zeros(len(vocab))
    for doc in tokenised:
        for w in set(doc):
            df[w2i[w]] += 1
    idf = np.log((N + 1) / (df + 1)) + 1
    
    # TF-IDF matrix
    X = np.zeros((N, len(vocab)))
    for i, doc in enumerate(tokenised):
        counts = Counter(doc)
        total  = sum(counts.values())
        for w, c in counts.items():
            if w in w2i:
                X[i, w2i[w]] = (c / total) * idf[w2i[w]]
    
    # L2-normalise rows (sklearn default)
    norms = np.linalg.norm(X, axis=1, keepdims=True) + 1e-8
    return X / norms, vocab

corpus = [
    "the cat sat on the mat",
    "the dog sat on the log",
    "cats and dogs are different animals",
]
X_tfidf, vocab = tfidf(corpus)
```

### 4.3 N-grams

Extend bag-of-words to sequences of $n$ words:

- Unigrams: `["the", "cat", "sat"]`
- Bigrams: `["the cat", "cat sat", "sat on"]`
- Trigrams: `["the cat sat", "cat sat on"]`

N-grams capture local word order at the cost of exponentially larger vocabulary. Bigrams are almost always helpful; trigrams are situationally useful.

```python
def ngrams(tokens, n):
    return [" ".join(tokens[i:i+n]) for i in range(len(tokens) - n + 1)]

tokens = ["the", "cat", "sat", "on", "the", "mat"]
print(ngrams(tokens, 2))
# ['the cat', 'cat sat', 'sat on', 'on the', 'the mat']
```

**TF-IDF + bigrams** is a strong baseline for text classification — often competitive with neural approaches on smaller datasets.

---

### 4.4 Connection to Deep Learning: Word Embeddings

TF-IDF is sparse and high-dimensional. Word embeddings (Word2Vec, GloVe, fastText) learn dense, low-dimensional representations (typically 50–300 dimensions) where semantic similarity corresponds to geometric proximity:

$$\cos(\phi(\text{"king"}) - \phi(\text{"man"}) + \phi(\text{"woman"}), \phi(\text{"queen"})) \approx 1$$

This is feature engineering at scale: rather than hand-crafting features, the embedding learns to compress the distributional structure of language into a dense vector. The features are not engineered by a human — they are _learned_ from a large corpus. This is the precursor to the representation learning paradigm.

## ✦ 5. Date and Time Features

### 5.1 The Problem with Raw Timestamps

A Unix timestamp (seconds since epoch) is a large integer. Models will see no pattern between 1609459200 (Jan 1 2021 midnight) and 1609545600 (Jan 2 2021 midnight) and 1641081600 (Jan 2 2022 midnight) — the distances in raw timestamp space do not correspond to relevant cyclical patterns.

### 5.2 Extracting Structural Features

```python
import pandas as pd
import numpy as np

df['timestamp'] = pd.to_datetime(df['timestamp'])

# Extracting components
df['hour']       = df['timestamp'].dt.hour         # 0–23
df['day_of_week']= df['timestamp'].dt.dayofweek    # 0=Monday
df['month']      = df['timestamp'].dt.month        # 1–12
df['is_weekend'] = df['day_of_week'].isin([5, 6]).astype(int)
df['quarter']    = df['timestamp'].dt.quarter

# Days since reference event (e.g., product launch)
launch = pd.Timestamp('2020-01-01')
df['days_since_launch'] = (df['timestamp'] - launch).dt.days
```

---

### 5.3 Cyclical Encoding (Crucial)

Hours, days of week, and months are _cyclical_: hour 23 is close to hour 0, not far from it. If you encode `hour` as an integer, the model sees a discontinuity at midnight. The fix is cyclical encoding:

$$\phi_{\sin}(t) = \sin!\left(\frac{2\pi t}{T}\right), \quad \phi_{\cos}(t) = \cos!\left(\frac{2\pi t}{T}\right)$$

where $T$ is the period (24 for hours, 7 for days, 12 for months).

**Why two dimensions?** A single $\sin$ is not injective on $[0, T)$ — $\sin(x) = \sin(\pi - x)$, so hours 3 and 9 would have the same encoding. The pair $(\sin, \cos)$ uniquely identifies any point on the circle. Geometrically, this represents the time as a point on the unit circle.

```python
def cyclical_encode(series, period):
    """Returns two columns: sin and cos encoding."""
    theta = 2 * np.pi * series / period
    return np.sin(theta), np.cos(theta)

hour = df['hour'].values
df['hour_sin'], df['hour_cos'] = cyclical_encode(hour, period=24)

day = df['day_of_week'].values
df['dow_sin'], df['dow_cos'] = cyclical_encode(day, period=7)
```

**Visualisation of cyclical encoding for hours:**

```
Hour 0 (midnight): (sin=0,  cos=1)   → top of circle
Hour 6 (6am):      (sin=1,  cos=0)   → right
Hour 12 (noon):    (sin=0,  cos=-1)  → bottom
Hour 18 (6pm):     (sin=-1, cos=0)   → left
Hour 23 → Hour 0: small distance in (sin,cos) space ✓

Without cyclical: |23 - 0| = 23  (large, wrong)
With cyclical:    ‖(sin23°, cos23°) - (sin0°, cos0°)‖ ≈ 0.26 (small, correct)
```

## ✦ 6. Feature Crossing

### 6.1 The Idea

A feature cross is an explicit interaction of two or more features, often categorical:

$$\phi(x_1, x_2) = (x_1 = v_1) \wedge (x_2 = v_2)$$

If `city ∈ {NYC, LA, Chicago}` and `time_of_day ∈ {morning, afternoon, evening}`, the cross `city × time_of_day` produces 9 binary features — one for each combination. A linear model using these features can learn that coffee demand in NYC in the morning is different from all other combinations, even if it cannot learn that from `city` and `time_of_day` separately.

### 6.2 In Large-Scale Linear Models

Feature crosses are the dominant feature engineering technique in large-scale advertising and recommendation systems. Google's Wide & Deep architecture (2016) explicitly uses a wide linear model with feature crosses alongside a deep neural network. The crosses handle memorisation (specific patterns); the neural network handles generalisation.

```python
# Implement cross manually
def feature_cross(x1_ohe, x2_ohe):
    """
    x1_ohe: (n, k1) one-hot matrix
    x2_ohe: (n, k2) one-hot matrix
    Returns: (n, k1*k2) cross matrix
    """
    n = x1_ohe.shape[0]
    k1, k2 = x1_ohe.shape[1], x2_ohe.shape[1]
    cross = np.zeros((n, k1 * k2))
    for i in range(k1):
        for j in range(k2):
            cross[:, i * k2 + j] = x1_ohe[:, i] * x2_ohe[:, j]
    return cross
```

## ✦ 7. Domain-Specific Examples

### 7.1 Tabular / Finance

|Raw Feature|Engineered Feature|Rationale|
|---|---|---|
|`open_price, close_price`|`return = (close - open) / open`|Relative change is stationary|
|`price_t, price_{t-1}`|`log(price_t / price_{t-1})`|Log returns approximately additive|
|`revenue, employees`|`revenue / employees`|Revenue per employee is scale-invariant|
|`date`|`days_to_quarter_end`|Captures reporting seasonality|
|`transaction_amount`|`log(amount)`|Compresses heavy tail|

### 7.2 Geospatial

|Raw Feature|Engineered Feature|Rationale|
|---|---|---|
|`latitude, longitude`|`distance_to_CBD`|Captures urban gradient|
|`lat, lon`|Haversine distance to reference points|Non-Euclidean geometry|
|`zip_code`|`target_encode(zip_code, median_price)`|Encodes neighbourhood effect|
|`lat, lon`|`cluster_id` (from K-Means)|Groups into neighbourhoods|

### 7.3 Images (Before Deep Learning Made This Unnecessary)

- HOG (Histogram of Oriented Gradients): captures edge structure
- SIFT (Scale-Invariant Feature Transform): keypoints invariant to scale and rotation
- Pixel statistics per colour channel: mean, std, percentiles

Deep learning replaced hand-crafted image features almost entirely. Convolutional layers _learn_ features equivalent to HOG/SIFT and beyond.

## ✦ 8. The Feature Engineering Workflow

```
1. Understand the domain
   - What does each raw variable actually measure?
   - What are the units? Are they arbitrary or meaningful?
   - What domain knowledge suggests relevant interactions?

2. Explore the data (EDA)
   - Distribution of each feature (histogram, boxplot)
   - Pairwise correlations (but correlation ≠ causation)
   - Scatter plots of feature vs. target
   - Look for: skew, outliers, bimodality, floor/ceiling effects

3. Engineer features
   - Apply transformations to fix distributional issues
   - Encode categoricals appropriately
   - Extract temporal and spatial structure
   - Create interactions from domain knowledge

4. Validate each feature
   - Does it help on cross-validation? (not just training accuracy)
   - Is it available at inference time? (no label leakage)
   - Is it stable? (computed the same way on future data)

5. Select features (→ [[03.Feature Selection]])
   - Remove redundant or harmful features
   - Use mutual information, correlation, or model-based importance
```

## ✦ 9. Connection to AI, Machine Learning, and Deep Learning

### 9.1 The Representation Learning Perspective

Feature engineering is _manual representation learning_. Deep learning largely replaces it by learning hierarchical representations automatically:

|Layer in CNN|Analogous Hand-Crafted Feature|
|---|---|
|Edge detectors (L1)|Sobel filter|
|Texture detectors (L2)|Gabor wavelets, HOG|
|Object parts (L3)|Keypoint descriptors|
|Semantic concepts (L4)|— (no hand-crafted equivalent)|

The triumph of deep learning on perception tasks (vision, speech, text) is largely a triumph over feature engineering — the model learns the representation. But on _tabular_ data, feature engineering remains important because:

1. Tabular data is typically low-dimensional — not enough raw structure for deep networks to discover features automatically.
2. Domain knowledge is more directly applicable to tabular data.
3. Tree-based models (which dominate tabular ML benchmarks) are not fully invariant to feature representation — good features still help.

### 9.2 Embeddings as Learnable Feature Engineering

Categorical embeddings in neural networks are a middle ground: instead of one-hot encoding (manual, sparse) or target encoding (manual, dense), the model learns a dense $d$-dimensional representation for each category as part of training:

$$\text{Embedding}(\text{zip_code}) = E[\text{zip_code}] \in \mathbb{R}^d$$

where $E \in \mathbb{R}^{|\text{vocab}| \times d}$ is a learnable matrix. The embedding captures the structure of the category in the task-relevant space. This is exactly what Word2Vec does for words.

### 9.3 Feature Engineering and Inductive Bias

Every feature engineering choice is equivalent to specifying an inductive bias — an assumption about the form of the true function. Applying a log transform says "I believe this relationship is multiplicative." Adding cyclical features says "I believe this feature has periodic structure with period $T$." Adding an interaction feature says "I believe these two variables have a non-additive joint effect."

In deep learning, these biases are instead baked into the architecture: convolutional layers assume translation equivariance, recurrent layers assume temporal order matters, attention assumes arbitrary pairwise dependencies. The choice of architecture is feature engineering at a meta-level.

## ✦ 10. Common Mistakes

|Mistake|Why It Fails|
|---|---|
|Computing TF-IDF on full corpus before splitting|IDF uses test document frequencies — leakage|
|One-hot encoding with no intercept drop|Perfect multicollinearity — singular gram matrix|
|Label-encoding nominal features|Imposes false ordinal structure|
|Applying target encoding without smoothing|Rare categories encode the target exactly — overfit|
|Adding features without cross-validation check|Can reduce performance despite "looking informative"|
|Engineering features at inference time differently than at training|Distribution shift — immediate production failure|
|Log-transforming features with zeros|$\log(0) = -\infty$ — use $\log(x + 1)$|
|Cyclical features without both sin and cos|Encoding is not injective — same encoding for two different times|

## ✦ Key Facts to Derive

**Fact 1:** For a linear model with polynomial features of degree $d$ in $p$ original features, the number of features is $\binom{p+d}{d}$. For $p=10, d=3$: $\binom{13}{3} = 286$.

**Fact 2:** TF-IDF with L2 normalisation is equivalent to computing cosine similarity between documents in the TF-IDF space. This is why it is the default for document similarity tasks.

**Fact 3:** The pair $(\sin(2\pi t / T), \cos(2\pi t / T))$ is the unique unit-circle embedding of the point at angle $2\pi t / T$. The Euclidean distance between two such embeddings equals $2|\sin(\pi \Delta t / T)|$, which is small when $\Delta t$ is small modulo $T$.

**Fact 4:** Target encoding with Laplace smoothing is MAP estimation under a Beta prior (for binary targets) or Normal prior (for continuous targets). The smoothing parameter $\lambda$ controls the strength of the prior.
