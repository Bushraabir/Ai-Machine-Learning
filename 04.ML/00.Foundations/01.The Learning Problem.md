# 01. The Learning Problem

> "A computer program is said to learn from experience E with respect to some class of tasks T and performance measure P, if its performance at tasks in T, as measured by P, improves with experience E." — Tom Mitchell, _Machine Learning_, 1997

Part of: [[04.Machine Learning]] > Part 0 — Foundations Next: [[02. Statistical Learning Theory]]

---

## 1. What Is Learning?

Before writing a single line of code or touching a dataset, the question has to be asked precisely: what does it mean for a machine to _learn_?

The intuitive answer — "it gets better with experience" — is Mitchell's definition above. But intuition is not enough. To build algorithms, to prove guarantees, to know when things will fail, we need a mathematical formalisation.

The formalisation has three parts.

---

## 2. The Three Components of a Learning Problem

Every learning problem can be decomposed as a triple $(T, E, P)$.

**Task $T$** — what the system is trying to do. Examples: classify an email as spam or not spam; predict tomorrow's temperature; generate a caption for an image; recommend a product.

**Experience $E$** — what the system has access to during training. Examples: a dataset of labelled emails; historical temperature records; image-caption pairs; user click logs.

**Performance measure $P$** — how we evaluate whether the system is doing $T$ well. Examples: classification accuracy; mean squared error; BLEU score; click-through rate.

The learning problem is then: given $E$, improve performance on $T$ as measured by $P$.

This decomposition matters because it forces precision. "Build a model that predicts customer churn" is not a well-posed learning problem. "Train a binary classifier ($T$) on 12 months of transaction history ($E$) to maximise F1 score on a held-out test set ($P$)" is.

---

## 3. The Formal Setup

Let the input space be $\mathcal{X}$ (e.g., $\mathbb{R}^d$, the space of all images, the space of all sentences) and the output space be $\mathcal{Y}$ (e.g., $\mathbb{R}$ for regression, ${0,1}$ for binary classification, ${1,\ldots,k}$ for multi-class).

We assume there exists an unknown target function:

$$f^* : \mathcal{X} \to \mathcal{Y}$$

We do not know $f^*$. If we did, the problem would be solved. Instead, we observe a training set:

$$\mathcal{D} = {(x_1, y_1), (x_2, y_2), \ldots, (x_n, y_n)}$$

where each $(x_i, y_i)$ is drawn i.i.d. from some unknown joint distribution $P_{\mathcal{X} \times \mathcal{Y}}$.

The goal of learning is to find a function $h : \mathcal{X} \to \mathcal{Y}$ that approximates $f^*$ well — not just on the training data we have seen, but on new, unseen inputs drawn from the same distribution.

This last point is the entire difficulty. Fitting training data is easy. Generalising is hard.

### 3.1 The Hypothesis Class

We do not search over all possible functions $h : \mathcal{X} \to \mathcal{Y}$. The space of all such functions is astronomically large (uncountably infinite for continuous inputs). Instead, we restrict the search to a **hypothesis class** $\mathcal{H}$:

$$\mathcal{H} = {h_\theta : \mathcal{X} \to \mathcal{Y} \mid \theta \in \Theta}$$

where $\Theta$ is a parameter space. Learning then becomes: find the $\theta \in \Theta$ such that $h_\theta$ best approximates $f^*$ on unseen data.

Examples of hypothesis classes:

|Model|Hypothesis class|
|---|---|
|Linear regression|$\mathcal{H} = {x \mapsto w^Tx + b \mid w \in \mathbb{R}^d, b \in \mathbb{R}}$|
|Logistic regression|$\mathcal{H} = {x \mapsto \sigma(w^Tx + b) \mid w \in \mathbb{R}^d, b \in \mathbb{R}}$|
|Depth-$k$ decision tree|$\mathcal{H} = {\text{all binary trees with depth} \leq k}$|
|2-layer neural network|$\mathcal{H} = {x \mapsto W_2 \sigma(W_1 x + b_1) + b_2}$|

The choice of $\mathcal{H}$ is arguably the most consequential modelling decision you make. It encodes what you believe about the structure of $f^*$ before seeing any data. This belief is called **inductive bias**.

---

## 4. Inductive Bias

No algorithm can learn without assumptions. This is a theorem, not a matter of opinion. An algorithm with no inductive bias treats every unseen input as equally likely to have any output — which means it has learned nothing from the training data.

**Definition.** The inductive bias of a learning algorithm is the set of assumptions it makes about the target function $f^*$, beyond what the training data alone can establish.

Examples:

- Linear regression assumes $f^*$ is a linear function of the inputs. It will fail systematically on data generated by a quadratic.
- KNN with $k=1$ assumes $f^*$ is locally constant — nearby inputs have the same output. It makes no assumption about global structure.
- A Gaussian Process with an RBF kernel assumes $f^_$ is smooth — small changes in $x$ produce small changes in $f^_(x)$.
- A convolutional neural network assumes $f^*$ is translation-equivariant — the same pattern at different positions should be detected in the same way. This is why CNNs work on images.

The practical consequence: **inductive bias is not a flaw to be minimised — it is a design choice to be made deliberately.** Choosing a linear model for a nonlinear problem is not being rigorous; it is being wrong. Choosing a rich model for a simple problem invites overfitting.

The right inductive bias is the one that matches the structure of the actual problem.

---

## 5. The No Free Lunch Theorem

The No Free Lunch (NFL) theorem (Wolpert, 1996) formalises why inductive bias is unavoidable.

**Theorem (informal).** Averaged over all possible target functions $f^* : \mathcal{X} \to \mathcal{Y}$, every learning algorithm performs equally well.

More precisely: for any two algorithms $A$ and $B$, if you average their out-of-sample error over all possible data-generating distributions, they are equal.

**What this means.** There is no universally superior algorithm. An algorithm that outperforms all others on some problems must underperform on others. The gains are always elsewhere.

**What this does not mean.** The NFL theorem is not an argument for nihilism or for ignoring algorithm selection. It applies when averaged over _all_ possible functions, including completely random and pathological ones. In practice, we are not interested in all possible functions — we are interested in the specific structure of the physical, biological, or social processes that generated our data. Those processes have regularity. Exploiting that regularity is what machine learning is.

The NFL theorem does not say: "don't bother choosing algorithms." It says: "your choice of algorithm embeds an assumption about the problem. Make that assumption explicit and make sure it's reasonable."

A linear model assumes the world is linear. If it is, the linear model wins. If it is not, the linear model loses, and something more flexible is needed. There is no free lunch. You pay with assumptions.

## 6. Types of Learning

The taxonomy below is about the _structure of experience_ $E$ available to the learner.

### 6.1 Supervised Learning

The training set consists of input-output pairs:

$$\mathcal{D} = {(x_i, y_i)}_{i=1}^n$$

The labels $y_i$ are provided by a supervisor (a human annotator, a measurement instrument, a simulation). The task is to learn a mapping from inputs to outputs.

Sub-cases:

- **Regression** — $\mathcal{Y} = \mathbb{R}$ or $\mathbb{R}^k$. Predict a continuous quantity.
- **Classification** — $\mathcal{Y} = {1, \ldots, k}$. Predict a discrete category.
- **Structured prediction** — $\mathcal{Y}$ is a structured object (sequence, tree, graph). Examples: machine translation, dependency parsing.

The fundamental challenge: labels are expensive. Human annotation does not scale. This is why the field has invested heavily in the paradigms below.

### 6.2 Unsupervised Learning

The training set consists of inputs only:

$$\mathcal{D} = {x_i}_{i=1}^n$$

There are no labels. The task is to discover the underlying structure of the data distribution $P(\mathcal{X})$.

Sub-tasks:

- **Clustering** — partition inputs into coherent groups.
- **Dimensionality reduction** — find a low-dimensional representation that preserves structure.
- **Density estimation** — learn an explicit or implicit model of $P(x)$.
- **Representation learning** — learn features useful for downstream tasks.

Unsupervised learning is harder to evaluate (there is no ground truth) but scales to vast amounts of unlabelled data.

### 6.3 Semi-Supervised Learning

The training set contains a small number of labelled examples and a large number of unlabelled examples:

$$\mathcal{D} = {(x_i, y_i)}_{i=1}^m \cup {x_j}_{j=m+1}^n \quad \text{where } m \ll n$$

The key assumption: unlabelled data carries information about the structure of $P(\mathcal{X})$, which can improve learning of the conditional $P(\mathcal{Y} | \mathcal{X})$.

This is the most realistic setting for many practical problems, where labelling is expensive but raw data is abundant.

### 6.4 Self-Supervised Learning

A special case of unsupervised learning where the supervision signal is generated automatically from the data itself — no human annotation required.

Examples:

- **Masked language modelling (BERT):** mask 15% of tokens, predict the masked tokens. The "labels" are the original tokens.
- **Next-token prediction (GPT):** predict the next token given all previous tokens.
- **Contrastive learning (SimCLR):** create two augmented views of the same image, train a network to represent them similarly and represent different images differently.

Self-supervised learning has become the dominant paradigm for pretraining large models. The pretrained representations are then fine-tuned on small labelled datasets — this is _transfer learning_.

### 6.5 Reinforcement Learning

There is no fixed dataset. An agent interacts with an environment, takes actions, and receives scalar reward signals. The task is to learn a policy $\pi : \mathcal{S} \to \mathcal{A}$ (from states to actions) that maximises cumulative discounted reward:

$$G_t = \sum_{k=0}^{\infty} \gamma^k r_{t+k}, \quad \gamma \in [0,1)$$

The challenge: the agent must explore to discover what actions are rewarding, but must also exploit what it already knows to collect reward. This exploration-exploitation tradeoff is fundamental.

RL differs from supervised learning in two crucial ways:

1. There is no supervisor providing correct actions — only a reward signal that may be delayed and noisy.
2. The data distribution depends on the agent's own behaviour — the agent influences what it sees.

## 7. The Generalisation Problem

Return to the core question: why is generalisation hard?

Suppose we have a training set $\mathcal{D}$ of $n$ examples. We can always find a hypothesis $h \in \mathcal{H}$ that perfectly fits $\mathcal{D}$ — just memorise the training set. But memorisation is not learning. A memorising model has zero training error and potentially very high test error.

This is **overfitting**: the model has learned the training data rather than the underlying pattern.

The opposite failure is **underfitting**: the hypothesis class $\mathcal{H}$ is too restricted to capture the true pattern even with unlimited data. A linear model trying to fit a cubic relationship will underfit regardless of how much data it sees.

The tension between these two failures is the central problem of supervised learning. It is studied formally through:

- **Bias-variance decomposition** — see [[02. Statistical Learning Theory]]
- **VC theory and PAC learning** — see [[02. Statistical Learning Theory]]
- **Regularisation** — see [[04.Bias Variance Tradeoff]]

For now, the key intuition: the size of the hypothesis class $\mathcal{H}$ must be matched to the amount of data $n$. Too small $\mathcal{H}$ relative to the complexity of $f^*$ → underfitting. Too large $\mathcal{H}$ relative to $n$ → overfitting.

More data always helps. More flexible models help only when data is sufficient to constrain them.

## 8. A Taxonomy of ML Problems

The table below is not exhaustive. It is a map for orientation.

|Dimension|Options|
|---|---|
|**Supervision**|Supervised, Unsupervised, Semi-supervised, Self-supervised, Reinforcement|
|**Output type**|Continuous (regression), Discrete (classification), Structured (sequence, graph, tree)|
|**Output cardinality**|Binary, Multi-class, Multi-label, Ordinal|
|**Data modality**|Tabular, Image, Text, Audio, Time series, Graph|
|**Online vs. batch**|Batch (fixed dataset), Online (stream of data, model updates incrementally)|
|**Stationary vs. non-stationary**|Stationary ($P$ is fixed), Non-stationary (distribution shifts over time — concept drift)|
|**Single-task vs. multi-task**|Single task, Multi-task (shared representation across tasks), Transfer learning|

Identifying where your problem sits in this taxonomy is the first step before choosing any algorithm.

## 9. The Machine Learning Pipeline

From raw data to a deployed model, the pipeline has a fixed structure — even when the algorithms change.

```
Problem framing
      |
      v
Data collection and storage
      |
      v
Exploratory data analysis (EDA)
      |
      v
Data preprocessing and feature engineering
      |
      v
Model selection (choose H)
      |
      v
Training (find theta* in H)
      |
      v
Evaluation (measure P on held-out data)
      |
      v
Error analysis (understand failure modes)
      |
      v
Iteration
      |
      v
Deployment and monitoring
```

Each stage is a note in this vault. This note is concerned only with the first stage: understanding what kind of problem you have and what the formal learning framework is.

## 10. Worked Example — Framing a Problem

**Scenario.** You work at a bank. The task is to detect fraudulent transactions.

Walk through the $(T, E, P)$ decomposition:

**Task $T$:** Binary classification. Given a transaction $x$ (amount, merchant category, location, time, account history, ...), predict $y \in {0, 1}$ where $y = 1$ means fraud.

**Experience $E$:** Historical transaction records. Each record has been reviewed and labelled as fraudulent or legitimate by a human fraud team. Note: the dataset is severely class-imbalanced — fraud is rare, perhaps 0.1% of transactions.

**Performance measure $P$:** Not accuracy. With 0.1% fraud rate, a classifier that always predicts "not fraud" achieves 99.9% accuracy while being completely useless. Better choices:

- **Precision** — of the transactions flagged as fraud, how many are actually fraud? (Minimise false accusations)
- **Recall** — of all actual fraud, how many did we catch? (Minimise missed fraud)
- **F1 score** — harmonic mean of precision and recall
- **AUC-PR** — area under the precision-recall curve — the right choice here
- Business cost: false negatives (missed fraud) cost more than false positives (wrongly blocked transactions) — so weight recall higher

**Hypothesis class $\mathcal{H}$:** Not yet determined. But the problem has structured tabular features, moderate $n$ (millions of transactions), known class imbalance, and a need for some interpretability (for regulatory compliance). This points toward gradient boosting as a strong first candidate — see [[ML Algorithms Quick Reference]].

**Inductive bias:** Gradient boosting assumes the decision boundary is a piecewise-constant function that can be approximated by an additive combination of decision trees. For tabular fraud detection with threshold-based rules and interaction effects between features, this is a defensible assumption.

This is what proper problem framing looks like. The algorithm choice is almost a secondary concern once the framing is precise.
