
> _"A single train/val split gives you one noisy estimate of generalisation error. Cross-validation gives you $k$ of them — and their mean has lower variance by a factor of $k$. The price is $k$ times the computation. Usually worth it."_

---

## ✦ The Problem with a Fixed Validation Split

From [[01.Train Val Test|Train, Validation, and Test Sets]], the validation error $\hat{R}_{\text{val}}(\hat{f})$ is an unbiased estimator of $R(\hat{f})$ — but it has variance. How much variance?

For a fixed split with $n_{\text{val}}$ samples, the variance of the validation error estimator is roughly:

$$\text{Var}\left[\hat{R}_{\text{val}}\right] \approx \frac{R(1-R)}{n_{\text{val}}}$$

Two problems compound each other when $n$ is small:

1. **High variance:** small $n_{\text{val}}$ means noisy estimates — your "best" model may be best by luck
2. **Training set shrinkage:** what you don't use for validation, you use for training — small $n_{\text{val}}$ means large $n_{\text{train}}$, but large $n_{\text{val}}$ means you're training on less data, so $\hat{f}$ itself is worse

Cross-validation resolves this tension: **use all the data for training (at different times) and all the data for validation (at different times).**

---

## ✦ $k$-Fold Cross-Validation

### The Algorithm

Partition $\mathcal{D}$ into $k$ disjoint folds $F_1, F_2, \ldots, F_k$ of approximately equal size $\lfloor n/k \rfloor$.

For $i = 1, 2, \ldots, k$:

1. Train on $\mathcal{D} \setminus F_i$ — all folds except fold $i$
2. Evaluate on $F_i$ — the held-out fold
3. Record $\hat{R}_i = \hat{R}(F_i)$

The cross-validation estimate of generalisation error:

$$\hat{R}_{\text{CV}} = \frac{1}{k} \sum_{i=1}^{k} \hat{R}_i$$

Each fold produces a model trained on $\frac{k-1}{k} \cdot n$ samples. You produce $k$ models total. None of them is your final model — they exist solely to estimate $R$.

```
Dataset:  [====|====|====|====|====]
           F1   F2   F3   F4   F5

Iter 1:   [VVVV|----|----|----|----|]  train on 2–5, validate on 1
Iter 2:   [----|VVVV|----|----|----|]  train on 1,3–5, validate on 2
Iter 3:   [----|----|VVVV|----|----|]  train on 1–2,4–5, validate on 3
Iter 4:   [----|----|----| VVVV|----|]  train on 1–3,5, validate on 4
Iter 5:   [----|----|----|----| VVVV]  train on 1–4, validate on 5

Final estimate: mean of 5 validation errors
```

### Choosing $k$

The bias-variance tradeoff of $k$ is real and important:

|$k$|Training set size|Bias of $\hat{R}_{\text{CV}}$|Variance of $\hat{R}_{\text{CV}}$|Computation|
|---|---|---|---|---|
|Small (e.g., $k=2$)|$n/2$|**High** (models trained on half the data are worse than the final model)|Low|Low|
|Large (e.g., $k=n$)|$n-1$|**Low** (models trained on nearly all data; LOOCV)|**High** (folds are highly correlated)|Very high|
|$k=5$ or $k=10$|$4n/5$ or $9n/10$|Low|Moderate|Moderate|

**The standard recommendation is $k = 5$ or $k = 10$**, validated empirically by Kohavi (1995) across many datasets. $k=10$ is more common in the literature; $k=5$ is preferred when computation is expensive.

### Why Large $k$ Has High Variance — The Correlation Argument

This is subtle and worth deriving. The variance of the mean of $k$ terms $\hat{R}_1, \ldots, \hat{R}_k$ is:

$$\text{Var}\left[\frac{1}{k}\sum_i \hat{R}_i\right] = \frac{1}{k^2} \sum_i \text{Var}[\hat{R}_i] + \frac{2}{k^2} \sum_{i < j} \text{Cov}[\hat{R}_i, \hat{R}_j]$$

If the $\hat{R}_i$ were independent, the covariance terms vanish and variance shrinks as $1/k$. But they are **not independent** — all $k$ models are trained on overlapping subsets of $\mathcal{D}$.

For large $k$ (e.g., LOOCV), each pair of models shares $n-2$ training points out of $n-1$ — they are nearly identical. The covariance $\text{Cov}[\hat{R}_i, \hat{R}_j]$ is large and positive. The variance of $\hat{R}_{\text{CV}}$ does **not** shrink as $1/k$ — it saturates.

Bengio & Grandvalet (2004) proved that there is no unbiased estimator of the variance of $\hat{R}_{\text{CV}}$ — a sobering theoretical result. In practice, people report the standard deviation across folds as a proxy for uncertainty, but treat it with appropriate scepticism.

## ✦ Implementation From Scratch

```python
import numpy as np
from sklearn.datasets import load_breast_cancer
from sklearn.linear_model import LogisticRegression
from sklearn.preprocessing import StandardScaler

def kfold_cv(model_fn, X, y, k=5, random_state=42):
    """
    k-fold cross-validation from scratch.
    model_fn: callable that returns a fresh (unfitted) model
    Returns: array of per-fold validation scores
    """
    n = len(X)
    rng = np.random.default_rng(random_state)
    indices = rng.permutation(n)  # shuffle before folding
    
    fold_sizes = np.full(k, n // k)
    fold_sizes[:n % k] += 1       # distribute remainder
    
    scores = []
    start = 0
    for fold_size in fold_sizes:
        val_idx = indices[start : start + fold_size]
        train_idx = np.concatenate([indices[:start], indices[start + fold_size:]])
        
        X_train, y_train = X[train_idx], y[train_idx]
        X_val, y_val = X[val_idx], y[val_idx]
        
        # ⚠ Preprocessing must happen inside the loop
        scaler = StandardScaler().fit(X_train)
        X_train_s = scaler.transform(X_train)
        X_val_s   = scaler.transform(X_val)
        
        model = model_fn()
        model.fit(X_train_s, y_train)
        scores.append(model.score(X_val_s, y_val))
        
        start += fold_size
    
    return np.array(scores)


# Usage
X, y = load_breast_cancer(return_X_y=True)
scores = kfold_cv(lambda: LogisticRegression(max_iter=1000), X, y, k=10)

print(f"Per-fold: {scores.round(4)}")
print(f"Mean:     {scores.mean():.4f}")
print(f"Std:      {scores.std():.4f}")

# Verify against sklearn
from sklearn.model_selection import cross_val_score
from sklearn.pipeline import Pipeline

pipe = Pipeline([("sc", StandardScaler()), ("clf", LogisticRegression(max_iter=1000))])
sk_scores = cross_val_score(pipe, X, y, cv=10)
print(f"\nsklearn mean: {sk_scores.mean():.4f}")  # Should match
```

**The critical invariant inside the loop:** the scaler is fitted on `X_train` and _only_ applied (transformed) on `X_val`. Fitting the scaler outside the loop on all of `X` is the preprocessing leakage mistake — see [[01.Train Val Test|Train, Validation, and Test Sets]].

## ✦ Leave-One-Out Cross-Validation (LOOCV)

LOOCV is $k$-fold CV with $k = n$: each fold contains exactly one sample.

$$\hat{R}_{\text{LOO}} = \frac{1}{n} \sum_{i=1}^{n} \ell!\left(\hat{f}^{(-i)}(x_i),, y_i\right)$$

where $\hat{f}^{(-i)}$ is the model trained on all data except $(x_i, y_i)$.

### When LOOCV Makes Sense

- **Small datasets** ($n < 50$): every sample is precious; LOOCV wastes none on validation
- **Expensive models** where you want maximum training data in each fold
- **Linear models**: LOOCV has a closed-form shortcut (see below) — $O(n)$ not $O(n^2)$

### The LOOCV Shortcut for Linear Models

For ordinary least squares, the LOOCV error can be computed from a single fit on the full dataset:

$$\hat{R}_{\text{LOO}} = \frac{1}{n} \sum_{i=1}^{n} \left(\frac{y_i - \hat{y}_i}{1 - h_{ii}}\right)^2$$

where $h_{ii} = [H]_{ii}$ is the $i$-th diagonal of the **hat matrix** $H = X(X^TX)^{-1}X^T$.

**Derivation sketch.** When you remove sample $i$ and refit OLS, the leave-one-out prediction at $x_i$ can be expressed in terms of the full-data residual $e_i = y_i - \hat{y}_i$ and the leverage $h_{ii}$:

$$\hat{f}^{(-i)}(x_i) = \hat{y}_i - \frac{h_{ii}}{1 - h_{ii}} e_i$$

So the LOO residual is:

$$y_i - \hat{f}^{(-i)}(x_i) = e_i + \frac{h_{ii}}{1 - h_{ii}} e_i = \frac{e_i}{1 - h_{ii}}$$

This is a remarkable result: you get the LOO error for _all_ $n$ leave-one-out splits from _one_ matrix factorisation. No loops needed.

**Leverage $h_{ii}$** ranges from $1/n$ to 1. High-leverage points (outliers in feature space) have $h_{ii} \to 1$, meaning $1 - h_{ii} \to 0$, meaning the LOO residual blows up — these are the points that most influence the fit.

### Why LOOCV Has High Variance

From the correlation argument above: each pair of LOOCV models shares $n-2$ training points. The estimates $\hat{R}_1, \ldots, \hat{R}_n$ are nearly perfectly correlated. The mean has low bias but does **not** benefit from averaging — its variance is approximately equal to the variance of a single $\hat{R}_i$, not $1/n$ of it.

## ✦ Stratified $k$-Fold

For classification, random folding may produce folds with unrepresentative class distributions — exactly the same risk as in a single train/test split.

**Stratified $k$-fold** constructs folds such that each fold has approximately the same class proportion as the full dataset. The procedure: sort samples by class, then interleave them across folds.

```python
from sklearn.model_selection import StratifiedKFold

skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

for fold, (train_idx, val_idx) in enumerate(skf.split(X, y)):
    X_train, X_val = X[train_idx], X[val_idx]
    y_train, y_val = y[train_idx], y[val_idx]
    # ...
```

**Always use stratified $k$-fold for classification, especially under class imbalance.** Plain `KFold` with imbalanced data can produce folds where a minority class is entirely absent.

## ✦ Time-Series Cross-Validation

Standard $k$-fold shuffles the data before folding. For time series, this is **catastrophically wrong**: it allows models to be trained on future data and validated on past data, producing unrealistically optimistic estimates.

### Forward Chaining (Walk-Forward Validation)

The only correct procedure: train on all data up to time $t$, validate on data from time $t$ to $t + h$, then advance.

```
Train: [=======]            Validate: [---]
Train: [===========]        Validate:     [---]
Train: [===============]    Validate:         [---]
Train: [===================] Validate:            [---]
```

Formally, for split $i$ with gap $g$ (forecasting horizon):

$$\mathcal{D}_{\text{train}}^{(i)} = {(x_t, y_t) : t \leq t_i}, \quad \mathcal{D}_{\text{val}}^{(i)} = {(x_t, y_t) : t_i + g \leq t \leq t_i + g + h}$$

The gap $g$ prevents look-ahead when features are computed using rolling windows.

```python
from sklearn.model_selection import TimeSeriesSplit

tscv = TimeSeriesSplit(n_splits=5, gap=0)

for fold, (train_idx, val_idx) in enumerate(tscv.split(X)):
    print(f"Fold {fold}: train={train_idx[[0,-1]]}, val={val_idx[[0,-1]]}")
    # train_idx is always strictly before val_idx
```

**Expanding vs. rolling window:**

- **Expanding window** (default): training set grows with each fold. Uses all available history. Preferred when more data always helps.
- **Rolling window**: training set has fixed size, slides forward. Use when older data is no longer representative (concept drift, structural breaks).

## ✦ Nested Cross-Validation

**The problem:** if you use $k$-fold CV to select hyperparameters _and_ report the resulting CV score as your generalisation estimate, the estimate is biased — the hyperparameters were chosen to optimise it.

**Nested CV** separates these concerns with two loops:

- **Outer loop:** estimates the generalisation error of the full pipeline (training + hyperparameter selection)
- **Inner loop:** selects hyperparameters within each outer fold

```
Outer fold i:
    Train (outer):  [all data except outer fold i]
    Test  (outer):  [outer fold i]
    
    Inner CV on Train (outer):
        For each hyperparameter config h:
            k-fold CV on Train (outer) → val score(h)
        Select h* = argmax val score
    
    Retrain with h* on full Train (outer)
    Evaluate on Test (outer) → outer score_i

Report: mean of outer scores
```

```python
from sklearn.model_selection import cross_val_score, GridSearchCV, KFold
from sklearn.svm import SVC
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline

pipe = Pipeline([("sc", StandardScaler()), ("svm", SVC())])
param_grid = {"svm__C": [0.1, 1, 10], "svm__gamma": [0.01, 0.1, 1]}

# Inner CV: GridSearchCV with 5-fold CV
inner_cv = GridSearchCV(pipe, param_grid, cv=5, scoring="accuracy")

# Outer CV: 10-fold
outer_cv = KFold(n_splits=10, shuffle=True, random_state=42)
outer_scores = cross_val_score(inner_cv, X, y, cv=outer_cv, scoring="accuracy")

print(f"Nested CV estimate: {outer_scores.mean():.4f} ± {outer_scores.std():.4f}")
```

**Nested CV gives an unbiased estimate of the generalisation error of the model selection procedure itself** — not just one particular model. It answers the question: "if I use this algorithm (including hyperparameter tuning) on data from this distribution, what test error can I expect?"

The outer scores will generally be slightly worse than a naive CV estimate, because they account for the variance introduced by hyperparameter selection. That's not a bug — it's an honest number.

**When to use nested CV:**

- Small datasets where overfitting to the validation set is a real risk
- Research requiring honest comparison of model selection pipelines
- When you want to report a generalisation estimate alongside the final model

**When it's overkill:** large datasets ($n > 10^5$) where the inner/outer distinction has negligible effect on the estimates, and computational cost is prohibitive.

## ✦ The Bias-Variance Tradeoff of the CV Estimator — Summary

|Method|Bias|Variance|Cost|Notes|
|---|---|---|---|---|
|Single holdout|Moderate (depends on split size)|**High**|$O(1)$ fits|Sensitive to random split|
|5-fold CV|Low|Moderate|$O(5)$ fits|Good default|
|10-fold CV|Lower|Moderate|$O(10)$ fits|Standard in literature|
|LOOCV|**Lowest**|**High**|$O(n)$ fits (or $O(1)$ for linear models)|Unstable estimates|
|Nested CV|Unbiased for pipeline|Moderate|$O(k_{\text{out}} \times k_{\text{in}} \times M)$ fits|Expensive but honest|

## ✦ Practical Checklist

Before running cross-validation, ask:

**1. Is my data i.i.d.?**

- If grouped (patients, users): use `GroupKFold` — split by group, not by sample
- If temporal: use `TimeSeriesSplit` — never shuffle time series

**2. Is my data imbalanced?**

- Use `StratifiedKFold` for classification — always

**3. Is preprocessing happening inside the loop?**

- Scalers, imputers, feature selectors must be **fitted on the training fold only**
- Use `Pipeline` inside the CV loop — sklearn's `cross_val_score` does this correctly when given a pipeline

**4. Am I selecting hyperparameters using the CV score I'm reporting?**

- If yes, use nested CV or be explicit that the reported number is optimistic

**5. Do I need the final model, or just the performance estimate?**

- CV gives you an estimate of generalisation error, not the final model
- After CV-based model selection, refit on the **full training set** (train + val) with the chosen hyperparameters
- Evaluate the final model on the test set **once**

## ✦ Concrete Worked Example — End to End

```python
import numpy as np
from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import StratifiedKFold, GridSearchCV, cross_val_score
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.svm import SVC
from sklearn.metrics import accuracy_score

X, y = load_breast_cancer(return_X_y=True)

# Step 1: Reserve test set — untouched until the very end
from sklearn.model_selection import train_test_split
X_dev, X_test, y_dev, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)

# Step 2: Nested CV on X_dev to estimate pipeline performance
pipe = Pipeline([("sc", StandardScaler()), ("svm", SVC())])
param_grid = {"svm__C": [0.01, 0.1, 1, 10], "svm__kernel": ["rbf", "linear"]}

inner_cv  = StratifiedKFold(n_splits=5,  shuffle=True, random_state=0)
outer_cv  = StratifiedKFold(n_splits=10, shuffle=True, random_state=1)
inner_gs  = GridSearchCV(pipe, param_grid, cv=inner_cv, scoring="accuracy", refit=True)
outer_scores = cross_val_score(inner_gs, X_dev, y_dev, cv=outer_cv, scoring="accuracy")

print(f"Nested CV estimate: {outer_scores.mean():.4f} ± {outer_scores.std():.4f}")

# Step 3: Select best hyperparameters on all of X_dev
inner_gs.fit(X_dev, y_dev)
print(f"Best params: {inner_gs.best_params_}")

# Step 4: Final evaluation on test set — once
test_acc = accuracy_score(y_test, inner_gs.predict(X_test))
print(f"Test accuracy: {test_acc:.4f}")
# Development stops here.
```

## ✦ Connection to AI, Machine Learning, and Deep Learning

### CV in Deep Learning: Why It's Rarely Used

Standard $k$-fold CV requires training $k$ models. For a ResNet-50 training for 100 epochs on ImageNet, one training run takes hours on multiple GPUs. $k=5$ would multiply this by 5. The cost is prohibitive.

In practice, deep learning uses:

- **A single fixed val split**, often 10–20% of the training set
- **Early stopping** on validation loss as the primary regularisation mechanism
- **Large enough datasets** that the variance of a single split is acceptable

Cross-validation in deep learning is reserved for:

- Small datasets (medical imaging, scientific ML) where CV is justified
- **Neural architecture search (NAS):** CV is approximated by training proxy tasks
- **Few-shot learning benchmarks:** CV across episodes

### The Meta-Learning Connection

Cross-validation is structurally identical to **meta-learning** (learning to learn). In $k$-fold CV:

- Each fold is a "task"
- The model is adapted to each task (trained on $k-1$ folds)
- Evaluated on the held-out fold

In **MAML (Model-Agnostic Meta-Learning)** (Finn et al., 2017):

- Each episode is a few-shot task sampled from a distribution
- The meta-model is trained so that it can adapt quickly to a new task with few gradient steps
- The meta-objective is exactly the average loss across tasks — the same structure as $\hat{R}_{\text{CV}}$

The mathematical object $\hat{R}_{\text{CV}} = \frac{1}{k}\sum_i \hat{R}_i$ reappears as the meta-learning objective. CV is the special case where tasks share the same dataset and differ only in the train/val partition.

### Hyperparameter Optimisation at Scale

Nested CV is conceptually the right framework for hyperparameter tuning, but computationally expensive for DL. Modern alternatives:

- **Bayesian optimisation** (Spearmint, Optuna): builds a surrogate model of the validation loss surface, uses it to pick the next hyperparameter to evaluate — far more efficient than grid search
- **Hyperband / ASHA**: allocates budget adaptively — kill poorly performing runs early, concentrate computation on promising ones. Inner loop of nested CV replaced by early termination
- **Population-Based Training (PBT):** evolves hyperparameters during training rather than selecting them beforehand — used at DeepMind for large-scale RL and language model training

All of these are approximations to the nested CV ideal: find the hyperparameters that maximise held-out performance, without overfitting to the validation set.
