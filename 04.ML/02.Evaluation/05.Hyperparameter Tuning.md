
> _"The art is not in the model — it's in knowing which knob to turn, how far, and why."_

## ✦ What Even Is a Hyperparameter?

Before we tune anything, let's be precise. A **parameter** is learned from data during training — the weights $w$ in linear regression, the split thresholds in a decision tree. A **hyperparameter** is set _before_ training and governs the learning process itself: the regularisation strength $\lambda$, the tree depth, the learning rate $\alpha$.

The distinction matters because hyperparameters are **not optimised by the training objective** $\hat{R}(\hat{f})$. If you tried to include them in gradient descent, you'd collapse: setting $\lambda = 0$ always minimises the regularised training loss, giving you a degenerate solution. So we need a separate loop.

```
Outer loop (hyperparameter search):
  for each candidate config θ_h:
    Inner loop (training):
      fit model on training data → get parameters w*(θ_h)
    Evaluate w*(θ_h) on validation data
  Return θ_h* that minimises validation loss
```

This is the meta-learning loop. The training loss tells you how well you fit the data you've seen. The validation loss tells you how well you _generalise_. You optimise the latter by searching over $\theta_h$.

---

## ✦ The Setup: What You're Searching Over

Let $\Theta$ be the hyperparameter space. In practice this is a product of different types:

$$\Theta = \Theta_{\text{continuous}} \times \Theta_{\text{integer}} \times \Theta_{\text{categorical}}$$

**Example (Random Forest):**

- $n_estimators \in {50, 100, 200, 500, \ldots}$ — integer
- $max_depth \in {3, 5, 10, \text{None}}$ — integer + None
- $max_features \in {\text{"sqrt"}, \text{"log2"}, 0.5, 1.0}$ — categorical + continuous
- $min_samples_split \in [2, 50]$ — integer

**Example (Neural Network):**

- $\alpha \in [10^{-5}, 10^{-1}]$ — continuous, log scale
- $\text{batch_size} \in {32, 64, 128, 256}$ — integer
- $\text{optimizer} \in {\text{Adam}, \text{SGD}, \text{RMSProp}}$ — categorical
- $\text{n_layers} \in {2, 3, 4, 5}$ — integer
- $\text{dropout} \in [0, 0.5]$ — continuous

The space is high-dimensional, mixed-type, and **the objective function is expensive to evaluate** — each evaluation requires a full training run. This is what makes hyperparameter optimisation hard.

## ✦ 1. Grid Search

### The Method

Enumerate a finite grid over the hyperparameter space and evaluate every combination.

$$\Theta_{\text{grid}} = {\lambda_1, \lambda_2, \ldots} \times {C_1, C_2, \ldots} \times \cdots$$

For $k$ hyperparameters with $n_i$ values each, the number of evaluations is:

$$N_{\text{eval}} = \prod_{i=1}^k n_i$$

**Example:**

- $\lambda \in {0.001, 0.01, 0.1, 1.0}$ → 4 values
- $C \in {0.1, 1.0, 10.0}$ → 3 values
- $\text{kernel} \in {\text{rbf}, \text{poly}}$ → 2 values

Total evaluations: $4 \times 3 \times 2 = 24$. With 5-fold CV, that's $24 \times 5 = 120$ model fits.

### Properties

- **Exhaustive within the grid** — you will find the best combination that exists in your grid
- **Wasteful by design** — it allocates equal budget to every dimension, even unimportant ones
- **Scales exponentially** — adding one 5-value hyperparameter quintuples the cost
- **Blind to continuous structure** — if the true optimum is between $\lambda = 0.01$ and $\lambda = 0.1$, you'll never find it

### When to Use

Small search spaces (2-3 hyperparameters, few values each). When you want a reproducible, interpretable sweep. Baseline to compare against.

### sklearn API

```python
from sklearn.model_selection import GridSearchCV
from sklearn.svm import SVC

param_grid = {
    'C': [0.1, 1.0, 10.0, 100.0],
    'gamma': [0.001, 0.01, 0.1, 1.0],
    'kernel': ['rbf', 'poly']
}

gs = GridSearchCV(
    SVC(),
    param_grid,
    cv=5,
    scoring='accuracy',
    n_jobs=-1,
    verbose=2
)
gs.fit(X_train, y_train)

print(gs.best_params_)        # {'C': 10.0, 'gamma': 0.01, 'kernel': 'rbf'}
print(gs.best_score_)         # CV score of best config
# IMPORTANT: final evaluation on test set
print(gs.score(X_test, y_test))
```

## ✦ 2. Random Search

### The Key Insight (Bergstra & Bengio, 2012)

> _"Random search is more efficient than grid search for hyperparameter optimisation of neural networks."_

Why? Consider searching over two hyperparameters, one important and one unimportant. Grid search wastes most of its budget exploring the unimportant dimension. Random search, sampling independently, gets more distinct values of the _important_ parameter for the same budget.

**Formal argument.** Suppose the effective dimensionality of the problem is $d_{\text{eff}} \ll d$ (only a few hyperparameters actually matter). With $n$ random trials, you get $n$ distinct values along each important dimension. With $n$ grid trials over a $d$-dimensional space, you get only $n^{1/d}$ distinct values along each dimension.

For $n = 100$, $d = 5$: grid gives you $100^{1/5} \approx 2.5$ values per dimension. Random gives you 100.

### Distribution Choices Matter

Sampling from the wrong distribution concentrates probability mass in the wrong region.

|Hyperparameter|Wrong Distribution|Right Distribution|
|---|---|---|
|Learning rate $\alpha$|Uniform $[10^{-5}, 10^{-1}]$|Log-uniform: $10^{\mathcal{U}(-5, -1)}$|
|Regularisation $\lambda$|Uniform $[0, 1]$|Log-uniform: $10^{\mathcal{U}(-4, 0)}$|
|$n_estimators$|Uniform $[10, 1000]$|Uniform integers|
|Dropout rate|—|Uniform $[0, 0.5]$|

**Why log-uniform for scale parameters?** If the optimal $\alpha$ is around $0.001$, uniform sampling gives that neighbourhood probability $\approx 0.001 / 0.1 = 1%$. Log-uniform assigns equal probability to each order of magnitude.

```python
import numpy as np
from scipy.stats import loguniform

param_distributions = {
    'C': loguniform(1e-3, 1e3),          # 10^(-3) to 10^3
    'gamma': loguniform(1e-4, 1e1),
    'kernel': ['rbf', 'poly', 'linear']   # categorical: uniform
}

from sklearn.model_selection import RandomizedSearchCV
rs = RandomizedSearchCV(
    SVC(),
    param_distributions,
    n_iter=100,       # evaluate 100 random configs
    cv=5,
    scoring='accuracy',
    random_state=42,
    n_jobs=-1
)
rs.fit(X_train, y_train)
```

### Properties

- **Anytime algorithm** — stop early and still have a good result
- **No combinatorial explosion** — cost is $O(n_iter)$ regardless of dimension
- **Non-exhaustive** — no guarantee of finding the global optimum
- **Embarrassingly parallel** — each trial is independent

## ✦ 3. Bayesian Optimisation

This is where things get principled. Grid and random search are _model-free_ — they ignore what they've learned from previous evaluations. Bayesian optimisation builds a **probabilistic model of the objective function** and uses it to decide where to evaluate next.

### The Framework

Let $f: \Theta \to \mathbb{R}$ be the validation loss as a function of hyperparameters. We want: $$\theta^* = \arg\min_{\theta \in \Theta} f(\theta)$$

$f$ is expensive to evaluate (each call = full training run), so we want to be smart about which $\theta$ to query.

**The Bayesian approach:**

1. Place a prior over $f$: typically a **Gaussian Process** $f \sim \mathcal{GP}(m(\theta), k(\theta, \theta'))$
2. After observing $\mathcal{D}_t = {(\theta_i, f(\theta_i))}_{i=1}^t$, compute the posterior $p(f | \mathcal{D}_t)$
3. Use an **acquisition function** $\alpha(\theta; \mathcal{D}_t)$ to decide $\theta_{t+1} = \arg\max_\theta \alpha(\theta; \mathcal{D}_t)$
4. Evaluate $f(\theta_{t+1})$, update posterior, repeat

The posterior gives us both a **mean prediction** $\mu_t(\theta)$ and an **uncertainty estimate** $\sigma_t(\theta)$. The acquisition function balances _exploitation_ (go where $\mu_t$ is small) and _exploration_ (go where $\sigma_t$ is large).

### Gaussian Process Surrogate

Recall from [[03.Gaussian Processes|Gaussian Processes]]: a GP is fully specified by its mean and covariance function. The posterior after $t$ observations has closed form:

$$\mu_t(\theta) = k_t(\theta)^T (K_t + \sigma^2 I)^{-1} \mathbf{f}_t$$ $$\sigma_t^2(\theta) = k(\theta, \theta) - k_t(\theta)^T (K_t + \sigma^2 I)^{-1} k_t(\theta)$$

where $k_t(\theta) = [k(\theta, \theta_1), \ldots, k(\theta, \theta_t)]^T$ and $K_t$ is the $t \times t$ Gram matrix.

The cost is $O(t^3)$ — GPs don't scale to large $t$. In practice, this means Bayesian optimisation is used for $\leq$ a few hundred evaluations.

### Acquisition Functions

#### Expected Improvement (EI)

Let $f^* = \min_{i \leq t} f(\theta_i)$ be the current best. Define improvement $I(\theta) = \max(f^* - f(\theta), 0)$.

$$\text{EI}(\theta) = \mathbb{E}_{f(\theta) \sim \mathcal{N}(\mu_t, \sigma_t^2)}[\max(f^* - f(\theta), 0)]$$

This has a closed form:

$$\text{EI}(\theta) = (f^* - \mu_t(\theta))\Phi(Z) + \sigma_t(\theta)\phi(Z), \quad Z = \frac{f^* - \mu_t(\theta)}{\sigma_t(\theta)}$$

where $\Phi$ and $\phi$ are the standard normal CDF and PDF.

**Intuition:** EI is high when either (a) $\mu_t(\theta)$ is much better than $f^*$ (exploitation) or (b) $\sigma_t(\theta)$ is large (exploration). The balance emerges automatically.

#### Upper Confidence Bound (UCB)

$$\text{UCB}(\theta) = -\mu_t(\theta) + \kappa \sigma_t(\theta)$$

(Negative because we minimise $f$.) The parameter $\kappa$ directly controls the exploration-exploitation tradeoff. $\kappa = 0$ is pure exploitation; $\kappa \to \infty$ is pure exploration.

UCB has strong theoretical guarantees — the cumulative regret $\sum_{t=1}^T (f(\theta_t) - f(\theta^*))$ grows sublinearly for many kernel families.

#### Probability of Improvement (PI)

$$\text{PI}(\theta) = P(f(\theta) < f^* - \xi) = \Phi!\left(\frac{f^* - \xi - \mu_t(\theta)}{\sigma_t(\theta)}\right)$$

The $\xi > 0$ "exploration jitter" prevents the algorithm from being too greedy. PI is simpler than EI but tends to be more exploitative.

### Visualisation: The Bayesian Optimisation Loop

```
Iteration 1:
  Prior GP (no data)
  ───────────────────────────────────────────────
  μ(θ) = 0 everywhere, σ(θ) = large everywhere
  EI is flat → sample at random → evaluate f(θ₁)

Iteration 2:
  Posterior conditioned on (θ₁, f(θ₁))
  ───────────────────────────────────────────────
  Near θ₁: σ collapses (we know what's there)
  Away from θ₁: σ still large
  EI peaks somewhere promising → evaluate f(θ₂)

...

Iteration t:
  Posterior tightly fits observations
  ───────────────────────────────────────────────
  μ(θ) is a good surrogate for f(θ)
  EI peaks at the suspected optimum or a
  high-uncertainty region → converges
```

### Alternative Surrogate: Tree-structured Parzen Estimator (TPE)

Used by Hyperopt and Optuna. Instead of modelling $p(f|\theta)$ directly, TPE models two densities:

- $\ell(\theta)$: distribution of $\theta$ given $f(\theta) < f^*_{\gamma}$ (good configs)
- $g(\theta)$: distribution of $\theta$ given $f(\theta) \geq f^*_{\gamma}$ (bad configs)

where $f^*_\gamma$ is the $\gamma$-quantile of observed $f$ values ($\gamma = 0.15$ typically).

The acquisition function is $\text{EI} \propto \ell(\theta) / g(\theta)$. New candidates are sampled from $\ell$ and selected by maximising $\ell/g$.

**Why TPE scales better:** It uses kernel density estimates in the original hyperparameter space, handles conditional hyperparameters naturally, and avoids the $O(t^3)$ GP bottleneck.

### sklearn/Optuna API

```python
import optuna

def objective(trial):
    # Define hyperparameter space
    C = trial.suggest_float('C', 1e-3, 1e3, log=True)
    gamma = trial.suggest_float('gamma', 1e-4, 1e1, log=True)
    kernel = trial.suggest_categorical('kernel', ['rbf', 'poly', 'linear'])
    
    # Evaluate
    clf = SVC(C=C, gamma=gamma, kernel=kernel)
    score = cross_val_score(clf, X_train, y_train, cv=5, scoring='accuracy').mean()
    return score   # Optuna maximises by default

study = optuna.create_study(
    direction='maximize',
    sampler=optuna.samplers.TPESampler(seed=42)
)
study.optimize(objective, n_trials=100)

# Visualise the search

```

## ✦ 4. Successive Halving and Hyperband

### The Problem with Standard Methods

Every evaluation in grid/random/Bayesian search uses the _full_ training budget — all epochs, all data. But most configurations are clearly bad after seeing $1%$ of the data. We're wasting $99%$ of the budget on doomed runs.

### Successive Halving

**Setup:** $n$ configurations, total budget $B$. Allocate equal budget $B/n$ to each, then eliminate the bottom half, double the budget for survivors, repeat.

**Algorithm:**

```
Input: n configurations {θ₁, ..., θₙ}, total budget B, η = 2 (halving rate)
───────────────────────────────────────────────────────────────────────────
S₀ = {θ₁, ..., θₙ}
r₀ = B / (n · ceil(log_η(n)))   # initial budget per config

for k = 0, 1, ..., ceil(log_η(n)) - 1:
    Evaluate each θ ∈ Sₖ with budget rₖ
    Sₖ₊₁ = top ⌊|Sₖ|/η⌋ configs by validation loss
    rₖ₊₁ = rₖ · η
    
Return best config in S_final
```

**Tradeoff:** Large $n$ (explore many configs) vs. large $r_0$ (accurate early estimates). Successive halving forces you to choose.

### Hyperband

Hyperband removes the $n$ vs. $r_0$ dilemma by running _multiple brackets_ of Successive Halving with different initial budgets.

For maximum budget $R$ and $\eta$:

- $s_{\max} = \lfloor \log_\eta R \rfloor$
- Run brackets $s = s_{\max}, s_{\max}-1, \ldots, 0$
- Bracket $s$ starts with $n_s = \lceil \frac{s_{\max}+1}{s+1} \eta^s \rceil$ configs and budget $r_s = R/\eta^s$

**Why it works:** Bracket $s_{\max}$ (many configs, small budget) is great at exploration. Bracket $0$ (one config, full budget) is great at exploitation. Hyperband gets both, spending total budget proportional to $s_{\max} \cdot R$.

```python
from sklearn.model_selection import HalvingRandomSearchCV
import numpy as np

param_distributions = {
    'n_estimators': np.arange(10, 500),
    'max_depth': [None, 3, 5, 10, 20],
    'min_samples_split': np.arange(2, 50),
    'max_features': ['sqrt', 'log2', 0.5, 1.0]
}

from sklearn.ensemble import RandomForestClassifier
hs = HalvingRandomSearchCV(
    RandomForestClassifier(random_state=42),
    param_distributions,
    n_candidates='exhaust',
    resource='n_estimators',   # budget = number of trees
    factor=3,                   # η: keep top 1/3 each round
    cv=5,
    random_state=42,
    n_jobs=-1
)
hs.fit(X_train, y_train)
```

## ✦ The Leakage Trap

This deserves its own section because it kills more projects than wrong algorithms do.

**The trap:** You tune hyperparameters on the validation set. After tuning, you use the same validation set to report final performance. You've effectively trained on the validation set — your reported metric is optimistic.

**The fix:** Three-way split (or nested CV).

```
Wrong protocol:
  Train → fit model parameters w*(θ_h)
  Val   → select θ_h*
  Val   → report final score ← LEAKAGE

Correct protocol:
  Train → fit model parameters w*(θ_h)
  Val   → select θ_h*
  Test  → report final score ← held out the entire time
```

The test set must be **held out the entire time** — never used for model selection, hyperparameter tuning, feature selection, or anything that drives decisions. It is a single-use oracle.

If you don't have enough data for a three-way split, use **nested cross-validation**:

```
Outer loop (k_outer folds):
  for fold (train_outer, test_outer):
    Inner loop (k_inner folds on train_outer):
      for fold (train_inner, val_inner):
        fit model, evaluate → find best θ_h*
    Retrain best θ_h* on full train_outer
    Evaluate on test_outer → one unbiased estimate
Average estimates over outer folds → final generalisation estimate
```

Nested CV is expensive ($k_{\text{outer}} \times k_{\text{inner}} \times n_iter$ fits) but gives an unbiased estimate of generalisation performance.

```python
from sklearn.model_selection import cross_val_score, GridSearchCV
from sklearn.svm import SVC
from sklearn.pipeline import Pipeline

inner_cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
outer_cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

param_grid = {'svc__C': [0.1, 1, 10], 'svc__gamma': [0.01, 0.1, 1]}

pipe = Pipeline([('svc', SVC())])
inner_search = GridSearchCV(pipe, param_grid, cv=inner_cv, scoring='accuracy')

# This is the unbiased estimate
nested_scores = cross_val_score(inner_search, X, y, cv=outer_cv, scoring='accuracy')
print(f"Nested CV: {nested_scores.mean():.3f} ± {nested_scores.std():.3f}")
```

## ✦ Practical Notes

### Log-scale Hyperparameters

Whenever a hyperparameter spans multiple orders of magnitude, search on a log scale:

$$\lambda \sim 10^{\mathcal{U}(a, b)} \quad \text{not} \quad \lambda \sim \mathcal{U}(10^a, 10^b)$$

The difference: uniform in $[10^{-4}, 1]$ gives $99.99%$ of mass to $[0.1, 1]$. Log-uniform gives $25%$ to each decade.

### Early Stopping as Regularisation

For gradient-based models, early stopping during hyperparameter search is legitimate and cheap. Stop training when validation loss stops decreasing. This is itself a hyperparameter (patience), but usually less sensitive than learning rate.

### Warm Starting

Some algorithms (sklearn estimators with `warm_start=True`) allow you to reuse previous fits. This can dramatically speed up grid search over $n_estimators$ in Random Forest: fit with 100 trees, then extend to 200, then 500 — rather than fitting from scratch three times.

### Population Based Training (PBT)

A more aggressive strategy for deep learning: run a population of models _simultaneously_, and periodically replace poorly-performing models with mutated copies of the best ones. Conceptually: genetic algorithms applied to hyperparameter schedules. Used at DeepMind for game-playing agents.

## ✦ Connection to AI / ML / Deep Learning

Hyperparameter tuning is not a peripheral concern — it is central to modern ML:

**Deep Learning is especially sensitive.** A neural network with $\alpha = 0.1$ might diverge entirely. The same network with $\alpha = 10^{-3}$ converges to 94% accuracy. With $\alpha = 10^{-4}$, it converges to 96%. The difference between a useless and state-of-the-art model is often the hyperparameters, not the architecture.

**AutoML automates this loop.** Systems like Auto-sklearn, AutoKeras, and Google's AutoML pipeline use Bayesian optimisation + Hyperband + Neural Architecture Search (NAS) to automate the entire model selection and tuning process. The "learning" in AutoML is learning the hyperparameter objective.

**Neural Architecture Search (NAS)** treats the architecture itself as a hyperparameter. DARTS reformulates discrete architecture choices as continuous parameters and differentiates through them. This is hyperparameter optimisation taken to its logical conclusion.

**Meta-learning ("learning to learn")** goes further: train a model to predict good hyperparameters for a new task based on dataset meta-features. This is the same spirit as warm-starting Bayesian optimisation with knowledge from previous tasks. MAML (Model-Agnostic Meta-Learning) learns initial parameters that require minimal fine-tuning — another form of optimising across the outer loop.

**Foundation Model Fine-Tuning.** When fine-tuning BERT or GPT-style models, the learning rate schedule, warmup steps, and weight decay are the most critical hyperparameters. The LLM literature has accumulated strong defaults (e.g., linear warmup + cosine decay, $\alpha \approx 2 \times 10^{-5}$ for BERT fine-tuning), but these are themselves the product of massive hyperparameter searches.

## ✦ Implementation From Scratch: Bayesian Optimisation (GP + EI)

```python
import numpy as np
from scipy.stats import norm
from scipy.optimize import minimize

class GaussianProcess:
    def __init__(self, length_scale=1.0, noise=1e-6):
        self.ls = length_scale
        self.noise = noise
    
    def rbf_kernel(self, X1, X2):
        # K[i,j] = exp(-||x_i - x_j||^2 / (2 * ls^2))
        diff = X1[:, None, :] - X2[None, :, :]
        return np.exp(-0.5 * np.sum(diff**2, axis=-1) / self.ls**2)
    
    def fit(self, X, y):
        self.X_train = X
        self.y_train = y
        self.K = self.rbf_kernel(X, X) + self.noise * np.eye(len(X))
        self.L = np.linalg.cholesky(self.K)
        self.alpha = np.linalg.solve(self.L.T, np.linalg.solve(self.L, y))
    
    def predict(self, X_test):
        k_star = self.rbf_kernel(self.X_train, X_test)
        mu = k_star.T @ self.alpha
        v = np.linalg.solve(self.L, k_star)
        sigma2 = self.rbf_kernel(X_test, X_test) - v.T @ v
        sigma = np.sqrt(np.maximum(np.diag(sigma2), 0))
        return mu, sigma


def expected_improvement(X, gp, f_best, xi=0.01):
    mu, sigma = gp.predict(X)
    Z = (f_best - mu - xi) / (sigma + 1e-9)
    ei = (f_best - mu - xi) * norm.cdf(Z) + sigma * norm.pdf(Z)
    ei[sigma < 1e-10] = 0.0
    return ei


def bayesian_optimisation(f, bounds, n_init=5, n_iter=25):
    """
    f: function to minimise (returns scalar)
    bounds: list of (low, high) per dimension
    """
    dim = len(bounds)
    
    # Random initialisation
    X = np.array([[np.random.uniform(lo, hi) for lo, hi in bounds]
                   for _ in range(n_init)])
    y = np.array([f(x) for x in X])
    
    for i in range(n_iter):
        gp = GaussianProcess(length_scale=1.0)
        gp.fit(X, y)
        f_best = y.min()
        
        # Maximise EI via random restarts
        best_ei, best_x = -np.inf, None
        for _ in range(100):
            x0 = np.array([np.random.uniform(lo, hi) for lo, hi in bounds])
            res = minimize(
                lambda x: -expected_improvement(x.reshape(1, -1), gp, f_best),
                x0,
                bounds=bounds,
                method='L-BFGS-B'
            )
            if -res.fun > best_ei:
                best_ei = -res.fun
                best_x = res.x
        
        # Evaluate
        y_new = f(best_x)
        X = np.vstack([X, best_x])
        y = np.append(y, y_new)
        
        print(f"Iter {i+1:3d}: f({best_x}) = {y_new:.4f}, best so far = {y.min():.4f}")
    
    return X[y.argmin()], y.min()


# Example: tune C and gamma for an SVC
from sklearn.svm import SVC
from sklearn.model_selection import cross_val_score
from sklearn.datasets import make_classification

X_data, y_data = make_classification(n_samples=500, n_features=20, random_state=42)

def objective(theta):
    # theta = [log10(C), log10(gamma)]
    C = 10 ** theta[0]
    gamma = 10 ** theta[1]
    clf = SVC(C=C, gamma=gamma, kernel='rbf')
    score = cross_val_score(clf, X_data, y_data, cv=5, scoring='accuracy').mean()
    return 1 - score  # minimise 1 - accuracy

best_theta, best_val = bayesian_optimisation(
    objective,
    bounds=[(-3, 3), (-4, 1)],  # log-scale bounds
    n_init=5,
    n_iter=30
)
print(f"\nBest C = {10**best_theta[0]:.4f}, gamma = {10**best_theta[1]:.6f}")
print(f"Best validation accuracy = {1 - best_val:.4f}")
```
