## ✦ Why This Note Exists

You have derived the algorithm. You have tuned the hyperparameters. You have verified SHAP values. The model is good.

Now what?

The gap between "model that works in a notebook" and "model that works in the world" is where most ML projects die. This note covers that gap: how a trained model becomes a live system, what can go wrong after deployment, and how to build infrastructure that detects and recovers from it. The mathematics here is lighter than in the modelling notes; the engineering discipline required is heavier.

A useful framing from Sculley et al. (2015): in a production ML system, the ML code is typically a small fraction of the total codebase. The surrounding infrastructure — data pipelines, serving layers, monitoring, configuration management — constitutes most of the complexity and most of the failure surface.

## ✦ Part I — Model Serialisation

Before a model can be served, it must be saved in a form that can be loaded in a different process, on a different machine, possibly with a different operating system.

### 1.1 The Serialisation Problem

Training a model produces a Python object in memory: a fitted `XGBClassifier`, a `Pipeline`, a `torch.nn.Module`. To use it later, you must serialise it — convert the in-memory object to bytes on disk — and deserialise it — reconstruct the object from bytes.

The serialisation must preserve not just the model weights but the entire preprocessing pipeline. A `StandardScaler` fitted on training data has a `.mean_` and `.scale_` attribute that must be saved alongside the model. If you lose the scaler, your model produces incorrect predictions on unnormalised inputs — silently, with no error.

**The invariant:** every artefact needed to transform raw input $x$ into a prediction $\hat{y}$ must be saved together. The model and its preprocessing are a single deployable unit.

### 1.2 Formats

**`pickle` / `joblib` — sklearn and general Python objects:**

```python
import joblib

# Save
joblib.dump(pipeline, 'models/cancellation_pipeline_v3.pkl')

# Load
pipeline = joblib.load('models/cancellation_pipeline_v3.pkl')
predictions = pipeline.predict(X_new)
```

`joblib` is preferred over raw `pickle` for large NumPy arrays (uses memory-mapping for efficiency). Both are Python-specific — they produce files that only Python can load.

Critical warnings about pickle:

- Pickle deserialisation executes arbitrary code. Never unpickle a file from an untrusted source.
- Pickle files are not forward-compatible. A model pickled with sklearn 1.3 may fail to load with sklearn 1.5. Always save the exact library version alongside the model.
- Add version metadata explicitly:

```python
import sklearn, joblib

model_bundle = {
    'pipeline': pipeline,
    'metadata': {
        'sklearn_version': sklearn.__version__,
        'training_date': '2026-04-24',
        'val_auc': 0.891,
        'feature_names': list(feature_names),
        'git_hash': '3a7b2f1'
    }
}
joblib.dump(model_bundle, 'models/bundle_v3.pkl')
```

**`torch.save` — PyTorch models:**

```python
# Save state dict (preferred — portable, version-independent)
torch.save(model.state_dict(), 'models/net_v3.pt')

# Load: must reconstruct architecture first
model = CancellationNet(input_dim=47, hidden_dim=256)
model.load_state_dict(torch.load('models/net_v3.pt', map_location='cpu'))
model.eval()
```

Always save the state dict, not the entire model object. `torch.save(model, ...)` pickles the model class, which breaks if the class definition changes. `torch.save(model.state_dict(), ...)` saves only the parameters — the architecture must be reconstructed separately, but it is stable.

**ONNX — Open Neural Network Exchange:**

ONNX is a cross-platform, cross-language format. A model exported to ONNX can be run in Python, C++, Java, C#, and on specialised hardware (GPUs, NPUs) via the ONNX Runtime.

```python
import torch.onnx

# Export
dummy_input = torch.randn(1, 47)   # batch_size=1, input_dim=47
torch.onnx.export(
    model,
    dummy_input,
    'models/cancellation_v3.onnx',
    input_names=['features'],
    output_names=['cancellation_prob'],
    dynamic_axes={'features': {0: 'batch_size'}}
)

# Run with ONNX Runtime (no PyTorch required)
import onnxruntime as ort
session = ort.InferenceSession('models/cancellation_v3.onnx')
outputs = session.run(['cancellation_prob'], {'features': X_val_np.astype(np.float32)})
```

ONNX exports are typically faster than native PyTorch at inference time (the runtime applies graph optimisations) and are the standard format for deploying to edge devices, mobile, and production inference servers.

## ✦ Part II — Serving Infrastructure

A serialised model is a file. To make predictions on demand, you need to load the model and expose an interface through which clients can send inputs and receive predictions. This is the serving layer.

### 2.1 The Three Serving Patterns

**Pattern 1: Batch inference** — offline, scheduled.

The model runs over a large dataset and writes predictions to a database or file. Consumers read the pre-computed predictions.

```
Trigger (daily at 2am)
    → Load model
    → Read all users from database
    → Run model.predict(all_users)
    → Write predictions back to database
    → Consumers query: SELECT cancellation_prob FROM predictions WHERE booking_id = ?
```

When to use: predictions don't need to be real-time (daily churn scores, weekly recommendation updates), input data is available in batch, latency is not a constraint.

Advantages: simple, cheap, model errors don't affect live user traffic directly, easy to audit.

Disadvantage: predictions are stale — they don't reflect events since the last batch run.

**Pattern 2: Online inference (REST API)** — real-time, synchronous.

The model is hosted as a service. Clients send an HTTP request with features and receive a prediction in the response.

```
Client → POST /predict {"lead_time": 45, "deposit_type": "No_Deposit", ...}
Server → {"cancellation_prob": 0.23, "model_version": "v3", "latency_ms": 12}
```

Latency requirements are strict (typically $< 100$ ms for user-facing systems, $< 10$ ms for high-frequency systems). The serving server must be efficient — model loading at startup, not per-request.

**Pattern 3: Streaming inference** — event-driven, asynchronous.

Predictions are triggered by events on a message queue (Kafka, Kinesis). The model consumes events, produces prediction events, downstream consumers process predictions.

```
Event stream: {booking_created: {booking_id: 123, features: {...}}}
    → Model consumer → predict → {cancellation_prob: 0.87}
    → Prediction stream consumed downstream: fraud check, pricing engine
```

Common in high-throughput systems where predictions are one step in a larger processing pipeline.

### 2.2 Building a REST API with FastAPI

```python
# serve.py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import joblib, pandas as pd, numpy as np, logging, time
from typing import Optional

app = FastAPI(title="Hotel Cancellation Predictor", version="v3")
logger = logging.getLogger(__name__)

# Load at startup — not per request
bundle = joblib.load('models/bundle_v3.pkl')
pipeline = bundle['pipeline']
metadata = bundle['metadata']

class BookingFeatures(BaseModel):
    lead_time: int
    adr: float
    total_of_special_requests: int
    country: str
    deposit_type: str
    room_type: str

class PredictionResponse(BaseModel):
    booking_id: Optional[str]
    cancellation_prob: float
    model_version: str
    latency_ms: float

@app.get("/health")
def health():
    return {"status": "ok", "model_version": metadata['training_date']}

@app.post("/predict", response_model=PredictionResponse)
def predict(features: BookingFeatures, booking_id: Optional[str] = None):
    start = time.time()
    try:
        df = pd.DataFrame([features.dict()])
        prob = pipeline.predict_proba(df)[0, 1]
    except Exception as e:
        logger.error(f"Prediction error: {e}", exc_info=True)
        raise HTTPException(status_code=500, detail=str(e))
    latency = (time.time() - start) * 1000
    logger.info(f"booking_id={booking_id} prob={prob:.4f} latency_ms={latency:.1f}")
    return PredictionResponse(
        booking_id=booking_id,
        cancellation_prob=float(prob),
        model_version=metadata['training_date'],
        latency_ms=latency
    )

# Run: uvicorn serve:app --host 0.0.0.0 --port 8000 --workers 4
```

Key production considerations: `--workers 4` runs four processes, each loading the model independently — $4\times$ memory cost. The `/health` endpoint is required for load balancers and Kubernetes liveness probes. Always log: booking ID, prediction, latency. This log is your ground truth for computing online metrics and detecting failures.

### 2.4 Managed Serving Frameworks

For production at scale, purpose-built serving frameworks handle load balancing, batching, model versioning, and hardware acceleration:

**TorchServe** (PyTorch): handles PyTorch models natively, supports GPU batching, provides a model archive format (`.mar`) that bundles model + handler code.

**TFServing** (TensorFlow): similar for TF models, uses SavedModel format, gRPC and REST interfaces.

**Triton Inference Server** (NVIDIA): multi-framework (ONNX, TensorRT, TF, PyTorch), designed for GPU inference, dynamic batching, concurrent model execution.

**BentoML**: framework-agnostic, integrates with MLflow model registry, handles packaging and serving together.

For most teams: BentoML or a custom FastAPI service is sufficient. Triton is appropriate when GPU utilisation and batching efficiency are critical.

## ✦ Part III — Distribution Shift

This is the central problem of deployed ML. The model was trained on data from distribution $\mathcal{D}_{\text{train}}$. It is deployed into the world, where data comes from distribution $\mathcal{D}_{\text{deploy}}$. If $\mathcal{D}_{\text{train}} \neq \mathcal{D}_{\text{deploy}}$, model performance degrades — sometimes slowly, sometimes catastrophically, almost always silently without monitoring.

### 3.1 A Taxonomy of Distribution Shift

Let $P_{\text{train}}(x, y)$ and $P_{\text{deploy}}(x, y)$ denote the joint distributions at training and deployment time. Factor the joint as:

$$P(x, y) = P(x) \cdot P(y | x) = P(y) \cdot P(x | y)$$

Each factor can shift independently.

**Covariate shift:** $P_{\text{train}}(x) \neq P_{\text{deploy}}(x)$, but $P_{\text{train}}(y|x) = P_{\text{deploy}}(y|x)$.

The input distribution changes but the true relationship between features and labels is stable.

_Example:_ a hotel cancellation model trained on 2019 data is deployed in 2020. The feature `deposit_type` distribution shifts — during COVID, "Non_Refund" bookings drop. The model's conditional prediction $P(\text{cancel} | \text{Non_Refund})$ may still be correct, but the model's aggregate accuracy degrades because it sees fewer of the input types it was trained on.

_Detection:_ compare $P_{\text{train}}(x)$ to $P_{\text{current}}(x)$ using statistical tests per feature.

_Correction:_ **importance weighting** — reweight training samples to match the deployment distribution. Fit a classifier to distinguish train from deploy samples; use its outputs as importance weights:

$$w(x) = \frac{P_{\text{deploy}}(x)}{P_{\text{train}}(x)} \approx \frac{P(\text{deploy} | x)}{P(\text{train} | x)}$$

Retrain with these weights, or use them to adjust predictions.

**Label shift (prior probability shift):** $P_{\text{train}}(y) \neq P_{\text{deploy}}(y)$, but $P_{\text{train}}(x|y) = P_{\text{deploy}}(x|y)$.

The class prior shifts but the class-conditional feature distributions are stable. This is the opposite factorisation from covariate shift.

_Example:_ a fraud detection model trained when fraud rate is 0.1% is deployed after a new attack campaign where fraud rate is 2%. The features of fraudulent transactions look the same, but there are many more of them.

_Correction:_ **prior correction**. If you can estimate $P_{\text{deploy}}(y)$, adjust the posterior:

$$P_{\text{deploy}}(y | x) \propto \frac{P_{\text{deploy}}(y)}{P_{\text{train}}(y)} \cdot P_{\text{train}}(y | x)$$

This requires an estimate of the new class prior — obtainable from a small labelled sample or domain knowledge.

**Concept drift:** $P_{\text{train}}(y|x) \neq P_{\text{deploy}}(y|x)$.

The true relationship between features and labels changes. This is the most severe form of shift — no weighting correction helps, because the model has learned the wrong function.

_Example:_ a model predicting housing prices learns that "proximity to city centre" positively predicts price. After a remote-work shift, suburban properties appreciate more. The model's learned $P(\text{price} | x)$ is now wrong for any $x$.

_Detection:_ requires labels — monitor model accuracy on newly labelled data over time. Performance decay signals concept drift.

_Correction:_ retraining, possibly with domain adaptation. No shortcut.

**Summary:**

|Type|What changes|Detectable without labels?|Correction|
|---|---|---|---|
|Covariate shift|$P(x)$|Yes|Importance weighting|
|Label shift|$P(y)$|With label proxy|Prior correction|
|Concept drift|$P(y\|x)$|No (needs labels)|Retrain|
|Train-serve skew|Feature computation|Feature comparison|Fix pipeline|

Train-serve skew deserves special mention: the features computed at training time are computed differently at serving time. Not a distributional shift — a pipeline bug. Examples: different SQL aggregation windows, timezone handling, NULL treatment. Often the most common "distribution shift" in practice, and entirely preventable with proper feature stores.

### 3.2 Formal Detection: Two-Sample Tests

Given training feature distribution ${x_i}_{i=1}^n$ and current deployment distribution ${x'_j}_{j=1}^m$, test whether the two samples are drawn from the same distribution.

**Kolmogorov-Smirnov test** (continuous features):

$$D_{n,m} = \sup_t |F_n(t) - G_m(t)|$$

where $F_n$ and $G_m$ are empirical CDFs. Under $H_0: F = G$, $D_{n,m}$ converges to the Kolmogorov distribution.

```python
from scipy.stats import ks_2samp

ks_stat, p_value = ks_2samp(X_train['lead_time'], X_current['lead_time'])
if p_value < 0.01:
    print(f"Drift detected: KS={ks_stat:.4f}, p={p_value:.4f}")
```

**Chi-squared test** (categorical features): test of homogeneity, $\chi^2 = \sum_k (O_k - E_k)^2 / E_k$.

**Maximum Mean Discrepancy (MMD)** — multivariate, in RKHS:

$$\text{MMD}^2(\mathcal{P}, \mathcal{Q}) = \mathbb{E}_{x,x' \sim \mathcal{P}}[k(x,x')] - 2\mathbb{E}_{x \sim \mathcal{P}, y \sim \mathcal{Q}}[k(x,y)] + \mathbb{E}_{y,y' \sim \mathcal{Q}}[k(y,y')]$$

With RBF kernel, MMD $= 0$ iff $\mathcal{P} = \mathcal{Q}$. Testing each feature independently ignores joint distribution shifts; MMD catches them.

**Classifier-based drift detection:**

Train a classifier to distinguish training samples (label 0) from deployment samples (label 1). If AUC $> 0.5$, distributions are distinguishable — drift exists. SHAP values on the drift classifier identify which features drifted most.

```python
X_combined = np.vstack([X_train, X_current])
y_combined = np.concatenate([np.zeros(len(X_train)), np.ones(len(X_current))])

drift_clf = LogisticRegression()
auc = cross_val_score(drift_clf, X_combined, y_combined, scoring='roc_auc').mean()
# AUC ≈ 0.5 → no drift.  AUC >> 0.5 → drift detected.
```

### 3.3 Population Stability Index (PSI)

PSI is a standard industry metric (originating in credit scoring) for quantifying feature distribution shift:

$$\text{PSI} = \sum_{b=1}^B (P_b - Q_b) \ln!\left(\frac{P_b}{Q_b}\right)$$

where $P_b$ is the fraction of training data in bin $b$, and $Q_b$ the fraction of deployment data. PSI is essentially the symmetrised KL divergence: $D_{\text{KL}}(P | Q) + D_{\text{KL}}(Q | P)$.

|PSI|Interpretation|
|---|---|
|$< 0.1$|No significant change|
|$0.1$ – $0.2$|Moderate change, investigate|
|$> 0.2$|Significant shift, action required|

```python
def psi(expected: np.ndarray, actual: np.ndarray, bins: int = 10) -> float:
    breakpoints = np.percentile(expected, np.linspace(0, 100, bins + 1))
    breakpoints[0] -= 1e-6;  breakpoints[-1] += 1e-6
    P = np.histogram(expected, breakpoints)[0] / len(expected)
    Q = np.histogram(actual,   breakpoints)[0] / len(actual)
    P = np.clip(P, 1e-6, None);  Q = np.clip(Q, 1e-6, None)
    return float(np.sum((P - Q) * np.log(P / Q)))
```

## ✦ Part IV — Monitoring

Monitoring is the observability layer: the system that detects, surfaces, and alerts on failures in production ML systems. Without monitoring, you are flying blind.

### 4.1 What to Monitor

The monitoring stack has four layers:

**Layer 1: Infrastructure metrics** (standard DevOps) — server CPU/memory/disk, API latency (p50, p95, p99), request throughput and error rate, container restarts.

**Layer 2: Model input metrics** (ML-specific) — feature distributions per time window (PSI, KS), missing value rates, feature range violations, input schema validation failures.

**Layer 3: Model output metrics** (ML-specific) — prediction distribution (mean, std, quantiles), class distribution in predictions, calibration: predicted probability vs. empirical positive rate.

**Layer 4: Ground truth metrics** (requires label collection) — accuracy, AUC, F1 on labelled live data, performance by subgroup, performance over time.

The difficulty: ground truth is often delayed. A hotel cancellation label is observed only when the booking date passes. A credit default label may take months. For delayed-label problems, Layers 1–3 are the primary early warning system; Layer 4 is the retrospective audit.

### 4.2 Alerting Strategy

Tiered alerts prevent alert fatigue:

|Alert tier|Condition|Response|
|---|---|---|
|P0 — Page immediately|API error rate $> 5%$, latency $> 5\times$ baseline|On-call engineer, rollback|
|P1 — Notify within 1 hour|Feature PSI $> 0.2$, prediction distribution shift|Investigation, possible retrain|
|P2 — Daily digest|Feature PSI $0.1$–$0.2$, slow calibration drift|Monitor, schedule retrain|

**Statistical alerting with CUSUM:**

Simple threshold alerts produce many false positives due to natural variability. CUSUM (Cumulative Sum) detects sustained shifts more reliably:

$$S_t = \max(0,; S_{t-1} + (x_t - \mu_0) - k)$$

where $\mu_0$ is the in-control mean and $k$ is the allowable slack. An alarm is raised when $S_t > h$. CUSUM is sensitive to small but persistent shifts — exactly the signature of distribution drift.

### 4.3 The Monitoring Stack

**Prometheus + Grafana** (standard open-source):

## ✦ Part V — Model Versioning and Rollback

### 5.1 Semantic Versioning for Models

```
model_name-v{major}.{minor}.{patch}

v1.0.0 — first production deployment
v1.1.0 — retrained on 3 additional months of data, same architecture
v1.2.0 — added two new features
v2.0.0 — changed model type (random forest → XGBoost), incompatible input schema
```

Major version bumps signal input/output schema changes — clients must update. Minor and patch bumps are backward-compatible.

### 5.2 The MLflow Model Registry Lifecycle

```
Training run → Registered model → Staging → Production → Archived
```

```python
from mlflow.tracking import MlflowClient

client = MlflowClient()

# Transition to staging after training
client.transition_model_version_stage(
    name="hotel-cancellation", version=7, stage="Staging")

# After validation: promote to production, archive v6
client.transition_model_version_stage(
    name="hotel-cancellation", version=7, stage="Production",
    archive_existing_versions=True)

# Load production model anywhere
model = mlflow.sklearn.load_model("models:/hotel-cancellation/Production")
```

The registry provides a single source of truth: "what model is currently in production?" is always answerable, as is "what hyperparameters does it have?" and "what data was it trained on?"

### 5.3 Deployment Strategies

**Blue-green deployment:** run two identical environments (blue = current, green = new). Route all traffic to green after validation. Rollback = switch traffic back to blue. Zero downtime, but doubles infrastructure cost.

```
Load Balancer
    ├── 100% → blue (v6, currently live)
    └── 0%  → green (v7, warming up)

# Validation passes:
    ├── 0%  → blue (v6, standby for rollback)
    └── 100% → green (v7, now live)
```

**Canary deployment:** route a small fraction of traffic (1–5%) to the new model, monitor metrics, gradually increase fraction if healthy.

```
Traffic split:
    ├── 95% → v6 (production)
    └── 5%  → v7 (canary)
    # After 24h, healthy → ramp to 20% → 50% → 100%
```

Canary is safer for ML because it allows real-world validation with bounded blast radius. If v7 has a bug, only 5% of users are affected.

**Shadow deployment (dark launch):** route all traffic to v6 but also send it asynchronously to v7. Log v7's predictions but don't serve them. Compare predictions offline before affecting anyone. Gold standard for validating behaviour without any user risk, but requires running two models simultaneously.

## ✦ Part VI — Retraining and the ML Flywheel

### 6.1 When to Retrain

|Signal|Likely cause|Response|
|---|---|---|
|Feature PSI $> 0.2$|Covariate shift|Retrain on recent data|
|Prediction distribution shift|Covariate or label shift|Investigate + retrain|
|Accuracy decay (with labels)|Concept drift|Retrain, possibly with new features|
|Business metric decay|Multiple possible causes|Error analysis + retrain|
|Scheduled|Preventive, data accumulation|Retrain on schedule|

**Triggered retraining** (event-based): retrain when monitoring detects drift. Reactive but responsive.

**Scheduled retraining** (time-based): retrain on a fixed schedule (weekly, monthly) regardless of detected drift. Simpler to implement, appropriate when drift is expected and relatively constant.

**Online learning** (continual): update model parameters as new data arrives, without a discrete retrain step. Appropriate for highly non-stationary environments (real-time recommendation, financial markets). Requires careful engineering to prevent catastrophic forgetting.

### 6.2 The Training Data Window

When retraining, how much history to use?

**Full history:** more data, better statistical estimates. But older data may reflect a different $P(y|x)$ under concept drift.

**Rolling window:** use only the last $W$ days. Adapts to recent patterns, reduces stale data influence. Loses long-term and rare-event patterns.

**Exponential decay weighting:** weight recent observations more heavily without discarding old data:

$$w_t = \lambda^{T - t}, \quad 0 < \lambda < 1$$

$\lambda = 0.99$ gives recent data $\sim 100\times$ more weight than data from $\approx 460$ days ago. The optimal $\lambda$ is problem-specific and should be selected by backtesting the retraining policy on historical data.

### 6.3 The ML Flywheel

In successful ML products, deployment creates a feedback loop that improves the model over time:

```
Deploy model
    → Model makes predictions
    → Users interact with predictions (accept, reject, ignore)
    → Interactions become labels (implicit or explicit feedback)
    → Labels go into training data
    → Retrain on richer dataset
    → Better model → Deploy → ...
```

**The flywheel is also a liability if mismanaged:**

**Selection bias:** if the model decides which inputs to act on (e.g., content recommendation), the feedback loop is biased. You only observe outcomes for content the model showed — not for content it didn't show. This is the **missing counterfactual problem**.

**Self-fulfilling prophecy feedback loops:** a model predicts high cancellation probability and triggers an overbooking policy, causing customers to actually cancel, confirming its prediction. Common in recommendation, pricing, and risk systems.

**Label leakage through time:** if the prediction influences the outcome (e.g., a fraud prediction triggers a review that catches fraud), labels from periods when the model was active are not ground truth — they are partly a function of the model's own predictions.

These pathologies require careful experiment design (holdout groups, randomised interventions) and are active research areas in causal ML.

## ✦ Part VII — A/B Testing and Online Evaluation

### 7.1 The Offline-Online Gap

A model with higher offline AUC does not necessarily have higher online business metrics. The offline-online gap exists because offline evaluation uses historical labels collected under the old model's policy; the new model changes behaviour, which changes which outcomes are observed; and user responses to model outputs are not captured in offline data.

**The only ground truth is online performance.** Offline evaluation selects candidates; online A/B tests validate them.

### 7.2 A/B Test Design for ML Models

An A/B test for a model change is a randomised controlled experiment: Control (A) = existing model, Treatment (B) = new model. Randomisation is typically at the user or booking ID level (consistent assignment).

**Sample size calculation:**

For a two-sided test detecting effect size $\delta$ in a metric with standard deviation $\sigma$, at significance level $\alpha$ and power $1 - \beta$:

$$n \geq \frac{2\sigma^2 (z_{1-\alpha/2} + z_{1-\beta})^2}{\delta^2}$$

For $\alpha = 0.05$, $\beta = 0.2$: $(z_{0.975} + z_{0.8})^2 \approx 7.85$, so $n \geq 15.7\sigma^2/\delta^2$.

```python
from scipy.stats import norm
import numpy as np

def required_sample_size(baseline_rate, mde, alpha=0.05, power=0.80):
    """Two-sided test for proportions. mde: minimum detectable effect (absolute)."""
    p1, p2 = baseline_rate, baseline_rate + mde
    z_alpha, z_beta = norm.ppf(1 - alpha/2), norm.ppf(power)
    p_bar = (p1 + p2) / 2
    n = (z_alpha*(2*p_bar*(1-p_bar))**0.5 + z_beta*(p1*(1-p1)+p2*(1-p2))**0.5)**2
    return int(np.ceil(n / (p2 - p1)**2))

# Baseline cancellation = 30%, detect 2pp improvement
print(required_sample_size(0.30, 0.02))
```

**Common A/B testing pitfalls:**

**Novelty effect:** users interact differently with a new system simply because it is new. Run tests long enough to see post-novelty behaviour.

**Network effects:** if users interact with each other, a user's experience in the control group is affected by treatment-group users. This violates SUTVA. Use cluster-level randomisation instead.

**Multiple testing:** running many simultaneous tests inflates the false positive rate. Apply Bonferroni correction or use sequential testing (always-valid p-values) for continuous monitoring.

**Deep learning (PyTorch/TF):** same concepts, different tools. TorchServe or Triton instead of FastAPI; GPU-aware containerisation; higher stakes for latency (transformer inference is expensive). PyTorch Lightning wraps training + logging + checkpointing in a clean contract:

**Large language models:** the challenges are the same but amplified. Serving a 70B parameter model requires sharded deployment across multiple GPUs. Distribution shift is harder to detect (inputs are text — defining "the distribution" requires embedding-space monitoring). The feedback loop (RLHF) is itself a training procedure. Monitoring requires human evaluation on samples, not just automated metrics.

### 8.2 Model Compression for Deployment

Large neural networks are often too slow or too large for production. Three main approaches:

**Quantisation:** reduce parameter precision from float32 to float16, int8, or int4. 8-bit quantisation typically recovers $\sim 99%$ of full-precision accuracy with $4\times$ memory reduction and $2$–$4\times$ speedup.

```python
import torch.quantization

model_quantized = torch.quantization.quantize_dynamic(
    model, {torch.nn.Linear}, dtype=torch.qint8)
```

**Pruning:** set small weights to zero (unstructured) or remove entire neurons/attention heads (structured). Structured pruning produces smaller, faster models; unstructured pruning requires sparse computation support to realise speedups.

**Knowledge distillation:** train a small "student" model to mimic a large "teacher" model. The student minimises $D_{\text{KL}}(\text{student output} | \text{teacher output})$ rather than just the hard labels. Distillation often achieves $> 90%$ of teacher performance at $10%$ of the parameters.

```
Teacher (large model): [features] → [soft probability distribution]
Student (small model): minimise KL(student || teacher) + α · cross_entropy(student, labels)
```

**ONNX + TensorRT:** converting to ONNX and compiling with TensorRT typically yields $3$–$10\times$ speedup on NVIDIA GPUs through kernel fusion and precision optimisation.

### 8.3 The Alignment Connection

For LLMs in production, distribution shift takes on a different character. The model's outputs can themselves shift user behaviour — a recommendation model shifts what users watch; a language model shifts how users write. This is the feedback loop problem at civilisational scale, and it is one motivation for the field of AI alignment: ensuring that deployed ML systems continue to behave in intended ways as they interact with and shape the world.

The monitoring and evaluation concepts here — detecting when a model's behaviour deviates from intended, designing feedback loops that don't corrupt themselves — are the engineering manifestation of alignment concerns that operate at a much deeper level for general-purpose AI systems.

## ✦ The Hidden Technical Debt Paper — Key Lessons

Sculley et al. (2015) identified structural patterns that accumulate debt in ML systems. Worth knowing explicitly:

**Entanglement:** when multiple features are fed to a model together, changing any one feature changes the model's effective use of all others. Adding a new feature may silently degrade performance on previously well-served slices. The CACE principle: Changing Anything Changes Everything.

**Correction cascades:** when model A's output is an input to model B, B's behaviour is coupled to A's performance. Errors compound and debugging causality is difficult.

**Undeclared consumers:** a model's outputs are consumed by downstream systems that may not be aware of its versioning or schema. Silently changing output format breaks consumers.

**Configuration debt:** ML systems have many hyperparameters, thresholds, and flags. Without discipline, configuration accumulates until no one knows what each knob does or why it was set.

**Pipeline jungle:** multiple overlapping preprocessing pipelines for the same data, maintained by different teams, producing subtly different features. The model was trained on pipeline A; production uses pipeline B. Welcome to train-serve skew.

The prescription: treat ML systems with the same engineering discipline as any software system — versioning, testing, documentation, code review, modular design. The notebook is a prototype; the production ML system is software.
