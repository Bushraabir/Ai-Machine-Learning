## ✦ The Shape of the Problem

Before a single line of code, the most important skill in ML is **problem framing**. Most project failures are not algorithmic — they are framing failures. A beautifully optimised model solving the wrong problem is waste.

The three questions to answer before anything else:

1. **What is the task $T$?** — classification, regression, ranking, generation, clustering?
2. **What does success look like?** — what metric $P$ matters _to the business_, not just to a paper?
3. **What data exists and is it sufficient?** — do you actually have experience $E$?

These three questions correspond directly to Mitchell's definition of learning: $T$, $P$, $E$. You are instantiating that abstract definition into a concrete system.

## ✦ Step 0 — Frame the Problem

### 0.1 Task Taxonomy

Before anything else, know which regime you are in:

|Task|$\mathcal{Y}$|Loss (canonical)|Baseline|
|---|---|---|---|
|Regression|$\mathbb{R}$|MSE / MAE|Predict mean of training targets|
|Binary classification|${0,1}$|Cross-entropy|Predict majority class|
|Multi-class|${1,...,K}$|Cat. cross-entropy|Predict most frequent class|
|Ranking|Ordered pairs|Pairwise loss|Sort by popularity|
|Sequence labelling|$\mathcal{Y}^T$|Token cross-entropy|Most frequent tag|
|Clustering|—|No supervised loss|—|

### 0.2 The Metric Must Match the Business Problem

This is where projects die. The canonical traps:

**Trap 1: Optimise accuracy on an imbalanced problem.** Fraud detection: 99.9% of transactions are legitimate. A classifier that always predicts "not fraud" achieves 99.9% accuracy and is useless. The right metrics are **precision, recall, AUC-PR**.

**Trap 2: Optimise the wrong loss.** If the business cost of a false negative is 100× the cost of a false positive (e.g., medical diagnosis, content moderation), your loss function should encode this asymmetry explicitly — class-weighted cross-entropy, or a custom loss.

**Trap 3: Confuse online and offline metrics.** A recommendation system may achieve excellent offline NDCG but perform poorly online (A/B test) because users interact with the recommendations, changing the distribution. Offline metrics are proxies; the ground truth is online behaviour.

**Rule:** always derive your offline metric from the business objective, and document the derivation.

### 0.3 The Data Audit

Before modelling, you need honest answers to:

- How was the data collected? (Introduces selection bias if collection mechanism correlates with target)
- How large is $n$? Rule of thumb: need at least $\sim 10 \times d$ observations for a linear model to be well-posed; deep learning often needs $10^5$–$10^9$.
- Is the data labelled? If yes: who labelled it? What was their agreement (inter-annotator reliability)?
- Is there a time dimension? If yes: do not shuffle randomly — preserve temporal order.
- Are there ethical or legal constraints on features? (Sensitive attributes, GDPR, etc.)

## ✦ Step 1 — Get & Version the Data

### 1.1 The Train / Val / Test Split

This deserves a brief but rigorous treatment because getting it wrong corrupts everything downstream. See [[01.Train Val Test]] for the full discussion.

**The cardinal rule:** your test set represents the world the model will be deployed into. It must be:

1. Drawn from the same distribution as deployment data
2. Never touched during model development
3. Used at most once (or very few times — each peek costs you)

**The split strategy depends on the data structure:**

- **i.i.d. data:** random stratified split. Typical: 70/15/15 or 60/20/20.
- **Time-series data:** chronological split — everything before $t_0$ is train, $[t_0, t_1]$ is val, $[t_1, \infty)$ is test. _Never shuffle._
- **Group structure** (e.g., multiple records per patient): group-aware split — all records from a patient go to the same fold. Otherwise your model learns the patient, not the disease.
- **Geospatial data:** spatial block splits to prevent leakage through spatial autocorrelation.

**Example — the patient leakage problem:**

Suppose you are predicting hospital readmission. Each patient has 10 visits. If you randomly split rows, the model sees 8 visits from patient $i$ in training and 2 in test. It learns to recognise patient $i$ by their demographic and history — it's doing patient matching, not generalising to new patients. **Group k-fold splits by patient ID.**

### 1.2 Data Versioning and Provenance

In any serious project, data is a first-class artefact. Version it.

- **DVC (Data Version Control):** tracks datasets in Git, stores them in object storage (S3, GCS, Azure Blob)
- **Feature stores** (Feast, Tecton, Hopsworks): centralise feature computation, prevent train-serve skew
- Document every preprocessing step: transformations applied, columns dropped, imputation strategies. The pipeline itself _is_ the documentation — use sklearn `Pipeline` objects (see [[02.Pipelines and Experiments]]).

## ✦ Step 2 — Exploratory Data Analysis

EDA is not optional and not aesthetic. It is the stage where you discover:

- Whether the problem is solvable given the data
- What preprocessing the data requires
- Whether the signal exists at all
- Whether there is leakage

### 2.1 Distributions

For every feature $x_j$:

```python
import pandas as pd
import matplotlib.pyplot as plt
import seaborn as sns

df.describe()           # count, mean, std, min, quartiles, max
df.isnull().mean()      # fraction missing per column
df.hist(bins=50, figsize=(20, 15))  # all numeric features
sns.boxplot(...)        # outlier inspection
```

**What to look for:**

- **Skewness:** right-skewed features (incomes, counts) hurt linear models — consider log transform
- **Bimodality:** may indicate a hidden grouping variable that should be an explicit feature
- **Spikes at round numbers:** often imputation artefacts (e.g., age 99 = unknown)
- **Feature ranges:** huge range differences → must scale for distance-based models and NNs

### 2.2 Correlations and Target Relationships

```python
corr = df.corr()
sns.heatmap(corr, annot=True)

# For classification: group by target
df.groupby('target').mean()

# Mutual information (handles nonlinear relationships)
from sklearn.feature_selection import mutual_info_classif
mi = mutual_info_classif(X, y)
```

High mutual information between a feature and target is a signal worth investigating. Suspiciously high mutual information (e.g., $I(x_j; y) > 0.9$) should trigger a leakage investigation.

### 2.3 The Leakage Hunt

**Data leakage** is when information about the target bleeds into the features in a way that wouldn't exist at inference time. It is the most common reason ML models fail in production.

**Categories of leakage:**

**1. Target leakage:** a feature encodes the target (possibly indirectly). Example: you are predicting loan default, and one feature is "received debt forgiveness" — which only happens _after_ default. At inference time, this feature is 0 for all live loans, but the model learned to rely on it.

**2. Temporal leakage:** you use future information to predict the past. A feature computed from data after the prediction timestamp. Arises from careless joins on rolling windows.

**3. Training-set contamination:** preprocessing (e.g., StandardScaler) fit on the entire dataset including test. The test set "told" the scaler its statistics. Fix: fit all preprocessing only on training data.

**4. Duplicate rows:** if the same observation appears in both train and test sets, test accuracy measures memorisation.

**Leakage detection heuristic:** if your model achieves suspiciously high CV accuracy (e.g., 99% on a hard real-world problem), investigate leakage before celebrating.

**Formal check:** For each feature, examine: could this feature's value at prediction time $t$ have been derived from the target $y$ or from events occurring after $t$?

## ✦ Step 3 — Preprocessing & Feature Engineering Pipeline

See [[01.Data Preprocessing]] and [[02.Feature Engineering]] for full treatment. The key principle here is **pipeline discipline**.

### 3.1 The Leakage-Safe Pipeline

All preprocessing must be **fit on the training fold only and applied to all folds.** The sklearn `Pipeline` enforces this:

```python
from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer

numeric_transformer = Pipeline(steps=[
    ('imputer', SimpleImputer(strategy='median')),
    ('scaler', StandardScaler())
])

categorical_transformer = Pipeline(steps=[
    ('imputer', SimpleImputer(strategy='most_frequent')),
    ('encoder', OneHotEncoder(handle_unknown='ignore'))
])

preprocessor = ColumnTransformer(transformers=[
    ('num', numeric_transformer, numeric_features),
    ('cat', categorical_transformer, categorical_features)
])

full_pipeline = Pipeline(steps=[
    ('preprocessor', preprocessor),
    ('classifier', LogisticRegression())
])

# Cross-validation with Pipeline: fit/transform correctly per fold
from sklearn.model_selection import cross_val_score
scores = cross_val_score(full_pipeline, X_train, y_train, cv=5, scoring='roc_auc')
```

**Why this matters:** if you fit the `StandardScaler` on the full dataset outside the pipeline, then do CV, the validation fold's statistics have already "leaked" into the scaler. The scaler was trained on data that includes the validation samples. This is subtle and common.

### 3.2 The Complexity Ladder for Feature Engineering

Start minimal. Add complexity only if CV score improves:

```
Raw features
    → Handle missingness
    → Scale / encode
    → Log-transform skewed features
    → Interaction features (x_i · x_j)
    → Polynomial features (x_i^2)
    → Domain-specific constructions (e.g., ratios, windowed aggregates)
    → Learned embeddings (e.g., entity embeddings via NN, see deep learning)
```

Each step should be motivated by EDA, not applied blindly.

## ✦ Step 4 — Baseline → Model Selection → Iteration

### 4.1 The Baseline Imperative

**Always start with a dumb baseline.** This is not intellectual laziness — it is professional discipline.

The baseline establishes:

- Whether the problem is solvable at all (if your model barely beats the baseline, the signal may not exist)
- What "meaningful improvement" means quantitatively
- The performance floor against which all complexity is justified

**Standard baselines by task:**

|Task|Baseline|
|---|---|
|Regression|Predict $\bar{y}_{\text{train}}$ for all test points|
|Classification|Predict majority class (or class frequency)|
|Time series|Predict the last observed value (naïve forecast)|
|Ranking|Sort by global popularity|
|NLP|TF-IDF + logistic regression|

After the dumb baseline: **linear model baseline** (logistic regression / ridge regression). Cheap, fast, interpretable. Sets the bar for "is nonlinearity worth it?"

### 4.2 The Complexity Ladder for Models

Traverse this ladder in order. Add complexity only if CV improvement justifies it:

```
Majority class / mean predictor
    ↓
Linear model (logistic regression / ridge)
    ↓
Tree-based models (decision tree, then random forest)
    ↓
Gradient boosting (XGBoost / LightGBM)
    ↓
Neural networks
    ↓
Ensembles / stacking
```

**Why this order?** Each step:

- Is harder to train and tune than the previous
- Requires more data to generalise
- Is less interpretable
- May not improve over the simpler predecessor

The practitioner's heuristic: if the problem has tabular data with mixed feature types and $n < 10^6$, gradient boosting (XGBoost/LightGBM) is almost always the strongest out-of-the-box model. If $n > 10^6$ with structured data — still gradient boosting. For images, text, audio — neural networks.

### 4.3 Model Selection with Cross-Validation

**Never use test set for model selection.** Use cross-validation on the training set.

For standard i.i.d. data, stratified 5-fold or 10-fold CV is the workhorse. For time-series, forward-chaining CV:

```
Fold 1: train [1, t₁], val [t₁+1, t₂]
Fold 2: train [1, t₂], val [t₂+1, t₃]
Fold 3: train [1, t₃], val [t₃+1, t₄]
```

For simultaneous hyperparameter tuning and model evaluation, use **nested CV** (outer loop evaluates generalisation, inner loop selects hyperparameters). Standard CV for both gives an optimistically biased estimate.

See [[02.Cross Validation]] and [[05.Hyperparameter Tuning]] for details.

## ✦ Step 5 — Evaluation on the Held-Out Test Set

### 5.1 The Test Set as a Finite Resource

The test set provides an estimate of generalisation error. Every time you look at test performance and adjust the model or threshold or preprocessing accordingly, you perform **implicit optimisation on the test set**. After enough such peeks, test performance becomes an optimistic estimate — the variance of your peek-decisions is counted as signal.

**Spend the test set once.** This is the reason you need a validation set during development.

### 5.2 Reporting Multiple Metrics

A single metric rarely tells the full story. Standard practice:

**For classification:** report accuracy, precision, recall, F1, AUC-ROC, and AUC-PR. Disaggregate by class and by subgroup (see fairness).

**For regression:** report RMSE, MAE, and $R^2$. Also plot residuals — residual patterns reveal systematic failures invisible in summary statistics.

**For probabilistic models:** report calibration (reliability diagram) and log-loss. A model can have good AUC-ROC but terrible calibration — its probability estimates are unreliable even if its rankings are correct.

### 5.3 Statistical Significance

If your improvement over baseline is 0.3% AUC, is that real? A McNemar test (for classification), t-test over CV folds (with appropriate correction, e.g., Nadeau-Bengio), or bootstrap confidence intervals on the test metric are standard. Don't report improvements without uncertainty estimates.

## ✦ Step 6 — Error Analysis

> "Look at your mistakes" is one of the highest-ROI activities in ML. Ten minutes of error analysis often beats one week of architecture search.

### 6.1 The Error Analysis Loop

```
1. Collect the model's predictions on the validation set
2. Find the instances where the model is most wrong (largest loss, most confident errors)
3. Manually examine them — form hypotheses about why
4. Verify the hypotheses quantitatively
5. Fix: add data, add features, change model, adjust threshold
6. Repeat
```

### 6.2 Confusion Matrix Deep-Dive

For a classifier producing a $K \times K$ confusion matrix:

- Off-diagonal entries are errors. Which ones are large?
- Which classes are confused with each other? Does this make semantic sense?
- Are there systematic patterns? (e.g., model consistently confuses class 3 with class 7 — maybe these classes need a distinguishing feature)

### 6.3 Sliced Evaluation

Aggregate metrics hide failures in subpopulations. Always disaggregate:

- By feature buckets (e.g., short documents vs. long documents, young users vs. old users)
- By data source (e.g., mobile vs. desktop, US vs. EU)
- By time (is performance degrading over time? This is the first sign of distribution shift)

**Example:** a fraud detection model with 95% overall AUC may have 70% AUC on transactions from a new merchant category that was underrepresented in training. Without sliced evaluation, this failure is invisible.

### 6.4 Confidence Calibration

The model's predicted probability $\hat{p}$ should match empirical frequency: among all instances where $\hat{p} \approx 0.7$, about 70% should actually be positive. Plot the **reliability diagram** (calibration curve):

```python
from sklearn.calibration import calibration_curve
fraction_of_positives, mean_predicted_value = calibration_curve(y_test, y_prob, n_bins=10)
plt.plot(mean_predicted_value, fraction_of_positives, "s-")
plt.plot([0, 1], [0, 1], "k:", label="Perfectly calibrated")
```

Overconfident models (reliability curve below the diagonal) and underconfident models have different fixes: temperature scaling, Platt scaling, isotonic regression.

## ✦ Step 7 — Deployment Considerations (Orientation)

See [[04.Deployment and MLOps]] for the full treatment. The conceptual anchors:

### 7.1 Train-Serve Skew

The distribution of features at inference time may differ from training time. Sources:

- **Covariate shift:** $P(x)$ changes, $P(y|x)$ stays the same. (e.g., user demographics shift over time)
- **Label shift:** $P(y)$ changes. (e.g., fraud patterns evolve)
- **Concept drift:** $P(y|x)$ changes. (e.g., the same symptoms now indicate a different disease post-pandemic)

All three require different interventions. Covariate shift can be corrected with importance weighting. Label shift with prior correction. Concept drift requires retraining.

### 7.2 The ML Flywheel

In products, ML is not a one-shot deployment — it is a continuous loop:

```
More data → Better model → Better product → More users → More data
```

This means the deployment and monitoring infrastructure is as important as the model itself.

## ✦ Worked Example — End-to-End Tabular Project

**Problem:** predict whether a hotel booking will be cancelled (binary classification).

**Step 0 — Frame:**

- Task: binary classification
- Business metric: revenue impact of false positives (overbooking costs) vs. false negatives (empty rooms). Cost-weighted F-beta score.
- Data: booking records with features like lead time, ADR, special requests, customer history.

**Step 1 — Data:**

- Historical booking data with cancellation labels.
- Temporal split: train on bookings before 2018, val 2018, test 2019.
- Group by booking customer — same customer must not appear in multiple folds.

**Step 2 — EDA:**

- `lead_time` is heavily right-skewed → log transform.
- `adr` (average daily rate) has outliers at 0 (free rooms) and 5000+ (typos?) → clip.
- `country` has 170 unique values → target encoding or frequency encoding.
- High mutual information: `deposit_type` (non-refundable → almost always kept). _Investigate: is this leakage?_ It is not — deposit type is known at booking time.

**Step 3 — Features:**

```python
Pipeline: [
    ColumnTransformer([
        ('num', StandardScaler(), num_cols),
        ('cat', OneHotEncoder(sparse=False), cat_cols)
    ]),
    LogisticRegression()
]
```

**Step 4 — Models:**

|Model|Val AUC-ROC|
|---|---|
|Majority class|0.500|
|Logistic Regression|0.789|
|Random Forest|0.841|
|XGBoost|0.878|
|XGBoost + feature eng.|0.891|

Gradient boosting wins. Hyperparameter tuning via Bayesian optimisation: `n_estimators=600`, `max_depth=6`, `learning_rate=0.05`, `subsample=0.8`.

**Step 5 — Test Evaluation:** Test AUC-ROC = 0.887. Calibration is slightly off — apply isotonic regression.

**Step 6 — Error Analysis:** Most confident false negatives are transient bookings (1-night stays) with last-minute modifications. Feature: `days_since_last_modification / lead_time`. Val AUC improves to 0.894.

## ✦ Relation to AI / ML / Deep Learning

This workflow is not specific to tabular models — it is the universal scaffolding into which every ML system fits:

**Classical ML → Deep Learning:** Steps 0–2 (framing, data, EDA) are identical. Steps 3–4 change: feature engineering is largely replaced by **representation learning** (the neural network learns the features); model selection becomes architecture search (depth, width, attention, convolution). The stakes are higher — data requirements and compute are orders of magnitude larger, and overfitting is a more severe risk.

**The deep learning practitioner's workflow** follows the same logic but with different tools:

- Baseline = linear probe on pretrained features (e.g., fine-tune last layer of ResNet)
- Error analysis = activation maximisation, saliency maps, attention visualisation
- Evaluation = same metrics, but also robustness to distribution shift, adversarial examples
- Deployment = model compression (pruning, quantisation, distillation), serving infrastructure (TorchServe, TFServing)

**Supervised learning pipeline → this note:**

$$\text{Problem} \xrightarrow{\text{frame}} (\mathcal{X}, \mathcal{Y}, \mathcal{D}) \xrightarrow{\text{data}} \hat{\mathcal{D}} \xrightarrow{\text{feature eng.}} \tilde{\mathcal{X}} \xrightarrow{\text{ERM}} \hat{f} \xrightarrow{\text{evaluate}} R(\hat{f}) \xrightarrow{\text{error analysis}} \text{iterate}$$

The formalisation from [[02.Statistical Learning Theory]]: we are approximating the Bayes-optimal $f^* = \arg\min_f R(f)$ by solving empirical risk minimisation $\hat{f} = \arg\min_f \hat{R}(f)$ subject to inductive bias from $\mathcal{H}$, and using the entire workflow to control the generalisation gap $R(\hat{f}) - \hat{R}(\hat{f})$.

**Reinforcement learning** has a different structure but analogous concerns: environment design (analogous to problem framing), reward shaping (analogous to metric design), exploration vs. exploitation tradeoffs (analogous to model selection under uncertainty), and distribution shift from a changing policy.

## ✦ The Meta-Lesson

The single most important insight about ML in practice:

> **Data quality, problem framing, and error analysis matter more than model choice.**

The history of deployed ML systems suggests roughly:

- ~40% of failures: wrong problem framing or metric
- ~30% of failures: data issues (leakage, distribution shift, quality)
- ~20% of failures: deployment and monitoring failures
- ~10% of failures: model or algorithm choices

Most of the literature — and most of the time spent in courses — addresses the last 10%.
