## ✦ Why Interpretability Is Not Optional

There is a lazy framing of interpretability as a trade-off against accuracy — choose one. This is mostly wrong. Interpretability serves four distinct purposes, none of which are about sacrificing performance:

**1. Debugging.** A model that achieves 95% CV accuracy and learns that "hospital_id" predicts mortality is wrong, not good. Without interpretability you cannot detect this.

**2. Trust and deployment.** In regulated domains (credit, medicine, criminal justice), a prediction that cannot be explained is legally unusable in many jurisdictions. EU GDPR Article 22 provides a right to explanation for automated decisions.

**3. Scientific discovery.** When the model is a proxy for understanding a real process — genomics, drug discovery, climate — the features it learns _are_ the finding.

**4. Distribution shift detection.** If the model's top feature at deployment changes from what it was at training time, you are in trouble. You can only notice this if you track feature importance over time.

The accuracy-interpretability trade-off is real but narrower than commonly stated: for tabular data with $n < 10^6$, linear models and gradient-boosted trees are nearly always competitive with neural networks, and both are interpretable. The trade-off bites mainly in unstructured data (images, text, audio) where representation learning matters and deep networks dominate.

## ✦ A Taxonomy

```
Interpretability Methods
│
├── Intrinsic (model is interpretable by construction)
│   ├── Linear models          — coefficients are the explanation
│   ├── Decision trees         — the path is the explanation
│   └── Rule lists / GAMs      — additive, human-readable
│
└── Post-hoc (explain a fitted black-box model)
    │
    ├── Global (explain the model's overall behaviour)
    │   ├── Feature importance (impurity-based)
    │   ├── Permutation importance
    │   ├── Partial Dependence Plots (PDP)
    │   └── SHAP global summaries
    │
    └── Local (explain a single prediction)
        ├── LIME
        └── SHAP (local)
```

The global/local and intrinsic/post-hoc axes are independent. SHAP can be both global and local; a linear model is both intrinsic and global.

## ✦ Part I — Intrinsically Interpretable Models

### 1.1 Linear Models

A linear model $\hat{y} = w^T x + b$ has an exact, complete explanation built in: $w_j$ is the change in $\hat{y}$ per unit increase in $x_j$, holding all other features constant (the _ceteris paribus_ condition).

**But this interpretation requires correct setup:**

- Features must be standardised for coefficients to be _comparable_. Raw $w_j$ on unstandardised features reflects both the importance of feature $j$ and its scale. Standardise first: $\tilde{x}_j = (x_j - \mu_j)/\sigma_j$, then $w_j$ represents "change in $\hat{y}$ per standard deviation increase in $x_j$."
    
- L1 regularisation (Lasso) drives coefficients to exactly zero — implicit feature selection. The non-zero coefficients are the model's statement about which features matter.
    
- L2 regularisation (Ridge) shrinks all coefficients but keeps them non-zero. With highly correlated features, Ridge spreads weight across the group; Lasso arbitrarily selects one. Neither gives the true per-feature contribution in the presence of collinearity — SHAP does.
    

**Logistic regression:** the coefficient $w_j$ is the change in _log-odds_ per unit increase in $x_j$. Exponentiating gives the odds ratio $e^{w_j}$. The log-odds interpretation is exact; it is non-linear in probability space, which is often misunderstood.

### 1.2 Decision Trees

A single decision tree is fully interpretable: for any prediction, you can trace the exact path from root to leaf, read off each split condition, and hand the reasoning to a human.

```
Is lead_time > 45?
├── Yes: Is deposit_type == "Non_Refund"?
│   ├── Yes: → Predict CANCELLED  (leaf, 847/901 positive)
│   └── No:  → Predict KEPT       (leaf, 312/489 positive)
└── No:  → Predict KEPT           (leaf, 1203/1411 positive)
```

This is a complete, human-auditable explanation. The limitation is that trees with depth $> 4$ become cognitively intractable — the number of paths grows as $2^d$.

**Decision rules** (extracted from trees or learned directly via rule induction algorithms like RIPPER) are the most interpretable form: `IF lead_time > 45 AND deposit_type = Non_Refund THEN cancellation_probability = 0.94`. Each rule is independently meaningful.

### 1.3 Generalised Additive Models (GAMs)

GAMs occupy a middle ground: more expressive than linear models, more interpretable than arbitrary nonlinear models.

$$\hat{y} = \beta_0 + \sum_{j=1}^d f_j(x_j)$$

where each $f_j$ is a smooth univariate function (spline, polynomial, step function). Because the model is additive, $f_j(x_j)$ shows the _isolated_ contribution of feature $j$ to the prediction — there are no interaction terms (unless you add them explicitly).

**Neural Additive Models (NAMs)** (Agarwal et al., 2021) learn the $f_j$ as small neural networks while preserving the additive structure — giving the expressiveness of neural networks with the interpretability of a GAM. This is a live research area.

## ✦ Part II — Post-Hoc Global Methods

### 2.1 Impurity-Based Feature Importance

For tree-based models, sklearn computes feature importance as the **mean decrease in impurity (MDI)**: the total reduction in the splitting criterion (Gini, variance) attributable to each feature, averaged over all trees.

$$\text{Importance}_j = \frac{1}{|\mathcal{T}|} \sum_{t \in \mathcal{T}} \sum_{\substack{v \in t \ \text{split on } j}} \Delta I(v)$$

where $\Delta I(v)$ is the weighted impurity decrease at node $v$.

```python
import pandas as pd
importances = pd.Series(
    rf.feature_importances_,
    index=feature_names
).sort_values(ascending=False)
```

**Why MDI is unreliable and when to avoid it:**

MDI has two serious flaws identified in the literature:

**Flaw 1: Bias toward high-cardinality features.** A feature with many unique values can split the data in more ways, creating more opportunities for impurity reduction by chance. A random float column will appear important in MDI. This is not a bug in sklearn — it is a property of the impurity criterion itself. Strobl et al. (2007) demonstrated this rigorously.

**Flaw 2: Computed on training data.** MDI uses the training set impurity. A feature that overfits — memorising training noise — will have high MDI but zero predictive value on held-out data.

Use MDI only for fast, rough exploration. For anything consequential, use permutation importance or SHAP.

### 2.2 Permutation Importance

**Idea:** a feature is important if randomly permuting its values (breaking the feature-target relationship) degrades model performance on a held-out set.

$$\text{PI}_j = \text{score}(f, X_{\text{val}}, y_{\text{val}}) - \text{score}(f, X_{\text{val}}^{(\pi_j)}, y_{\text{val}})$$

where $X_{\text{val}}^{(\pi_j)}$ denotes the validation set with column $j$ randomly permuted, and the permutation is repeated $K$ times (typically $K = 10$) to get a distribution.

```python
from sklearn.inspection import permutation_importance

result = permutation_importance(
    model, X_val, y_val,
    n_repeats=10,
    random_state=42,
    scoring='roc_auc',
    n_jobs=-1
)

pi = pd.DataFrame({
    'feature':    feature_names,
    'importance': result.importances_mean,
    'std':        result.importances_std
}).sort_values('importance', ascending=False)
```

**Properties:**

- **Model-agnostic:** works for any model with a `predict` or `predict_proba` method
- **Computed on validation data:** reflects generalisation, not training fit
- **Unbiased for cardinality:** a random feature will have PI $\approx 0$ regardless of its number of unique values
- **Correlated features:** if features $j$ and $k$ are correlated, permuting $j$ still leaves much of its signal in $k$. Both will have lower PI than their true joint contribution. This is a fundamental limitation, not a bug — it reflects the model's actual redundancy.

**Interpretation of negative PI:** if permuting a feature _improves_ performance, the model was relying on that feature in a harmful way (possibly overfitting). This is a debugging signal.

### 2.3 Partial Dependence Plots (PDP)

A PDP shows the marginal effect of one (or two) features on the predicted outcome, averaged over all other features.

**Formal definition.** For a single feature $x_j$:

$$\hat{f}_j(x_j) = \mathbb{E}_{X_{\setminus j}}!\left[\hat{f}(x_j, X_{\setminus j})\right] \approx \frac{1}{n} \sum_{i=1}^n \hat{f}(x_j, x_i^{(\setminus j)})$$

where $x_i^{(\setminus j)}$ denotes the $i$-th training observation with feature $j$ replaced by the target value $x_j$.

Algorithmically: for each candidate value $v$ of feature $j$, set the entire column $j$ to $v$, get model predictions for all $n$ training observations, and average them. The result is the model's expected prediction as a function of $x_j$, marginalising over the empirical distribution of all other features.

```python
from sklearn.inspection import PartialDependenceDisplay

fig, ax = plt.subplots(figsize=(12, 4))
PartialDependenceDisplay.from_estimator(
    model, X_train,
    features=['lead_time', 'adr', ('lead_time', 'adr')],  # last: 2D interaction
    feature_names=feature_names,
    kind='average',   # 'individual' gives ICE curves
    ax=ax
)
```

**ICE curves (Individual Conditional Expectation):** instead of averaging over all observations, plot the curve $x_j \mapsto \hat{f}(x_j, x_i^{(\setminus j)})$ for each individual observation $i$. ICE curves reveal heterogeneity in the feature effect that the PDP average hides.

```python
PartialDependenceDisplay.from_estimator(
    model, X_train,
    features=['lead_time'],
    kind='both',     # plot both ICE (grey) and PDP (bold)
)
```

**Limitations of PDPs:**

- **Assumes feature independence.** The marginalisation is over the empirical (possibly correlated) joint distribution. For highly correlated features, you may be averaging over unrealistic $(x_j, x_k)$ combinations — extrapolating outside the data manifold.
- **Averages hide interactions.** If the effect of $x_j$ on $\hat{y}$ is positive for one subgroup and negative for another, the PDP average may be flat — showing "no effect" when the reality is opposing effects. ICE curves fix this.
- **Two-feature PDPs** require a grid of $m^2$ predictions and become cognitively hard to read above two features.

## ✦ Part III — SHAP (SHapley Additive exPlanations)

SHAP is the most principled post-hoc interpretability method available. It is grounded in cooperative game theory and provides a unified, theoretically justified attribution of each feature's contribution to a specific prediction.

### 3.1 Shapley Values — The Game Theory Foundation

Consider a cooperative game with $d$ players (features) and a payout (model prediction). The **Shapley value** of player $j$ is the average marginal contribution of player $j$ across all possible orderings (coalitions) of the other players.

Formally, let $\mathcal{S} \subseteq {1, \ldots, d} \setminus {j}$ be a coalition not containing $j$, and let $v(\mathcal{S})$ denote the model's prediction using only the features in $\mathcal{S}$ (marginalising over the others). The Shapley value of feature $j$ for observation $x$ is:

$$\phi_j(x) = \sum_{\mathcal{S} \subseteq {1,\ldots,d} \setminus {j}} \frac{|\mathcal{S}|!,(d - |\mathcal{S}| - 1)!}{d!} \left[v(\mathcal{S} \cup {j}) - v(\mathcal{S})\right]$$

The coefficient $\frac{|\mathcal{S}|!,(d - |\mathcal{S}| - 1)!}{d!}$ weights each coalition by the probability that this particular ordering arises if all $d!$ orderings are equally likely — ensuring symmetry and fair attribution.

**The four Shapley axioms** (which uniquely characterise Shapley values):

1. **Efficiency:** $\sum_{j=1}^d \phi_j(x) = \hat{f}(x) - \mathbb{E}[\hat{f}]$. The attributions sum to the model's deviation from its expected output. The prediction is fully explained.
    
2. **Symmetry:** if two features contribute equally to every coalition, they receive equal Shapley values.
    
3. **Dummy:** if a feature contributes nothing to any coalition, its Shapley value is zero.
    
4. **Linearity/Additivity:** the Shapley value of a sum of games equals the sum of Shapley values.
    

These axioms, together, uniquely determine the Shapley value. **No other attribution method satisfies all four simultaneously.** This is why SHAP is theoretically privileged over alternatives like LIME, gradient-based attribution, or attention weights.

### 3.2 From Shapley Values to SHAP

Naively computing Shapley values requires $2^d$ model evaluations (one per coalition subset) — exponential in $d$. Lundberg & Lee (2017) introduced SHAP as a practical framework with exact or approximate algorithms depending on model type.

**The SHAP explanation model** is an additive function:

$$g(z') = \phi_0 + \sum_{j=1}^d \phi_j z'_j$$

where $z' \in {0,1}^d$ is a binary vector indicating whether each feature is "included" (1) or "marginalised out" (0), and $\phi_0 = \mathbb{E}[\hat{f}(X)]$ is the base value.

For a specific observation $x$, setting $z' = \mathbf{1}$ (all features present) gives:

$$\hat{f}(x) = \phi_0 + \sum_{j=1}^d \phi_j(x)$$

Every prediction decomposes exactly into a sum of feature contributions plus a base rate. This is the efficiency axiom made concrete.

### 3.3 TreeSHAP — Exact and Efficient

For tree-based models (decision trees, random forests, XGBoost, LightGBM, CatBoost), Lundberg et al. (2018) derived an **exact** algorithm that runs in $O(TLD^2)$ time — polynomial, not exponential — where $T$ is the number of trees, $L$ is the number of leaves, and $D$ is the maximum depth.

The key insight: tree structures allow efficient computation of $v(\mathcal{S})$ for all subsets simultaneously by propagating each observation through the tree and correctly averaging over the marginalised features.

```python
import shap

# Tree models: exact TreeSHAP
explainer = shap.TreeExplainer(xgb_model)
shap_values = explainer(X_val)   # shape: (n_samples, n_features)

# shap_values[i, j] = φ_j(x_i): contribution of feature j to prediction for observation i
# shap_values.base_values[i] = E[f(X)]: the base rate
# shap_values.base_values[i] + shap_values.values[i].sum() = f(x_i)  ← efficiency axiom
```

**Verify the efficiency axiom:**

```python
import numpy as np

predictions = xgb_model.predict(X_val, output_margin=True)   # raw log-odds
reconstructed = shap_values.base_values + shap_values.values.sum(axis=1)
assert np.allclose(predictions, reconstructed, atol=1e-5)    # should pass
```

### 3.4 SHAP Visualisations

**Waterfall plot — single prediction explained:**

```python
shap.plots.waterfall(shap_values[0])
```

Shows, for observation 0: the base value $\mathbb{E}[\hat{f}]$, each feature's contribution (red = increases prediction, blue = decreases), and the final prediction. The most important contributors are shown largest. This is the natural output for explaining a single decision to a stakeholder.

**Beeswarm plot — global feature importance with direction:**

```python
shap.plots.beeswarm(shap_values)
```

Each row is a feature. Each dot is one observation. Position on x-axis = SHAP value (contribution to prediction). Colour = feature value (red = high, blue = low). This simultaneously shows:

- Which features matter most globally (vertical ordering by mean $|\phi_j|$)
- In which direction each feature pushes predictions
- Heterogeneity in feature effects (wide spread = interaction effects or non-linearity)

**Bar plot — mean absolute SHAP (global importance):**

```python
shap.plots.bar(shap_values)
# Equivalent to: mean |φ_j(x)| over all observations
```

This is the SHAP analogue of feature importance. It is strictly better than MDI: computed on validation data, not biased by cardinality, and accounts for correlated features more faithfully.

**Dependence plot — marginal effect + interaction:**

```python
shap.plots.scatter(shap_values[:, "lead_time"], color=shap_values[:, "deposit_type"])
```

x-axis: feature value of `lead_time`. y-axis: SHAP value $\phi_{\text{lead_time}}(x_i)$. Colour: value of a second feature. If colouring reveals a pattern, those two features interact in the model.

### 3.5 SHAP for Non-Tree Models

For models without tree structure, SHAP provides approximate methods:

**KernelSHAP (model-agnostic):** estimates Shapley values by solving a weighted linear regression over randomly sampled coalitions. Correct in expectation but slow — $O(2^d)$ evaluations for exact, $O(n_{\text{samples}} \cdot d)$ for the approximation.

```python
# For any model with predict_proba
background = shap.kmeans(X_train, 100)   # summarise training distribution
explainer = shap.KernelExplainer(model.predict_proba, background)
shap_values = explainer.shap_values(X_val[:50], nsamples=500)
```

**DeepSHAP:** for neural networks, uses DeepLIFT backpropagation rules combined with SHAP to efficiently approximate Shapley values through the network.

```python
explainer = shap.DeepExplainer(torch_model, background_tensor)
shap_values = explainer.shap_values(X_val_tensor)
```

**GradientSHAP:** combines integrated gradients with SHAP. Faster than KernelSHAP for differentiable models.

### 3.6 What SHAP Does Not Tell You

SHAP explains what the model learned. It does not tell you:

- Whether the model learned the right thing (that requires domain knowledge and error analysis)
- Causal relationships — SHAP values are attributions within a predictive model, not causal effects
- What to change to flip the prediction — for that, use **counterfactual explanations** (find the nearest $x'$ such that $\hat{f}(x') \neq \hat{f}(x)$)

Correlation between features makes SHAP values harder to interpret causally. If `lead_time` and `days_until_arrival` are nearly identical, their SHAP values will be split between them — neither individually reflects the "true" causal effect of advance booking on cancellation.

## ✦ Part IV — LIME (Local Interpretable Model-Agnostic Explanations)

### 4.1 The Idea

LIME explains a single prediction $\hat{f}(x)$ by fitting a simple, interpretable model (linear regression) to the _local neighbourhood_ of $x$.

**Algorithm:**

1. Sample $n'$ perturbed versions of $x$ by randomly zeroing out features: $x'_1, \ldots, x'_{n'}$
2. Get model predictions: $\hat{f}(x'_1), \ldots, \hat{f}(x'_{n'})$
3. Weight each sample by proximity to $x$: $\pi_{x}(x'_i) = \exp!\left(-\frac{d(x, x'_i)^2}{2\sigma^2}\right)$
4. Fit a weighted Lasso regression: $g = \arg\min_{g \in \mathcal{G}} \sum_i \pi_x(x'_i) \cdot (\hat{f}(x'_i) - g(x'_i))^2 + \Omega(g)$
5. Return the coefficients of $g$ as the local explanation

```python
import lime.lime_tabular

explainer = lime.lime_tabular.LimeTabularExplainer(
    X_train.values,
    feature_names=feature_names,
    class_names=['kept', 'cancelled'],
    mode='classification'
)

explanation = explainer.explain_instance(
    X_val.values[0],
    model.predict_proba,
    num_features=10
)
explanation.show_in_notebook()
```

### 4.2 LIME vs. SHAP — A Rigorous Comparison

|Property|LIME|SHAP|
|---|---|---|
|**Theoretical grounding**|Heuristic local approximation|Unique solution satisfying 4 game-theory axioms|
|**Efficiency axiom**|✗ Not guaranteed|✓ Guaranteed: $\phi_0 + \sum_j \phi_j = \hat{f}(x)$|
|**Consistency**|✗ Two runs can give different results (random sampling)|✓ Deterministic (TreeSHAP); approximate for others|
|**Speed**|Fast for tabular|TreeSHAP: fast; KernelSHAP: slow|
|**Global summaries**|✗ Local only by design|✓ Aggregate to global naturally|
|**Correlated features**|Problematic (perturbed samples may be unrealistic)|Problematic (but principled marginalisation)|

**The fundamental limitation of LIME:** the neighbourhood size $\sigma$ is a hyperparameter with no principled default. Different values of $\sigma$ can give qualitatively different explanations for the same prediction. SHAP's Shapley values, by contrast, are uniquely defined — there is only one correct answer given the model and the data distribution assumption.

**When LIME is still useful:** LIME is fast and model-agnostic. For quick local explanations on a complex model (e.g., a deep neural network) where TreeSHAP is not applicable and KernelSHAP is too slow, LIME is a practical first pass. Use it as exploration, not as a reportable result.

## ✦ Part V — The Accuracy-Interpretability Trade-off, Revisited

### 5.1 The Rashomon Set

Leo Breiman (2001) introduced the **Rashomon effect**: for a given dataset, there typically exist many models with similar predictive performance but very different structures. The set of near-optimal models is called the **Rashomon set**.

Formally, define $\mathcal{R}_\epsilon = {f \in \mathcal{F} : R(f) \leq R^* + \epsilon}$ where $R^* = \min_{f \in \mathcal{F}} R(f)$ is the best achievable risk. For small $\epsilon$, $\mathcal{R}_\epsilon$ typically contains models of varying complexity.

**Implication for practice:** before concluding that an accurate model must be a black box, verify that no simpler model in $\mathcal{R}_\epsilon$ achieves similarly low risk. If a linear model achieves AUC 0.87 and XGBoost achieves 0.89, the interpretable model is within $\epsilon = 0.02$ — whether that epsilon is worth the interpretability cost is a domain-specific decision, not a universal truth.

Rudin (2019), "Stop Explaining Black Box Machine Learning Models for High Stakes Decisions and Use Interpretable Models Instead," argues that in high-stakes domains (criminal justice, medical diagnosis), the Rashomon set is almost always non-empty at an acceptable $\epsilon$, and the interpretable model should always be preferred.

### 5.2 When You Must Use a Black Box

Neural networks dominate for unstructured data (images, text, audio) because:

- The inductive bias of convolution / attention is essential for generalisation
- No tabular-style feature engineering is feasible at the scale required
- The Rashomon set argument breaks down: simpler models achieve substantially worse performance

In these cases, post-hoc methods (SHAP, GradCAM for images, attention visualisation for transformers) are the only option. Accept that they are approximate and use them for debugging and plausibility checking, not for rigorous causal attribution.

## ✦ Part VI — Relation to AI / Deep Learning

### 6.1 Attention Is Not Explanation

In transformer models (BERT, GPT), attention weights are often cited as explanations: "the model attended to these tokens, therefore they caused the prediction." This is largely false.

Jain & Wallace (2019) showed that attention weights are **not faithful** to feature importance: you can change the attention distribution substantially without changing the prediction, and attention does not correlate with gradient-based importance measures. Attention is a routing mechanism — it determines which representations are combined, not which inputs cause the output.

For faithful explanation of transformer predictions, use:

- **Integrated Gradients:** attributes prediction to input features by integrating gradients along a path from a baseline to the input
- **SHAP DeepExplainer / GradientExplainer**
- **LIME on token embeddings**

### 6.2 GradCAM for Convolutional Networks

For image classifiers, **GradCAM** (Gradient-weighted Class Activation Mapping) produces a heatmap localising which regions of the image drove the prediction.

$$L_c^{\text{GradCAM}} = \text{ReLU}!\left(\sum_k \alpha_k^c A^k\right), \quad \alpha_k^c = \frac{1}{Z}\sum_i\sum_j \frac{\partial y^c}{\partial A^k_{ij}}$$

where $A^k$ is the activation map of the $k$-th convolutional feature, and $\alpha_k^c$ is the global-average-pooled gradient of class $c$'s score with respect to that feature map.

```python
from pytorch_grad_cam import GradCAM
from pytorch_grad_cam.utils.image import show_cam_on_image

cam = GradCAM(model=resnet, target_layers=[resnet.layer4[-1]])
grayscale_cam = cam(input_tensor=img_tensor)
visualisation = show_cam_on_image(img_np, grayscale_cam[0])
```

GradCAM is faithful (it uses actual gradients, not heuristic attribution) but only localises to the spatial resolution of the last convolutional layer — coarse for deep networks.

### 6.3 Interpretability in the Training Loop

For deep learning, interpretability is increasingly integrated into training, not just applied post-hoc:

- **Concept Bottleneck Models (CBMs):** the network must route its predictions through human-interpretable concept activations. Koh et al. (2020).
- **Neural Additive Models (NAMs):** enforce additivity, producing one learned function per feature.
- **Sparse autoencoders** (used in mechanistic interpretability of LLMs): find interpretable directions in activation space by enforcing sparsity. Active research in 2024–25.

**Mechanistic interpretability** is an emerging research area aiming to reverse-engineer the algorithms learned by neural networks — not just which inputs matter, but _what computation_ the model is performing. Relevant to understanding LLMs (see Anthropic's work on superposition, circuits, etc.).

### 6.4 The SHAP + Deep Learning Stack

For a neural network on tabular data, the recommended interpretability stack is:

```python
import shap

# GradientExplainer: uses expected gradients (faster than KernelSHAP, exact for linear models)
background = torch.tensor(X_train[:200].values, dtype=torch.float32)
explainer = shap.GradientExplainer(torch_model, background)
shap_values = explainer.shap_values(torch.tensor(X_val.values, dtype=torch.float32))

# Same visualisations as TreeSHAP
shap.summary_plot(shap_values, X_val, feature_names=feature_names)
```

## ✦ Worked Example — Hotel Cancellation

Continuing the hotel cancellation task from [[01.ML Project Workflow]], applying the full interpretability stack to a fitted XGBoost model.

```python
import shap
import xgboost as xgb
import matplotlib.pyplot as plt

# 1. Fit model (already done in workflow note)
model = xgb.XGBClassifier(n_estimators=600, max_depth=6, learning_rate=0.05)
model.fit(X_train, y_train)

# 2. SHAP explainer
explainer = shap.TreeExplainer(model)
shap_values = explainer(X_val)

# 3. Verify efficiency axiom
raw_preds = model.predict(X_val, output_margin=True)
assert np.allclose(shap_values.base_values + shap_values.values.sum(1), raw_preds, atol=1e-4)

# 4. Global importance — beeswarm
shap.plots.beeswarm(shap_values, max_display=15)
# → deposit_type=Non_Refund is the strongest predictor (high feature value → high SHAP)
# → lead_time has a monotone positive effect (longer lead time → more likely to cancel)
# → total_of_special_requests has a negative effect (requests → more committed guest)

# 5. Explain a single prediction — waterfall
shap.plots.waterfall(shap_values[42])
# Observation 42: predicted prob = 0.83
# Base rate: 0.37
# +0.31: deposit_type = Non_Refund
# +0.09: lead_time = 127 days
# +0.04: country = PRT (domestic guests cancel more)
# -0.02: total_of_special_requests = 2

# 6. Dependence plot — lead_time with interaction
shap.plots.scatter(shap_values[:, "lead_time"], color=shap_values[:, "deposit_type"])
# Reveals: lead_time effect is stronger for Non_Refund deposits (interaction)

# 7. Compare to permutation importance
from sklearn.inspection import permutation_importance
pi = permutation_importance(model, X_val, y_val, n_repeats=10, scoring='roc_auc')
# If top features by PI ≠ top features by SHAP: investigate — 
# possible collinearity, or model uses feature in unexpected way
```
