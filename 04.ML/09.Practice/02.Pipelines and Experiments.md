## ✦ Why This Note Exists

You can derive every algorithm in this vault from scratch. That is necessary. It is not sufficient.

The practitioner who fits a `StandardScaler` on the full dataset before cross-validation — a leakage error so common it has its own name — does not have a model problem. They have a pipeline problem. The researcher who runs forty experiments, cannot remember which hyperparameter configuration produced the best result, and cannot reproduce it — does not have an algorithm problem. They have a tracking problem.

This note is about the engineering substrate that makes ML work _correctly_ and _reproducibly_. The mathematics here is modest; the discipline required is not.

## ✦ Part I — The sklearn Pipeline

### 1.1 The Problem a Pipeline Solves

Consider the following apparently reasonable code:

```python
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import cross_val_score

scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)          # ← WRONG

scores = cross_val_score(LogisticRegression(), X_scaled, y, cv=5)
```

**What went wrong?** `fit_transform` was called on the entire dataset $X$ before cross-validation. The scaler computed $\mu$ and $\sigma$ using _all_ $n$ observations — including the observations in each validation fold. When CV later withholds fold $k$ as the validation set, that fold's statistics have already been "seen" by the scaler.

Formally: let $X_{\text{val}}^{(k)}$ be the $k$-th validation fold. The correct scaler should be:

$$\hat{\mu}_k = \frac{1}{|X_{\text{train}}^{(k)}|} \sum_{x \in X_{\text{train}}^{(k)}} x$$

But the contaminated scaler uses:

$$\hat{\mu}_{\text{all}} = \frac{1}{n} \sum_{x \in X} x$$

which includes $X_{\text{val}}^{(k)}$ in the mean. This is **data leakage through preprocessing.** For `StandardScaler` the bias is small. For **target encoding, PCA, feature selection, or imputation**, the contamination can be enormous and the CV score misleadingly optimistic.

**The fix:** wrap all preprocessing and the model inside a `Pipeline`. sklearn guarantees that within each CV fold, `fit` is called only on the training portion.

```python
from sklearn.pipeline import Pipeline

pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', LogisticRegression())
])

scores = cross_val_score(pipe, X, y, cv=5)   # ← CORRECT
```

Now, for each fold:

1. `pipe.fit(X_train_fold, y_train_fold)` → scaler fits on $X_{\text{train}}^{(k)}$, then transforms it; model trains on scaled data
2. `pipe.predict(X_val_fold)` → scaler _transforms_ (does not refit) $X_{\text{val}}^{(k)}$ using $\hat{\mu}_k$, then model predicts

The preprocessing and the model are a single estimator object. This is not cosmetic — it is a correctness guarantee.

### 1.2 The Formal Pipeline Contract

A sklearn `Pipeline` is a sequence of `(name, transformer)` pairs followed by a final `(name, estimator)`. Every intermediate step must implement `.fit()` and `.transform()` (i.e., be a `Transformer`). The final step must implement `.fit()` and `.predict()` (i.e., be an `Estimator`).

**The call chain:**

```
pipe.fit(X, y):
    for each step (name, transformer) in steps[:-1]:
        X = transformer.fit_transform(X, y)
    final_estimator.fit(X, y)

pipe.predict(X):
    for each step (name, transformer) in steps[:-1]:
        X = transformer.transform(X)   # NOT fit_transform
    return final_estimator.predict(X)
```

Note the asymmetry: `.fit_transform()` during training, `.transform()` only during inference. This is the invariant that prevents leakage.

**Named steps allow parameter access:**

```python
pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', LogisticRegression(C=1.0))
])

# Access individual steps
pipe.named_steps['scaler']
pipe.named_steps['clf'].coef_

# Set parameters using __ separator (for GridSearch)
pipe.set_params(clf__C=0.1)
```

The double-underscore `__` separator is the sklearn convention for nested parameter access. This is what makes `GridSearchCV` over a pipeline possible.

### 1.3 ColumnTransformer — Heterogeneous Feature Types

Real tabular data has mixed types: numeric features need scaling; categorical features need encoding; text features need vectorisation. `ColumnTransformer` applies different transformers to different column subsets and concatenates the results.

```python
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.pipeline import Pipeline

numeric_features = ['age', 'income', 'lead_time']
categorical_features = ['country', 'deposit_type', 'room_type']

numeric_transformer = Pipeline([
    ('imputer', SimpleImputer(strategy='median')),
    ('scaler', StandardScaler())
])

categorical_transformer = Pipeline([
    ('imputer', SimpleImputer(strategy='most_frequent')),
    ('encoder', OneHotEncoder(handle_unknown='ignore', sparse_output=False))
])

preprocessor = ColumnTransformer(transformers=[
    ('num', numeric_transformer, numeric_features),
    ('cat', categorical_transformer, categorical_features)
])

full_pipeline = Pipeline([
    ('preprocessor', preprocessor),
    ('classifier', LogisticRegression(max_iter=1000))
])
```

**What `ColumnTransformer` does internally:**

1. For each `(name, transformer, columns)` tuple: apply `transformer.fit_transform(X[:, columns])` during training
2. Concatenate the transformed outputs horizontally: $[\tilde{X}_{\text{num}} \mid \tilde{X}_{\text{cat}}]$
3. During inference: apply `.transform()` only

The output ordering is deterministic: transformers are applied in the order specified, then any passthrough columns, then remainder columns. Keep this in mind when interpreting feature importances — column indices shift.

**Getting feature names after transformation:**

```python
# After fitting:
full_pipeline.named_steps['preprocessor'].get_feature_names_out()
# → ['num__age', 'num__income', 'cat__country_AU', 'cat__country_DE', ...]
```

### 1.4 Custom Transformers

When built-in transformers are insufficient, implement the `TransformerMixin` interface:

```python
from sklearn.base import BaseEstimator, TransformerMixin
import numpy as np

class LogTransformer(BaseEstimator, TransformerMixin):
    """Apply log1p to specified columns. Safe for zero values."""
    
    def __init__(self, columns=None):
        self.columns = columns
    
    def fit(self, X, y=None):
        # Nothing to fit — stateless transformation
        return self
    
    def transform(self, X, y=None):
        X = X.copy()
        cols = self.columns or X.columns.tolist()
        X[cols] = np.log1p(X[cols])
        return X
```

**The three rules for a correct custom transformer:**

1. `fit()` learns statistics from training data only (or does nothing for stateless transforms). Always `return self`.
2. `transform()` applies the learned transformation. Never modifies `self` state.
3. `BaseEstimator` provides `get_params()` / `set_params()` for free — required for `GridSearchCV` to work.

### 1.5 FunctionTransformer — Lightweight Stateless Transforms

For stateless transformations (no fitting required), `FunctionTransformer` avoids the boilerplate:

```python
from sklearn.preprocessing import FunctionTransformer

log_transformer = FunctionTransformer(np.log1p, validate=True)

# Equivalent to the class above, for single-function transforms
```

### 1.6 Pipelines for Cross-Validation — The Full Correct Pattern

```python
from sklearn.model_selection import StratifiedKFold, cross_validate
from sklearn.metrics import roc_auc_score

cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

results = cross_validate(
    full_pipeline,
    X_train, y_train,
    cv=cv,
    scoring=['roc_auc', 'average_precision'],
    return_train_score=True,   # detect overfitting: if train >> val, overfit
    n_jobs=-1
)

print(f"Val AUC:  {results['test_roc_auc'].mean():.4f} ± {results['test_roc_auc'].std():.4f}")
print(f"Train AUC: {results['train_roc_auc'].mean():.4f} ± {results['train_roc_auc'].std():.4f}")
```

Returning both train and validation scores is essential: if $\text{train score} \gg \text{val score}$, you are overfitting. If both are low, you are underfitting.

### 1.7 GridSearchCV over a Pipeline

```python
from sklearn.model_selection import GridSearchCV

param_grid = {
    'preprocessor__num__imputer__strategy': ['mean', 'median'],
    'preprocessor__num__scaler__with_std': [True, False],
    'classifier__C': [0.01, 0.1, 1.0, 10.0],
    'classifier__penalty': ['l1', 'l2']
}

search = GridSearchCV(
    full_pipeline,
    param_grid,
    cv=StratifiedKFold(n_splits=5),
    scoring='roc_auc',
    n_jobs=-1,
    verbose=1
)

search.fit(X_train, y_train)

print(search.best_params_)
print(f"Best CV AUC: {search.best_score_:.4f}")
```

The `__` separator threads through nested pipeline and column transformer structures. `preprocessor__num__imputer__strategy` means: in the `preprocessor` step, in the `num` transformer, in the `imputer` step, set the `strategy` parameter.

**Why this is correct:** `GridSearchCV` wraps the entire pipeline as the estimator. For each parameter configuration, it runs CV where each fold calls `pipeline.fit()` on the training portion — which correctly refits all preprocessors on that fold's training data.

## ✦ Part II — Reproducibility

### 2.1 What Reproducibility Means in ML

A result is **reproducible** if, given the same code and data, you obtain the same numerical outputs. In ML, multiple sources of non-determinism can break this:

|Source|Fix|
|---|---|
|Random weight initialisation|`torch.manual_seed(42)` / `tf.random.set_seed(42)`|
|Data shuffling|`random_state=42` in `train_test_split`, `KFold`|
|Random forest / boosting|`random_state=42` in the estimator|
|GPU parallelism (CUDA)|`torch.backends.cudnn.deterministic = True` (costs speed)|
|Floating point ordering|Non-associativity of floating point: `sum([1e-8, 1.0]) ≠ sum([1.0, 1e-8])`|
|Python hash randomisation|`PYTHONHASHSEED=42` environment variable|
|Non-deterministic algorithms|Some sklearn solvers (`lbfgs`, `sag`) with `tol` convergence|

**The correct random seed pattern:**

```python
import random
import numpy as np
import torch

SEED = 42

random.seed(SEED)
np.random.seed(SEED)
torch.manual_seed(SEED)
torch.cuda.manual_seed_all(SEED)
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False
```

**A warning about `random_state` semantics:** setting `random_state=42` in sklearn gives reproducibility within a single Python process. If you run the same script twice, you get the same result. But if you change _anything else_ in the script that calls a random number generator before the seeded code, the state of the RNG shifts and results may differ. Use isolated, purpose-specific RNG instances:

```python
# Better: isolated RNG per operation
rng = np.random.default_rng(seed=42)
shuffled = rng.permutation(len(X))
```

### 2.2 Environment Pinning

Reproducibility requires not just the same code, but the same environment. Library versions matter: sklearn 1.2 and 1.3 may produce different results for the same code due to algorithm updates, default parameter changes, or numerical fixes.

**Minimum viable environment pinning:**

```bash
# Capture exact versions
pip freeze > requirements.txt

# Reproduce
pip install -r requirements.txt
```

**Better: `pyproject.toml` with version bounds + lockfile:**

```toml
[project]
dependencies = [
    "scikit-learn>=1.3,<2.0",
    "numpy>=1.24,<2.0",
    "pandas>=2.0,<3.0",
    "xgboost>=2.0,<3.0",
    "lightgbm>=4.0,<5.0",
]
```

**Best: containerisation.** A Docker image pins the entire stack — OS, Python version, CUDA version, all libraries:

```dockerfile
FROM python:3.11-slim
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . /app
WORKDIR /app
```

For GPU work: use official PyTorch or TensorFlow Docker images as base — they pin CUDA/cuDNN versions.

## ✦ Part III — Experiment Tracking

### 3.1 The Problem

Consider a realistic scenario: you run 50 experiments over two weeks, varying:

- Three model architectures
- Five learning rate schedules
- Four regularisation strengths
- Two feature sets

You achieve your best validation AUC on experiment 37. Three weeks later, you need to:

- Reproduce the exact model for a paper submission
- Explain to a colleague which configuration worked and why
- Continue from that checkpoint for further training
- Compare experiment 37 to experiment 51

Without experiment tracking infrastructure, all of this is difficult or impossible.

### 3.2 What to Log

For every experiment, log:

**Inputs:**

- Hyperparameters (all of them — not just the ones you varied)
- Dataset version / hash (a SHA256 of the data file is sufficient for small datasets)
- Code version (Git commit hash)
- Environment (requirements.txt / conda env)
- Random seeds

**Outputs:**

- Metrics at each evaluation step (training loss, validation loss, per-epoch)
- Final metrics (test set, if used)
- Model artefacts (serialised model weights)
- Plots (learning curves, confusion matrix, calibration curve)
- Wall clock time, hardware used

**The minimum viable log (manual):**

```python
import json, hashlib, subprocess, datetime

def get_git_hash():
    return subprocess.check_output(['git', 'rev-parse', 'HEAD']).decode().strip()

def hash_file(path):
    sha = hashlib.sha256()
    with open(path, 'rb') as f:
        sha.update(f.read())
    return sha.hexdigest()[:12]

experiment = {
    'timestamp': datetime.datetime.now().isoformat(),
    'git_hash': get_git_hash(),
    'data_hash': hash_file('data/train.csv'),
    'params': {'C': 1.0, 'penalty': 'l2', 'solver': 'lbfgs'},
    'metrics': {'val_auc': 0.891, 'val_f1': 0.743},
    'notes': 'Added log transform to lead_time'
}

with open(f"experiments/exp_{datetime.datetime.now().strftime('%Y%m%d_%H%M%S')}.json", 'w') as f:
    json.dump(experiment, f, indent=2)
```

This is the baseline. It degrades quickly as experiments multiply.

### 3.3 MLflow — The Standard Tool

**MLflow** is the de facto open-source experiment tracking library. It provides:

- **Tracking:** log parameters, metrics, artefacts per run
- **Registry:** versioned model storage with lifecycle (staging / production / archived)
- **Projects:** packaging ML code for reproducibility
- **UI:** a browser-based dashboard to compare runs

**Basic usage:**

```python
import mlflow
import mlflow.sklearn

mlflow.set_experiment("hotel-cancellation-v2")

with mlflow.start_run(run_name="xgb-baseline") as run:
    # Log hyperparameters
    mlflow.log_params({
        'n_estimators': 600,
        'max_depth': 6,
        'learning_rate': 0.05,
        'subsample': 0.8,
        'colsample_bytree': 0.8
    })
    
    # Train model
    model = xgb.XGBClassifier(n_estimators=600, max_depth=6, learning_rate=0.05)
    model.fit(X_train, y_train,
              eval_set=[(X_val, y_val)],
              early_stopping_rounds=50,
              verbose=False)
    
    # Log metrics
    val_auc = roc_auc_score(y_val, model.predict_proba(X_val)[:, 1])
    mlflow.log_metric('val_auc', val_auc)
    
    # Log per-epoch training curve
    for epoch, loss in enumerate(model.evals_result()['validation_0']['logloss']):
        mlflow.log_metric('val_logloss', loss, step=epoch)
    
    # Log the model artefact
    mlflow.sklearn.log_model(model, "model")
    
    # Log a plot
    fig, ax = plt.subplots()
    plot_roc_curve(model, X_val, y_val, ax=ax)
    mlflow.log_figure(fig, "roc_curve.png")
    
    print(f"Run ID: {run.info.run_id}")
    print(f"Val AUC: {val_auc:.4f}")
```

**Launch the UI:**

```bash
mlflow ui --port 5000
# → http://localhost:5000
```

The UI shows all runs in the experiment, lets you sort/filter by any logged metric, and renders learning curve plots side by side across runs.

### 3.5 The Experiment Naming and Tagging Convention

Even with tools, discipline in naming prevents chaos:

```
project/
├── experiments/
│   ├── 001_baseline_logistic/
│   ├── 002_random_forest_default/
│   ├── 003_xgb_no_feature_eng/
│   ├── 004_xgb_log_transform_lead_time/   ← what changed, not a timestamp
│   └── 005_xgb_interaction_features/
```

Every experiment gets:

- A sequential ID
- A descriptive slug summarising what changed from the previous best
- A notes field explaining the hypothesis being tested

**The experiment as a hypothesis test:** each run should be motivated by a specific hypothesis:

> _"Adding the ratio feature `days_since_modification / lead_time` will help because error analysis showed the model struggles with last-minute modifications."_

Experiments without hypotheses are just noise.

## ✦ Part IV — The Full Reproducible Experiment Template

Putting it all together: the structure of a fully reproducible, trackable ML experiment.

```python
# experiment.py
import argparse
import json
import logging
import random
import subprocess
from pathlib import Path

import mlflow
import numpy as np
import pandas as pd
from sklearn.compose import ColumnTransformer
from sklearn.impute import SimpleImputer
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import roc_auc_score
from sklearn.model_selection import StratifiedKFold, cross_validate
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import OneHotEncoder, StandardScaler


# ── 0. Config ─────────────────────────────────────────────────────────────────

def parse_args():
    p = argparse.ArgumentParser()
    p.add_argument('--seed',        type=int,   default=42)
    p.add_argument('--n-folds',     type=int,   default=5)
    p.add_argument('--C',           type=float, default=1.0)
    p.add_argument('--penalty',     type=str,   default='l2')
    p.add_argument('--data-path',   type=str,   default='data/train.csv')
    p.add_argument('--experiment',  type=str,   default='default')
    p.add_argument('--run-name',    type=str,   default=None)
    return p.parse_args()


# ── 1. Reproducibility ────────────────────────────────────────────────────────

def set_seeds(seed: int):
    random.seed(seed)
    np.random.seed(seed)


def get_git_hash() -> str:
    try:
        return subprocess.check_output(
            ['git', 'rev-parse', '--short', 'HEAD']
        ).decode().strip()
    except Exception:
        return 'unknown'


# ── 2. Data ───────────────────────────────────────────────────────────────────

def load_data(path: str):
    df = pd.read_csv(path)
    X = df.drop('is_cancelled', axis=1)
    y = df['is_cancelled']
    return X, y


# ── 3. Pipeline ───────────────────────────────────────────────────────────────

def build_pipeline(C: float, penalty: str) -> Pipeline:
    num_features = ['lead_time', 'adr', 'total_of_special_requests']
    cat_features = ['country', 'deposit_type', 'room_type']

    num_pipe = Pipeline([
        ('imputer', SimpleImputer(strategy='median')),
        ('scaler',  StandardScaler())
    ])
    cat_pipe = Pipeline([
        ('imputer', SimpleImputer(strategy='most_frequent')),
        ('encoder', OneHotEncoder(handle_unknown='ignore', sparse_output=False))
    ])
    preprocessor = ColumnTransformer([
        ('num', num_pipe, num_features),
        ('cat', cat_pipe, cat_features)
    ])
    return Pipeline([
        ('preprocessor', preprocessor),
        ('clf', LogisticRegression(C=C, penalty=penalty, max_iter=1000, solver='saga'))
    ])


# ── 4. Evaluate ───────────────────────────────────────────────────────────────

def evaluate(pipeline: Pipeline, X, y, n_folds: int, seed: int) -> dict:
    cv = StratifiedKFold(n_splits=n_folds, shuffle=True, random_state=seed)
    results = cross_validate(
        pipeline, X, y,
        cv=cv,
        scoring=['roc_auc', 'average_precision'],
        return_train_score=True,
        n_jobs=-1
    )
    return {
        'val_auc_mean':    float(results['test_roc_auc'].mean()),
        'val_auc_std':     float(results['test_roc_auc'].std()),
        'train_auc_mean':  float(results['train_roc_auc'].mean()),
        'val_ap_mean':     float(results['test_average_precision'].mean()),
    }


# ── 5. Main ───────────────────────────────────────────────────────────────────

def main():
    args = parse_args()
    set_seeds(args.seed)
    logging.basicConfig(level=logging.INFO)
    log = logging.getLogger(__name__)

    X, y = load_data(args.data_path)
    pipeline = build_pipeline(C=args.C, penalty=args.penalty)

    mlflow.set_experiment(args.experiment)
    with mlflow.start_run(run_name=args.run_name or f"C={args.C}_pen={args.penalty}"):
        # Log everything
        mlflow.log_params({
            'C': args.C,
            'penalty': args.penalty,
            'n_folds': args.n_folds,
            'seed': args.seed,
            'git_hash': get_git_hash(),
            'data_path': args.data_path,
        })

        metrics = evaluate(pipeline, X, y, args.n_folds, args.seed)
        mlflow.log_metrics(metrics)
        log.info(json.dumps(metrics, indent=2))

        # Refit on full training set and log model
        pipeline.fit(X, y)
        mlflow.sklearn.log_model(pipeline, "pipeline")


if __name__ == '__main__':
    main()
```

**Usage:**

```bash
# Run an experiment
python experiment.py \
    --C 0.1 \
    --penalty l2 \
    --experiment hotel-cancellation-v2 \
    --run-name "logistic-l2-C0.1"

# Compare runs
mlflow ui
```

## ✦ Part V — The Relation to AI / ML / Deep Learning

### 5.1 Pipelines in Deep Learning

The `sklearn.Pipeline` abstraction is specific to sklearn, but the _principle_ — compose preprocessing and model into a single object with a clean fit/transform/predict contract — applies universally.

**PyTorch Lightning** formalises this for deep learning:

```python
import pytorch_lightning as pl

class HotelCancellationModel(pl.LightningModule):
    def __init__(self, input_dim, hidden_dim, lr):
        super().__init__()
        self.save_hyperparameters()   # ← auto-logs all __init__ args
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, 1),
            nn.Sigmoid()
        )
    
    def training_step(self, batch, batch_idx):
        x, y = batch
        y_hat = self.net(x).squeeze()
        loss = F.binary_cross_entropy(y_hat, y.float())
        self.log('train_loss', loss, on_epoch=True)  # ← auto-tracked
        return loss
    
    def validation_step(self, batch, batch_idx):
        x, y = batch
        y_hat = self.net(x).squeeze()
        loss = F.binary_cross_entropy(y_hat, y.float())
        self.log('val_loss', loss, on_epoch=True)
    
    def configure_optimizers(self):
        return torch.optim.Adam(self.parameters(), lr=self.hparams.lr)

# Trainer handles: logging, checkpointing, early stopping
trainer = pl.Trainer(
    max_epochs=100,
    callbacks=[
        pl.callbacks.EarlyStopping(monitor='val_loss', patience=10),
        pl.callbacks.ModelCheckpoint(monitor='val_loss', save_top_k=1)
    ],
    logger=pl.loggers.MLFlowLogger(experiment_name="hotel-dl")
)
trainer.fit(model, train_dataloader, val_dataloader)
```

`save_hyperparameters()` is Lightning's analogue of MLflow's `log_params` — it stores all constructor arguments and makes them available in checkpoints.

### 5.2 The Preprocessing Pipeline in Neural Networks

For neural networks, preprocessing is often simpler (normalise inputs, handle missing values) but the leakage concern is identical. The correct pattern in PyTorch:

```python
from sklearn.preprocessing import StandardScaler

# Fit on training data only
scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)   # fit here
X_val_scaled   = scaler.transform(X_val)          # transform only
X_test_scaled  = scaler.transform(X_test)         # transform only

# Save the scaler alongside the model — you need it at inference time
import joblib
joblib.dump(scaler, 'models/scaler.pkl')
torch.save(model.state_dict(), 'models/model.pt')
```

The scaler is a component of the model. If you lose it, you cannot make predictions correctly.

### 5.3 Experiment Tracking at Scale — Hyperparameter Sweeps

In deep learning, hyperparameter search is more expensive and tracking is more critical. W&B Sweeps provide Bayesian optimisation with automatic tracking:

## ✦ A Note on the Pipeline as Mathematical Object

The sklearn `Pipeline` is an implementation of **function composition**. Given transformers $T_1, T_2, \ldots, T_k$ and estimator $f$, the pipeline implements:

$$\text{Pipeline} = f \circ T_k \circ \cdots \circ T_2 \circ T_1$$

The prediction function is:

$$\hat{y} = f(T_k(\cdots T_2(T_1(x))\cdots))$$

During training, each $T_i$ has a fitting phase (learning statistics from training data) and a transform phase. The correctness guarantee of the pipeline is that the fitting phase of every $T_i$ uses only the training portion of each CV fold — never the validation portion. This is the formal condition for unbiased CV estimation of generalisation error.

Contrast with the contaminated estimator: when the scaler sees the validation fold during `fit_transform`, the effective sample the CV is evaluating on is not truly held-out. The bias introduced is:

$$\text{Bias} = \mathbb{E}[\hat{R}_{\text{CV, contaminated}}] - R(f) < \mathbb{E}[\hat{R}_{\text{CV, correct}}] - R(f)$$

The contaminated CV estimate is optimistic. The magnitude of the bias depends on how much the preprocessing statistic (mean, PCA direction, target encoding statistic) changes between the full-data estimate and the training-only estimate — which is $O(n_{\text{val}} / n)$ for means, potentially much larger for target encoding or feature selection.
