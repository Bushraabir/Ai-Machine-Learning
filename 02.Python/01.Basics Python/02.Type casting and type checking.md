
## Core Idea

Python is dynamically typed — variables have no fixed type. But objects do. Every object in Python carries its type with it at runtime. Type checking lets you inspect that type. Type casting lets you convert between types.

These two operations are different things:

- **Type checking** — asking "what type is this object?"
- **Type casting** — converting an object from one type to another

---

## Type Checking

### type()

Returns the exact type of an object as a type object.

```python
print(type(42))        # <class 'int'>
print(type(3.14))      # <class 'float'>
print(type("hello"))   # <class 'str'>
print(type(True))      # <class 'bool'>
print(type(None))      # <class 'NoneType'>
print(type([1, 2]))    # <class 'list'>
```

You can compare the result directly against a type:

```python
x = 42
print(type(x) == int)    # True
print(type(x) == float)  # False
```

**Critical limitation of type() — it does not respect inheritance:**

```python
print(type(True) == bool)   # True
print(type(True) == int)    # False  — even though bool IS a subclass of int
```

This is the primary reason `type()` is rarely used for conditional logic. It gives you exact type only — no hierarchy awareness.

---

### isinstance()

Checks whether an object is an instance of a type, or any subclass of that type.

```python
x = True

print(isinstance(x, bool))   # True
print(isinstance(x, int))    # True  — bool is a subclass of int
```

This is the preferred method for type checking in real code.

```python
x = 42

print(isinstance(x, int))    # True
print(isinstance(x, float))  # False
print(isinstance(x, str))    # False
```

**isinstance() with multiple types — pass a tuple:**

```python
x = 3.14

print(isinstance(x, (int, float)))   # True  — matches float
print(isinstance(x, (str, bool)))    # False
```

This is cleaner than writing `isinstance(x, int) or isinstance(x, float)`.

---

### type() vs isinstance() — When to Use Which

```python
# Scenario: you want to accept both int and float as "numeric"
x = 3.14

# type() — too strict, misses subclasses
if type(x) == int or type(x) == float:
    print("numeric")

# isinstance() — correct approach
if isinstance(x, (int, float)):
    print("numeric")
```

```python
# Scenario: you explicitly need to distinguish bool from int
# (since bool is a subclass of int, isinstance would return True for both)

x = True

isinstance(x, int)      # True  — may not be what you want
type(x) == int          # False — exact check, distinguishes bool from int
type(x) == bool         # True
```

**Rule of thumb:**

- Use `isinstance()` by default — it is flexible and inheritance-aware.
- Use `type()` only when you need an exact match and want to exclude subclasses.

## Python's Type Hierarchy (Relevant Subset)

```
object
├── int
│   └── bool        # True and False are ints (1 and 0)
├── float
├── complex
├── str
├── bytes
├── NoneType
├── list
├── tuple
├── dict
└── set
```

Because `bool` inherits from `int`, the following holds:

```python
isinstance(True, int)    # True
isinstance(True, bool)   # True
True + True              # 2  — valid arithmetic
True * 5                 # 5
```

This is not a quirk — it is intentional design. Know it, because it creates non-obvious behavior in conditions and arithmetic.

## Type Casting

Type casting is explicitly converting a value from one type to another using constructor functions. These are not special operators — they are regular functions that return a new object of the target type.

|Function|Converts to|
|---|---|
|`int()`|integer|
|`float()`|float|
|`str()`|string|
|`bool()`|boolean|
|`list()`|list|
|`tuple()`|tuple|
|`set()`|set|

---

### Casting to int

```python
int(3.99)      # 3    — truncates toward zero, does NOT round
int(3.01)      # 3
int(-3.99)     # -3   — truncates toward zero (not floor)
int("42")      # 42   — string must contain a valid integer
int(True)      # 1
int(False)     # 0
```

**What fails:**

```python
int("3.14")    # ValueError — string looks like a float, not an int
int("hello")   # ValueError
int(None)      # TypeError
int([1, 2])    # TypeError
```

**Correct way to convert a float-string to int:**

```python
int(float("3.14"))   # 3
```

**Truncation vs rounding — know the difference:**

```python
int(3.9)        # 3   — truncated
round(3.9)      # 4   — rounded
import math
math.floor(3.9) # 3   — floor (toward negative infinity)
math.ceil(3.1)  # 4   — ceiling
```

---

### Casting to float

```python
float(42)        # 42.0
float("3.14")    # 3.14
float("inf")     # inf   — valid
float("-inf")    # -inf
float(True)      # 1.0
float(False)     # 0.0
```

**What fails:**

```python
float("hello")   # ValueError
float(None)      # TypeError
```

---

### Casting to str

```python
str(42)          # "42"
str(3.14)        # "3.14"
str(True)        # "True"
str(False)       # "False"
str(None)        # "None"   — the string "None", not None itself
str([1, 2, 3])   # "[1, 2, 3]"
```

`str()` never raises an error on standard objects — every Python object has a string representation. This is because every class inherits `__str__` from `object`.

---

### Casting to bool

This is more important than it appears. Python evaluates truthiness in every `if` statement, `while` loop, and logical expression.

```python
# Falsy — these all become False
bool(0)          # False
bool(0.0)        # False
bool("")         # False
bool([])         # False
bool({})         # False
bool(())         # False
bool(set())      # False
bool(None)       # False

# Truthy — everything else
bool(1)          # True
bool(-1)         # True   — non-zero integers are truthy
bool(0.001)      # True
bool("0")        # True   — non-empty string, even if it contains "0"
bool([0])        # True   — non-empty list
bool(" ")        # True   — whitespace is not empty
```

**This matters because Python conditionals implicitly call bool():**

```python
x = []
if x:              # equivalent to: if bool(x):
    print("has items")
else:
    print("empty") # this runs
```

## Implicit vs Explicit Casting

Python performs some type conversions automatically (implicit casting):

```python
# int + float -> float  (Python promotes to float)
result = 3 + 1.5
print(result)        # 4.5
print(type(result))  # <class 'float'>

# bool arithmetic (implicit int)
print(True + 1)      # 2
print(False + 10)    # 10
```

Everything else requires explicit casting. Python will not silently convert types that could lead to data loss:

```python
"3" + 3    # TypeError: can only concatenate str (not "int") to str
```

```python
# You must be explicit
int("3") + 3    # 6
"3" + str(3)    # "33"
```

## Safe Type Conversion Pattern

When input is untrusted (user input, file data, API responses), always wrap conversions in try/except:

```python
raw = input("Enter a number: ")

try:
    value = float(raw)
    print(f"Got: {value}")
except ValueError:
    print(f"'{raw}' is not a valid number")
```

More robust pattern with a helper function:

```python
def to_int(value, default=None):
    try:
        return int(value)
    except (ValueError, TypeError):
        return default

print(to_int("42"))       # 42
print(to_int("hello"))    # None
print(to_int("hello", 0)) # 0
print(to_int(None))       # None
```

## Type Checking in Functions — Defensive Programming

```python
def square(x):
    if not isinstance(x, (int, float)):
        raise TypeError(f"Expected int or float, got {type(x).__name__}")
    return x ** 2

square(4)       # 16
square(2.5)     # 6.25
square("4")     # TypeError: Expected int or float, got str
```

`type(x).__name__` gives you the clean name string (`"str"`, `"int"`, etc.) without the `<class '...'>` wrapper.

## Python Type Hints (Annotations)

Python 3.5+ supports type hints. These are not enforced at runtime — they are documentation and tooling signals only.

```python
def add(a: int, b: int) -> int:
    return a + b

def process(name: str, score: float) -> str:
    return f"{name}: {score}"
```

Type hints do not prevent wrong types at runtime:

```python
add("hello", "world")   # runs fine, returns "helloworld" — no error
```

To enforce types at runtime, you must check explicitly with isinstance() or use a library like `pydantic`. This matters in ML pipelines where silent type errors cause subtle bugs.

## Summary

|Operation|Function|Inheritance-aware|Use case|
|---|---|---|---|
|Check exact type|`type(x) == T`|No|Distinguish subclasses (e.g. bool vs int)|
|Check type/subclass|`isinstance(x, T)`|Yes|General type validation|
|Convert to int|`int(x)`|—|Numeric conversion, truncates floats|
|Convert to float|`float(x)`|—|Numeric promotion|
|Convert to string|`str(x)`|—|Always safe, never raises|
|Convert to bool|`bool(x)`|—|Truthiness evaluation|

## Common Mistakes to Avoid

```python
# Mistake 1: using == None instead of is None
x = None
if x == None: ...     # works but wrong — use 'is'
if x is None: ...     # correct

# Mistake 2: assuming int("3.14") works
int("3.14")           # ValueError — do int(float("3.14")) instead

# Mistake 3: assuming "0" is falsy
bool("0")             # True — it is a non-empty string

# Mistake 4: using type() when isinstance() is needed
isinstance(True, int) # True  — correct behavior
type(True) == int     # False — misleading if you expect bool to behave as int

# Mistake 5: not handling ValueError on user/external input
int(input())          # crashes if user types "abc" — always wrap in try/except
```
