
## Core Idea

A tuple is an **ordered, immutable** sequence of objects. It looks like a list but cannot be changed after creation — no adding, removing, or replacing elements.

- **Ordered** — elements have a defined position, indexed from 0
- **Immutable** — the tuple object cannot be modified after creation
- **Heterogeneous** — elements can be of different types (and this is often intentional with tuples, unlike lists)

The immutability is not a limitation — it is the point. A tuple signals that this collection of values belongs together and should not change. A list signals a collection of similar items that may grow or shrink.

```python
# List — homogeneous, mutable, order matters operationally
scores = [88, 92, 75, 90]

# Tuple — heterogeneous, immutable, structure matters semantically
person = ("Alice", 30, "Engineer")
point  = (3.5, -1.2)
rgb    = (255, 128, 0)
```

## Creating Tuples

```python
# Empty tuple
empty = ()
empty = tuple()

# Single element — trailing comma is REQUIRED
single = (42,)          # tuple with one element
not_tuple = (42)        # this is just the integer 42 in parentheses
print(type(single))     # <class 'tuple'>
print(type(not_tuple))  # <class 'int'>

# Multiple elements
point = (3, 4)
rgb = (255, 128, 0)
record = ("Alice", 30, True)

# Parentheses are optional — the comma makes a tuple
point = 3, 4
rgb = 255, 128, 0
print(type(point))      # <class 'tuple'>

# From an iterable
t = tuple([1, 2, 3])    # (1, 2, 3)
t = tuple("hello")      # ('h', 'e', 'l', 'l', 'o')
t = tuple(range(5))     # (0, 1, 2, 3, 4)
```

The single-element trailing comma is the most common tuple mistake. Forget it and you have an int wrapped in parentheses, not a tuple.

## Indexing and Slicing

Identical to lists — tuples are sequences.

```python
point = (10, 20, 30, 40, 50)
#         0   1   2   3   4
#        -5  -4  -3  -2  -1

print(point[0])      # 10
print(point[-1])     # 50
print(point[1:4])    # (20, 30, 40)   — returns a tuple
print(point[::-1])   # (50, 40, 30, 20, 10)
print(point[::2])    # (10, 30, 50)
```

Slicing a tuple returns a tuple, not a list.

## Immutability

Elements cannot be changed, added, or removed.

```python
point = (3, 4)

point[0] = 10       # TypeError: 'tuple' object does not support item assignment
point.append(5)     # AttributeError: 'tuple' object has no attribute 'append'
del point[0]        # TypeError: 'tuple' object doesn't support item deletion
```

However, if a tuple contains a mutable object, that object itself can be mutated:

```python
t = (1, [2, 3], 4)

t[1].append(99)     # modifying the list inside the tuple — allowed
print(t)            # (1, [2, 3, 99], 4)

t[1] = [5, 6]       # replacing the reference — TypeError
```

The tuple holds a reference to the list. The reference cannot change, but the list object the reference points to can. This distinction is important when using tuples as dictionary keys — a tuple containing a mutable object is not hashable and cannot be used as a key.

## Tuple Methods

Tuples have only two methods because they cannot be modified.

```python
t = (1, 2, 3, 2, 1, 2)

# count — number of occurrences of a value
print(t.count(2))       # 3

# index — index of first occurrence
print(t.index(3))       # 2
print(t.index(2))       # 1
# t.index(99)           # ValueError if not found
```

All sequence operations work on tuples:

```python
t = (3, 1, 4, 1, 5)

print(len(t))           # 5
print(min(t))           # 1
print(max(t))           # 5
print(sum(t))           # 14
print(3 in t)           # True
print(sorted(t))        # [1, 1, 3, 4, 5]  — returns a LIST
```

## Unpacking

Tuple unpacking is one of the most used Python features. It assigns elements of a tuple (or any iterable) to variables in one statement.

```python
point = (3, 4)
x, y = point
print(x)    # 3
print(y)    # 4
```

```python
# Works with any iterable on the right side
a, b, c = [1, 2, 3]
a, b, c = "xyz"
a, b = b, a             # swap without temp variable
```

**Extended unpacking with star:**

```python
first, *rest = (1, 2, 3, 4, 5)
print(first)    # 1
print(rest)     # [2, 3, 4, 5]   — star always creates a list

*init, last = (1, 2, 3, 4, 5)
print(init)     # [1, 2, 3, 4]
print(last)     # 5

a, *middle, z = (1, 2, 3, 4, 5)
print(middle)   # [2, 3, 4]
```

**Nested unpacking:**

```python
data = ((1, 2), (3, 4))
(a, b), (c, d) = data
print(a, b, c, d)    # 1 2 3 4
```

**Unpacking in loops:**

```python
pairs = [(1, "a"), (2, "b"), (3, "c")]

for number, letter in pairs:
    print(number, letter)

# dict.items() returns (key, value) tuples
scores = {"alice": 92, "bob": 85}
for name, score in scores.items():
    print(f"{name}: {score}")
```

## Returning Multiple Values from Functions

Python functions can return multiple values — they are packed into a tuple automatically.

```python
def min_max(data):
    return min(data), max(data)     # returns a tuple

result = min_max([3, 1, 4, 1, 5, 9])
print(result)           # (1, 9)
print(type(result))     # <class 'tuple'>

# Unpack directly
low, high = min_max([3, 1, 4, 1, 5, 9])
print(low, high)        # 1 9
```

```python
def train_step(model, batch):
    loss = compute_loss(model, batch)
    accuracy = compute_accuracy(model, batch)
    return loss, accuracy

loss, acc = train_step(model, batch)
```

This pattern is everywhere in ML code. Functions routinely return `(loss, accuracy)`, `(train_data, test_data)`, `(mean, variance)`, `(weights, biases)`. Recognize and use it fluently.

## Tuples as Dictionary Keys

Because tuples are immutable and hashable (when they contain only hashable elements), they can be used as dictionary keys. Lists cannot.

```python
# Grid coordinates as keys
grid = {}
grid[(0, 0)] = "start"
grid[(1, 2)] = "checkpoint"
grid[(5, 5)] = "end"

print(grid[(1, 2)])     # checkpoint

# Compound keys
cache = {}
cache[("relu", 128, 0.001)] = trained_model   # layer type, units, lr

# List as key — TypeError
d = {}
d[[1, 2]] = "value"    # TypeError: unhashable type: 'list'
```

This makes tuples the natural choice for multi-dimensional keys, memoization caches, and coordinate maps.

## Named Tuples

A named tuple is a tuple subclass that gives names to positions. It is readable like an object but immutable and memory-efficient like a tuple.

```python
from collections import namedtuple

# Define the structure
Point = namedtuple("Point", ["x", "y"])
Person = namedtuple("Person", ["name", "age", "role"])

# Create instances
p = Point(3, 4)
alice = Person("Alice", 30, "Engineer")

# Access by name or index
print(p.x)              # 3
print(p[0])             # 3
print(alice.name)       # Alice
print(alice[1])         # 30

# Still immutable
p.x = 10               # AttributeError

# Unpack like a regular tuple
x, y = p

# Useful repr
print(p)               # Point(x=3, y=4)
print(alice)           # Person(name='Alice', age=30, role='Engineer')
```

Named tuples are superior to plain tuples when the position of each element has meaning. They are also superior to dictionaries when the structure is fixed — less memory, faster access, immutable.

```python
# Without named tuple — what does index 2 mean?
record = ("Alice", 30, "Engineer")
print(record[2])        # "Engineer" — not obvious from context

# With named tuple — self-documenting
print(alice.role)       # "Engineer" — obvious
```

Python 3.7+ also offers `dataclasses` for mutable structured data and `typing.NamedTuple` for typed named tuples:

```python
from typing import NamedTuple

class Point(NamedTuple):
    x: float
    y: float
    z: float = 0.0      # default value

p = Point(1.0, 2.0)
print(p)                # Point(x=1.0, y=2.0, z=0.0)
```

## Tuple vs List — When to Use Which

|Situation|Use|
|---|---|
|Fixed structure with semantic meaning (coordinate, record, RGB)|Tuple|
|Collection of similar items that may grow or change|List|
|Return multiple values from a function|Tuple|
|Dictionary key|Tuple (lists cannot be keys)|
|Iteration target that should not be modified|Tuple|
|Sequence that will be sorted, filtered, extended|List|
|Heterogeneous fields (name, age, score)|Tuple or namedtuple|
|Homogeneous data (all floats, all strings)|List|

The choice communicates intent. A tuple says "these values belong together as a unit." A list says "here is a collection of items."

## Performance

Tuples are slightly faster and more memory-efficient than lists.

```python
import sys

lst = [1, 2, 3, 4, 5]
tup = (1, 2, 3, 4, 5)

print(sys.getsizeof(lst))   # 104 bytes (Python 3.10, may vary)
print(sys.getsizeof(tup))   # 80 bytes
```

Tuples skip the overhead of dynamic resizing since they never grow. For large collections of fixed data, this adds up.

Tuple creation is also faster than list creation:

```python
import timeit

# Tuple literal — faster, Python can use a constant
timeit.timeit("(1, 2, 3, 4, 5)", number=10_000_000)

# List literal — slower, allocates new object every time
timeit.timeit("[1, 2, 3, 4, 5]", number=10_000_000)
```

In practice, the performance difference matters only at scale. Do not choose tuple over list for performance — choose based on semantics. The performance is a bonus.

## Concatenation and Repetition

Tuples support `+` and `*` but return new tuples — the originals are never modified.

```python
a = (1, 2, 3)
b = (4, 5, 6)

print(a + b)        # (1, 2, 3, 4, 5, 6)
print(a * 3)        # (1, 2, 3, 1, 2, 3, 1, 2, 3)
print(a)            # (1, 2, 3)  — unchanged
```

## Converting Between List and Tuple

```python
lst = [1, 2, 3]
tup = tuple(lst)    # (1, 2, 3)

tup = (4, 5, 6)
lst = list(tup)     # [4, 5, 6]
```

Common pattern: store data as a tuple for safety, convert to list when you need to mutate, convert back to tuple when done.

## Summary

|Feature|Tuple|List|
|---|---|---|
|Syntax|`(1, 2, 3)`|`[1, 2, 3]`|
|Ordered|Yes|Yes|
|Mutable|No|Yes|
|Indexed|Yes|Yes|
|Slicing|Yes (returns tuple)|Yes (returns list)|
|Duplicates|Allowed|Allowed|
|Hashable|Yes (if contents hashable)|No|
|Dict key|Yes|No|
|Methods|`count`, `index` only|Full set|
|Memory|Less|More|
|Use case|Fixed structure, records|Dynamic collection|

## Common Mistakes to Avoid

```python
# Mistake 1: forgetting trailing comma for single-element tuple
single = (42)       # this is int 42, not a tuple
single = (42,)      # correct

# Mistake 2: trying to modify a tuple
t = (1, 2, 3)
t[0] = 99           # TypeError — use a list if you need mutability

# Mistake 3: assuming tuple containing a list is fully immutable
t = (1, [2, 3], 4)
t[1].append(99)     # allowed — the list inside can be mutated
                    # do not use such tuples as dict keys

# Mistake 4: ignoring return values from functions that return tuples
result = min_max(data)   # result is a tuple — you may need to unpack
low, high = min_max(data)  # unpack immediately

# Mistake 5: using a list when a tuple communicates intent better
def get_coordinates():
    return [x, y]    # signals "mutable collection" — misleading
    return (x, y)    # signals "fixed pair" — correct

# Mistake 6: using plain tuple instead of namedtuple for structured records
record = ("Alice", 30, "Engineer")
print(record[2])         # what is index 2? unclear
# use namedtuple — self-documenting and same performance
```
