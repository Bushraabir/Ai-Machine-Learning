## Core Idea

Comments and docstrings are both ways to embed human-readable text inside code. They serve different purposes and are used in different places.

- **Comments** — explain why code does something. For the developer reading the code.
- **Docstrings** — document what a module, class, or function does. For the user calling the code. They are also machine-readable and used by tools like `help()`, IDEs, and documentation generators.

The distinction matters. A comment is thrown away at runtime. A docstring is stored as an attribute on the object it describes and can be accessed programmatically.

---

## Comments

### Single-Line Comment

A `#` character starts a comment. Everything after it on that line is ignored by the interpreter.

```python
# This is a comment
x = 10    # this is an inline comment

# Compute the mean of the dataset
mean = sum(data) / len(data)
```

Python has no multi-line comment syntax. To comment out multiple lines, use `#` on each line.

```python
# This is line one of a long explanation
# that continues on this line
# and ends here
```

Most editors let you select multiple lines and toggle `#` with a keyboard shortcut. Use it.

---

### What to Comment — and What Not To

A comment should explain **why**, not **what**. The code already shows what it does. A comment that restates the code adds noise without value.

```python
# Bad comment — restates the obvious
x = x + 1    # add 1 to x

# Good comment — explains why
x = x + 1    # offset by 1 because the API uses 1-based indexing
```

```python
# Bad — no comment needed, code is self-explanatory
for i in range(len(data)):    # loop through data
    process(data[i])

# Good — explains a non-obvious decision
# Using floor division here because the downstream function
# expects an integer index, not a float
index = total // batch_size
```

**Write comments for:**

- Non-obvious algorithmic decisions
- Workarounds for bugs or edge cases
- Assumptions the code makes about input
- Why a particular formula or constant was chosen
- TODO / FIXME notes during development

**Do not write comments for:**

- Things the code clearly expresses on its own
- Every single line
- Restating variable names or operations

---

### TODO and FIXME Comments

Widely used conventions for leaving notes in code:

```python
# TODO: replace linear search with binary search once data is sorted
# FIXME: this crashes when input list is empty
# HACK: temporary workaround until upstream bug is fixed
# NOTE: this assumes all timestamps are UTC
```

Many IDEs highlight these automatically and let you search for them across a codebase.

---

### Commenting Out Code

During development you may temporarily disable code by commenting it out:

```python
# old implementation — keeping for reference
# result = slow_function(data)
result = fast_function(data)
```

Acceptable during development. Do not leave large blocks of commented-out code in finished work — use version control to recover old code instead.

## Docstrings

A docstring is a string literal that appears as the **first statement** in a module, class, or function body. It is enclosed in triple quotes.

```python
def add(a, b):
    """Return the sum of a and b."""
    return a + b
```

Python stores the docstring as the `__doc__` attribute of the object:

```python
print(add.__doc__)    # Return the sum of a and b.
help(add)             # displays formatted docstring in terminal
```

This is the key difference from comments — docstrings are live data attached to objects, not just ignored text.

---

### One-Line Docstrings

For simple, obvious functions. The entire docstring fits on one line.

```python
def square(x):
    """Return the square of x."""
    return x ** 2

def is_even(n):
    """Return True if n is even, False otherwise."""
    return n % 2 == 0
```

Rules for one-liners:

- Opening and closing `"""` on the same line
- No blank line before or after
- Written as a command, not a description ("Return..." not "Returns...")

---

### Multi-Line Docstrings

For functions with parameters, return values, exceptions, or non-trivial behavior.

```python
def divide(a, b):
    """
    Divide a by b and return the result.

    Parameters
    ----------
    a : float
        The numerator.
    b : float
        The denominator. Must not be zero.

    Returns
    -------
    float
        The result of a divided by b.

    Raises
    ------
    ValueError
        If b is zero.

    Examples
    --------
    >>> divide(10, 2)
    5.0
    >>> divide(7, 0)
    ValueError: denominator cannot be zero
    """
    if b == 0:
        raise ValueError("denominator cannot be zero")
    return a / b
```

Structure of a multi-line docstring:

1. Summary line on the same line as the opening `"""`
2. One blank line after the summary
3. Detailed sections: parameters, returns, raises, examples
4. Closing `"""` on its own line

## Docstring Formats

There is no single enforced format. Three formats are widely used. Pick one and stay consistent within a project.

---

### Google Style

Clean and readable. Used by Google, TensorFlow, and many modern projects.

```python
def train(model, data, epochs=10, lr=0.001):
    """Train the model on the given dataset.

    Args:
        model: The neural network model to train.
        data: Training dataset as a list of (input, label) pairs.
        epochs (int): Number of training iterations. Defaults to 10.
        lr (float): Learning rate. Defaults to 0.001.

    Returns:
        dict: Training history containing loss and accuracy per epoch.

    Raises:
        ValueError: If epochs is less than 1.
        TypeError: If data is not iterable.

    Example:
        >>> history = train(model, dataset, epochs=5)
        >>> print(history["loss"])
    """
```

---

### NumPy / SciPy Style

More structured, uses section headers with underlines. Standard in scientific Python — NumPy, SciPy, scikit-learn, pandas. You will read this format constantly in ML library source code.

```python
def normalize(x, axis=0):
    """
    Normalize array x to have zero mean and unit variance.

    Parameters
    ----------
    x : numpy.ndarray
        Input array of shape (n_samples, n_features).
    axis : int, optional
        Axis along which to normalize. Default is 0.

    Returns
    -------
    numpy.ndarray
        Normalized array of same shape as x.

    Notes
    -----
    Normalization is computed as (x - mean) / std.
    If std is zero, the function returns zeros to avoid division by zero.

    Examples
    --------
    >>> import numpy as np
    >>> x = np.array([[1.0, 2.0], [3.0, 4.0]])
    >>> normalize(x)
    array([[-1., -1.],
           [ 1.,  1.]])
    """
```

Learn this format. You will encounter it in NumPy, scikit-learn, and SciPy documentation throughout your ML studies.

---

### Sphinx / reStructuredText Style

Older format. Still found in many large Python projects and used by Python's official documentation generator.

```python
def connect(host, port, timeout=30):
    """
    Establish a connection to the server.

    :param host: Server hostname or IP address.
    :type host: str
    :param port: Port number to connect to.
    :type port: int
    :param timeout: Connection timeout in seconds. Defaults to 30.
    :type timeout: int
    :returns: An open socket connection.
    :rtype: socket.socket
    :raises ConnectionError: If the connection cannot be established.
    """
```

## Docstrings for Classes

```python
class LinearRegression:
    """
    Ordinary least squares linear regression.

    Fits a linear model to minimize the residual sum of squares
    between observed and predicted values.

    Attributes
    ----------
    coef_ : numpy.ndarray
        Estimated coefficients for the linear model.
    intercept_ : float
        Independent term in the linear model.

    Example
    -------
    >>> model = LinearRegression()
    >>> model.fit(X_train, y_train)
    >>> predictions = model.predict(X_test)
    """

    def __init__(self, fit_intercept=True):
        """
        Initialize the model.

        Parameters
        ----------
        fit_intercept : bool
            Whether to include an intercept term. Default is True.
        """
        self.fit_intercept = fit_intercept
        self.coef_ = None
        self.intercept_ = None
```

The class docstring describes what the class represents. The `__init__` docstring describes initialization parameters.

## Accessing Docstrings at Runtime

```python
def greet(name):
    """Return a greeting string for the given name."""
    return f"Hello, {name}"

print(greet.__doc__)
# Return a greeting string for the given name.

help(greet)
# Help on function greet:
# greet(name)
#     Return a greeting string for the given name.
```

This works on classes, modules, and built-ins:

```python
help(list)
help(str.split)
print(len.__doc__)
```

This is how you read documentation inside the Python REPL or a Jupyter notebook without leaving your environment. You will use this constantly.

## Docstrings vs Comments — Side by Side

```python
def compute_loss(predictions, targets):
    """
    Compute mean squared error loss between predictions and targets.

    Parameters
    ----------
    predictions : list of float
        Model output values.
    targets : list of float
        Ground truth values.

    Returns
    -------
    float
        Mean squared error.
    """
    # Using a generator to avoid allocating a full intermediate list in memory
    squared_errors = ((p - t) ** 2 for p, t in zip(predictions, targets))
    return sum(squared_errors) / len(targets)
```

The docstring tells a caller what the function does, what it accepts, and what it returns — without reading the implementation.

The comment inside the body explains a specific implementation decision — for the developer maintaining the code.

## Summary

|Feature|Comment|Docstring|
|---|---|---|
|Syntax|`#`|`"""..."""`|
|Placement|Any line|First statement in module, class, or function|
|Runtime|Ignored completely|Stored as `__doc__` attribute|
|Purpose|Explain why|Document what and how to use|
|Accessible via|Nothing|`help()`, `__doc__`, IDEs, doc generators|
|Multi-line|`#` on each line|Triple quotes natively|

## Common Mistakes to Avoid

```python
# Mistake 1: commenting what instead of why
x = x + 1    # increment x by 1      <- states the obvious

# Mistake 2: outdated comment (worse than no comment at all)
# multiply by 2
result = x * 4    # code changed, comment was not updated

# Mistake 3: docstring placed after code — becomes a plain string, not a docstring
def add(a, b):
    return a + b
    """This does not become a docstring — it is unreachable."""

# Mistake 4: no docstring on public functions in a project
def train_model(data, epochs):
    # forces every reader to read the full implementation to understand usage
    ...

# Mistake 5: using a comment instead of a docstring
def square(x):
    # returns x squared
    return x ** 2
# this is not accessible via help() or __doc__
```
