## Core Idea

`break`, `continue`, and `pass` are loop and block control statements. They alter the normal top-to-bottom execution flow inside loops and conditional blocks.

|Statement|Effect|
|---|---|
|`break`|Exit the loop entirely, immediately|
|`continue`|Skip the rest of this iteration, go to next|
|`pass`|Do nothing — syntactic placeholder|

They look similar but serve completely different purposes. `break` and `continue` affect control flow. `pass` affects nothing — it exists purely to satisfy Python's syntax requirement that a block cannot be empty.

---

## break

### What It Does

Terminates the nearest enclosing loop (`for` or `while`) immediately. Execution jumps to the first statement after the loop body.

```python
for n in range(10):
    if n == 5:
        break
    print(n)

print("after loop")

# 0, 1, 2, 3, 4
# after loop
```

The loop had 10 items to process. It stopped at 5. Everything from 5 onward was never reached.

---

### break in a while Loop

```python
x = 0

while True:
    if x >= 5:
        break
    print(x)
    x += 1

# 0, 1, 2, 3, 4
```

`while True` with `break` is intentional and common. The exit condition is checked inside the body rather than at the top.

---

### break Only Exits the Nearest Loop

In nested loops, `break` exits only the innermost loop. The outer loop continues.

```python
for i in range(3):
    for j in range(3):
        if j == 1:
            break           # breaks inner loop only
        print(i, j)

# (0, 0)
# (1, 0)
# (2, 0)
```

To break out of multiple levels, use one of these approaches:

**Approach 1 — flag variable:**

```python
found = False

for i in range(5):
    for j in range(5):
        if some_condition(i, j):
            found = True
            break
    if found:
        break
```

**Approach 2 — extract into a function and use return:**

```python
def search(data):
    for row in data:
        for item in row:
            if condition(item):
                return item     # exits both loops immediately
    return None
```

The function approach is cleaner. `return` exits the entire function, which exits all loops inside it. This is the preferred pattern.

---

### Common Use Cases for break

**Search — stop once found:**

```python
target = 42
found = False

for item in data:
    if item == target:
        found = True
        break

# More Pythonic:
found = target in data
```

**First valid result:**

```python
result = None

for candidate in candidates:
    if is_valid(candidate):
        result = candidate
        break
```

**Exit on error:**

```python
for record in records:
    if not is_parseable(record):
        print(f"Bad record: {record}")
        break
    process(record)
```

**Input loop:**

```python
while True:
    user_input = input("Enter value (or 'quit'): ")
    if user_input == "quit":
        break
    process(user_input)
```

---

### break with else

The `else` clause on a loop runs only if the loop completed without hitting a `break`. This distinguishes "found it" from "exhausted all options."

```python
for item in data:
    if condition(item):
        print("found:", item)
        break
else:
    print("not found")
```

```python
# Practical example: checking primality
def is_prime(n):
    if n < 2:
        return False
    for i in range(2, int(n ** 0.5) + 1):
        if n % i == 0:
            return False    # found a factor
    return True             # no factor found — prime
```

The `for/else` version of the same:

```python
def is_prime(n):
    if n < 2:
        return False
    for i in range(2, int(n ** 0.5) + 1):
        if n % i == 0:
            break
    else:
        return True
    return False
```

## continue

### What It Does

Skips the rest of the current iteration and jumps immediately to the next iteration — re-evaluating the loop condition for `while`, or advancing to the next item for `for`.

```python
for n in range(10):
    if n % 2 == 0:
        continue        # skip even numbers
    print(n)

# 1, 3, 5, 7, 9
```

Code below `continue` in the same iteration is not executed. Code in the next iteration runs normally.

---

### continue in a while Loop

```python
x = 0

while x < 10:
    x += 1              # increment BEFORE continue to avoid infinite loop
    if x % 2 == 0:
        continue
    print(x)

# 1, 3, 5, 7, 9
```

**The increment must happen before `continue`.** If the variable that drives the loop condition is updated after `continue`, it will never be reached and the loop runs forever.

```python
# Infinite loop — do not write this
x = 0
while x < 10:
    if x % 2 == 0:
        continue        # x is never incremented when even
    x += 1
    print(x)
```

---

### continue vs Nested if

`continue` flattens logic by inverting the condition and skipping early, instead of wrapping the real work in an `if` block.

```python
# Without continue — nested
for record in records:
    if record.is_valid():
        if record.value > 0:
            process(record)

# With continue — flat, guard-clause style
for record in records:
    if not record.is_valid():
        continue
    if record.value <= 0:
        continue
    process(record)
```

The second version is easier to read. Each guard condition is explicit and isolated. The main logic (`process(record)`) is at the same indentation level as the guards — not buried inside nested blocks.

This is the loop equivalent of guard clauses in functions.

---

### Common Use Cases for continue

**Filtering in a loop:**

```python
for line in file_lines:
    if line.startswith("#"):    # skip comments
        continue
    if line.strip() == "":      # skip blank lines
        continue
    parse(line)
```

**Skipping error-prone items:**

```python
for item in data:
    if item is None:
        continue
    process(item)
```

**Skipping already-processed items:**

```python
processed = set()

for item in data:
    if item.id in processed:
        continue
    process(item)
    processed.add(item.id)
```

## pass

### What It Does

Does absolutely nothing. It is a no-op — a syntactic placeholder that satisfies Python's requirement that every block must contain at least one statement.

```python
if condition:
    pass        # empty block — no error
```

Without `pass`, an empty block is a `SyntaxError`:

```python
if condition:
    # nothing here    <- IndentationError / SyntaxError
```

---

### Where pass Is Used

**Empty function or method — stub during development:**

```python
def train():
    pass

def evaluate():
    pass

def save_checkpoint():
    pass
```

Write the interface first, implement later. `pass` holds the structure while you figure out what each function should do.

---

**Empty class:**

```python
class DataLoader:
    pass

class CustomException(Exception):
    pass
```

`class CustomException(Exception): pass` is a real pattern — creating a named exception type with no additional behavior, just so you can catch it specifically.

---

**Empty conditional branch:**

```python
if error:
    pass        # will handle later
else:
    proceed()
```

---

**Empty loop body:**

```python
# Consume an iterator without doing anything with the values
for _ in range(10):
    pass
```

Rare, but occasionally useful when you need a loop to advance an iterator's state as a side effect.

---

**Abstract method placeholder (before abc module):**

```python
class Model:
    def predict(self, x):
        pass    # subclasses must implement this

    def train(self, data):
        pass
```

In real code, use `raise NotImplementedError` instead — it actively signals that the method must be overridden, rather than silently doing nothing if called.

```python
class Model:
    def predict(self, x):
        raise NotImplementedError("subclasses must implement predict()")
```

---

### pass vs comment vs ellipsis

Three ways to write an empty block — each has a different semantic meaning:

```python
# pass — explicit no-op, syntactically required placeholder
def stub():
    pass

# comment — ignored entirely, does not count as a statement
def broken():
    # this is not a statement — SyntaxError without pass

# ellipsis (...) — valid expression, used in type stubs and abstract methods
def abstract_method(self):
    ...
```

`...` (the ellipsis literal) is a valid Python object and a valid statement. It is increasingly used in modern Python as a cleaner alternative to `pass` in type stubs, abstract base classes, and placeholder functions. Both work — `pass` is more traditional, `...` is more modern.

```python
# Both are valid and equivalent for empty blocks
def func_a(): pass
def func_b(): ...
```

## Side-by-Side Comparison

```python
data = [1, -2, 3, -4, 5, 99, 6]

# break — stop as soon as we see a value over 10
for n in data:
    if n > 10:
        print(f"stopping at {n}")
        break
    print(n)

# 1, -2, 3, -4, 5
# stopping at 99


# continue — skip negative values, process the rest
for n in data:
    if n < 0:
        continue
    print(n)

# 1, 3, 5, 99, 6


# pass — placeholder, loop does nothing (for illustration only)
for n in data:
    pass

print("last value seen:", n)    # 6
```

## Execution Flow Diagram

```
for item in iterable:
    ┌─────────────────────┐
    │  start of iteration │
    │                     │
    │  if condition:      │
    │      break   ───────┼──────► exits loop entirely
    │                     │
    │  if condition:      │
    │      continue ──────┼──────► jumps to next iteration
    │                     │
    │  if condition:      │
    │      pass           │  (does nothing, continues normally)
    │                     │
    │  rest of body       │
    └─────────────────────┘
           │
           ▼
    next iteration
```

## Summary

|Statement|Exits loop?|Skips iteration?|Does anything?|Where valid|
|---|---|---|---|---|
|`break`|Yes|—|No|Inside any loop|
|`continue`|No|Yes|No|Inside any loop|
|`pass`|No|No|No|Any empty block|

## Common Mistakes to Avoid

```python
# Mistake 1: expecting break to exit nested loops
for i in range(5):
    for j in range(5):
        break           # only exits inner loop
# use a function with return instead

# Mistake 2: continue before loop variable update in while
x = 0
while x < 10:
    if x % 2 == 0:
        continue        # infinite loop — x never increments
    x += 1
# fix: increment before continue

# Mistake 3: using pass when raise NotImplementedError is correct
class MyModel:
    def fit(self, X, y):
        pass            # silently does nothing if called accidentally
# fix:
    def fit(self, X, y):
        raise NotImplementedError("fit() must be implemented")

# Mistake 4: using continue to mean pass
for item in data:
    continue            # pointless — just removes the loop body
    process(item)       # unreachable

# Mistake 5: misunderstanding for/else — else runs when no break occurred
for item in data:
    if condition(item):
        break
else:
    print("this runs if break was never hit")
    # NOT: "this runs if loop body ran zero times"

# Mistake 6: not knowing break only affects the nearest enclosing loop
# leads to logic bugs in nested loop searches
```
